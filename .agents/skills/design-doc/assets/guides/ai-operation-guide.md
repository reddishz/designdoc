# AI 操作指南 - 编码修改防护

## 核心原则

**编码一经分配，永不修改其含义和标题**

这是维护编码体系完整性的核心原则，所有 AI 智能体必须严格遵守。

**新增约束**：
1. **即时索引更新**：新生成编码 ID 时，必须立即更新所有相关索引
2. **标题稳定性**：编码 ID 对应的标题，一旦新建也不得修改
3. **标准引用格式**：引用细项编码时，除非特殊原因，要连同标题一起带上
4. **标题适度宽泛**：编码标题应适度宽泛，避免过于具体的细节

## 操作场景与处理方式

### 场景 1：用户要求修改已分配编码的含义

**用户请求示例**：
- "把 RUL-016 从'账号检查'改成'定时器规则'"
- "修改 FR-005 的标题"
- "更新 PRN-003 的内容"

**AI 必须执行以下步骤**：

1. **检查编码状态**：
   ```bash
   # 搜索编码是否已存在
   grep -r "RUL-016" ued/ --include="*.md"
   ```

2. **如编码已存在，拒绝修改**：
   ```
   ❌ 不能直接修改 RUL-016 的含义
   
   原因：编码 RUL-016 已分配给'账号检查'，根据编码体系规范，
   已分配编码的含义和标题永久固定，不能修改。
   ```

3. **提供正确解决方案**：
   ```
   ✅ 建议的正确处理方式：
   
   1. RUL-016 标注为"已废弃"，原因：需求变更
   2. 分配新编码 RUL-017 给"定时器规则"
   3. 如"账号检查"仍需要，分配 RUL-018
   4. 更新相关引用和全局索引
   ```

### 场景 4：新编码分配后索引更新

**用户请求示例**：
- "创建一个新的功能需求 FR-015"
- "添加一个新规则 RUL-020"

**AI 必须执行以下步骤**：

1. **分配编码**：
   ```
   - 从计数器获取下一个编号：FR-015
   - 确定编码标题：用户身份认证功能
   ```

2. **立即更新索引**：
   ```
   ✅ 全局编码索引（ued/README.md）：
   - 添加 FR-015 到功能需求表格
   - 更新 FR 类型码计数器：下一个编号 = 016
   
   ✅ 文档细项清单（当前文档末尾）：
   - 添加 FR-015 到细项编码清单
   
   ✅ 跨项目引用索引（如适用）：
   - 更新相关引用关系表
   ```

3. **验证更新**：
   ```
   - 检查索引是否完整
   - 验证计数器是否正确
   - 确认引用关系无误
   ```

**错误示例**：
```
❌ 错误做法：
- 分配 FR-015 后，说"稍后更新索引"
- 只更新了文档，未更新全局索引
- 忘记递增计数器

✅ 正确做法：
- 分配完成后立即更新所有索引
- 验证更新的一致性
- 确认计数器正确递增
```

### 场景 5：编码标题命名指导

**用户请求示例**：
- "创建一个规则：账号长度必须是 6-20 位，包含字母数字"
- "添加一个需求：用户使用手机号+密码登录系统"

**AI 必须提供标题命名指导**：

1. **分析用户描述**：
   ```
   用户描述过于具体，包含实现细节
   需要引导使用更宽泛的标题
   ```

2. **提供命名建议**：
   ```
   ❌ 不推荐标题：
   - "账号长度 6-20 位，包含字母数字"
   - "用户使用手机号+密码登录系统"
   
   ✅ 推荐标题：
   - "账号格式规范"
   - "用户身份认证功能"
   ```

3. **解释命名原则**：
   ```
   命名原则：
   - 适度宽泛：避免具体参数和实现细节
   - 稳定表述：使用相对稳定的业务概念
   - 考虑扩展：标题应能适应未来的变化
   ```

### 场景 6：引用格式检查和纠正

**用户请求示例**：
- "在文档中引用 CRM-FR-001"
- "检查文档中的引用格式"

**AI 必须执行引用格式检查**：

1. **检查当前引用格式**：
   ```bash
   # 查找所有编码引用
   grep -o '[A-Z]\+-[A-Z]\+-[0-9]\+' document.md
   ```

2. **验证引用完整性**：
   ```
   检查项：
   - 编码格式是否正确
   - 是否包含标题
   - 标题是否与索引一致
   - 空格分隔是否正确
   ```

3. **纠正错误格式**：
   ```
   ❌ 错误格式：
   - CRM-FR-001
   - CRM-FR-001用户登录功能
   - crm-fr-001 用户身份认证功能
   
   ✅ 正确格式：
   - CRM-FR-001 用户身份认证功能
   ```

4. **提供标准模板**：
   ```
   常用引用语句：
   - 根据 CRM-FR-001 用户身份认证功能 的要求...
   - 遵循 RUL-004 账号格式规范 的原则...
   - 满足 AC-001 登录成功标准 的条件...
   ```

### 场景 7：细项废弃处理

**用户请求示例**：
- "废弃 RUL-004 账号格式规范"
- "RUL-004 不需要了，怎么处理？"

**AI 必须执行以下步骤**：

1. **废弃评估**：
   ```
   评估清单：
   - 确认细项确实不再需要
   - 检查是否有其他文档引用此细项
   - 评估废弃的影响范围
   - 确认是否有替代方案
   ```

2. **废弃标记**：
   ```
   标准废弃标记格式：
   ### RUL-004 账号格式规范 ~~已废弃~~
   
   **废弃信息**：
   - 废弃时间：YYYY-MM-DD
   - 废弃原因：{具体原因}
   - 替代方案：{替代细项编码}（如有）
   - 废弃人：{废弃人姓名}
   
   ~~原内容：{原细项内容}~~
   ```

3. **引用处理**：
   ```
   搜索所有引用：
   grep -r "RUL-004" ued/ --include="*.md"
   
   处理方式：
   - 直接替代：用新细项替代
   - 移除引用：如无替代方案则移除
   - 保留说明：特殊情况需说明理由
   ```

4. **索引更新**：
   ```
   全局编码索引更新：
   | 编码 | 类型 | 标题 | 状态 | 废弃时间 | 废弃原因 | 替代方案 |
   |------|------|------|------|----------|----------|----------|
   | CRM-RUL-004 | RUL | 账号格式规范 | 已废弃 | 2024-03-23 | 需求变更 | CRM-RUL-018 |
   
   文档细项清单更新：
   | 编码 | 类型 | 标题 | 状态 | 备注 |
   |------|------|------|------|------|
   | CRM-RUL-004 | RUL | 账号格式规范 | ~~已废弃~~ | 2024-03-23 废弃，替代：RUL-018 |
   ```

5. **变更记录**：
   ```
   在相关文档中记录废弃变更：
   | 版本 | 日期 | 变更类型 | 变更内容 | 变更人 |
   |------|------|----------|----------|--------|
   | v1.2 | 2024-03-23 | 废弃 | 废弃 RUL-004 账号格式规范，采用 RUL-018 | 张三 |
   ```

**错误示例**：
```
❌ 错误做法：
- 直接删除废弃的细项
- 不处理引用关系
- 不更新索引
- 不记录废弃信息

✅ 正确做法：
- 完整的废弃标记和信息
- 正确处理所有引用
- 及时更新所有索引
- 详细记录变更历史
```

### 场景 8：批量废弃处理

**用户请求示例**：
- "业务重构，需要废弃一批规则"
- "项目调整，多个需求不再适用"

**AI 必须提供批量废弃指导**：

1. **制定废弃计划**：
   ```
   废弃计划清单：
   - 待废弃细项列表
   - 废弃原因分类
   - 影响范围评估
   - 处理优先级
   ```

2. **分批处理策略**：
   ```
   分批原则：
   - 按影响程度分批
   - 按依赖关系分批
   - 按业务模块分批
   - 避免同时处理过多
   ```

3. **统一更新方案**：
   ```
   批量更新：
   - 统一更新引用关系
   - 批量更新索引
   - 统一记录变更
   - 生成废弃报告
   ```

4. **沟通协调**：
   ```
   沟通要点：
   - 提前通知相关团队
   - 说明废弃影响
   - 提供替代方案
   - 收集反馈意见
   ```

**用户请求示例**：
- "更新 RUL-016 的详细描述"
- "补充 FR-005 的业务背景"

**AI 可以执行**：
- ✅ 允许修改：编码的详细描述、业务背景、说明文字
- ❌ 禁止修改：编码的含义、标题、类型

**判断标准**：
- **含义/标题**：编码代表的业务概念或技术规则（禁止修改）
- **描述内容**：对含义的详细解释和说明（允许修改）

### 场景 3：发现编码分配错误

**发现的问题**：
- RUL-016 应该是"定时器规则"，但被错误分配为"账号检查"

**正确处理流程**：
```
1. 立即废弃错误编码：
   - RUL-016 标注为"已废弃"
   - 原因：分配错误
   
2. 重新正确分配：
   - RUL-017 分配给"定时器规则"
   - RUL-018 分配给"账号检查"（如仍需要）
   
3. 处理引用关系：
   - 检查是否有文档引用了 RUL-016
   - 更新所有引用到新的正确编码
   
4. 更新索引：
   - 全局编码索引中标注 RUL-016 废弃
   - 添加 RUL-017 和 RUL-018 的索引信息
```

## 冲突检测机制

### 修改前检查清单

在执行任何编码相关操作前，AI 必须执行：

1. **编码存在性检查**：
   ```bash
   grep -r "{编码}" ued/ --include="*.md"
   ```

2. **引用关系检查**：
   ```bash
   grep -r "{编码}" ued/ --include="*.md" | grep -v "已废弃"
   ```

3. **影响范围评估**：
   - 有多少文档引用了这个编码？
   - 这些文档的状态是什么？
   - 修改会产生什么影响？

### 检查结果处理

| 检查结果 | 处理方式 |
|----------|----------|
| 编码不存在 | 可以分配新编码 |
| 编码已存在，无引用 | 可以废弃+新增 |
| 编码已存在，有引用 | 需先处理引用关系 |
| 编码已废弃 | 不能重用，分配新编码 |

## 审核要点

### 对 AI 的审核

审核 AI 是否正确执行了编码修改防护：

1. **是否检查了编码状态**？
2. **是否拒绝了不当修改请求**？
3. **是否提供了正确的解决方案**？
4. **是否执行了完整的废弃+新增流程**？
5. **是否更新了相关引用和索引**？

### 对文档的审核

检查文档中是否有违反编码修改防护的情况：

1. **编码含义是否被擅自修改**？
2. **废弃编码是否被重用**？
3. **引用关系是否正确更新**？
4. **全局索引是否同步更新**？

## 工具支持

### 编码检查脚本

```bash
#!/bin/bash
# 检查编码是否存在
check_coding_exists() {
    local coding=$1
    local result=$(grep -r "$coding" ued/ --include="*.md" | grep -v "已废弃")
    if [ -n "$result" ]; then
        echo "编码 $coding 已存在且未被废弃"
        return 1
    else
        echo "编码 $coding 不存在或已废弃"
        return 0
    fi
}

# 检查编码引用
check_coding_references() {
    local coding=$1
    local refs=$(grep -r "$coding" ued/ --include="*.md" | grep -v "已废弃")
    echo "编码 $coding 的引用："
    echo "$refs"
}
```
