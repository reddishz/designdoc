# 设计文档审核规范

## 审核流程

审核设计文档时，按以下检查项逐层执行：

### 必选项检查（可执行验证）

#### 状态值合规性
- [ ] **L0-L6 文档状态** ∈ {草稿, 正式, 废弃}
- [ ] **ADR 状态** ∈ {提议, 采纳, 拒绝, 废弃, 取代}
- [ ] **REF 状态** ∈ {有效, 废弃}
- [ ] **需求追溯状态** ∈ {待验证, 已验证, 失败}

#### 元信息完整性
- [ ] 文档编码存在且格式正确（如 L0-001）
- [ ] 层级字段与文档类型匹配
- [ ] 版本号符合语义化规范（vX.Y）
- [ ] 状态值从标准集合中选择
- [ ] 作者字段非空
- [ ] 创建日期和最后更新日期存在

#### 编码体系合规性
- [ ] 细项编码全局唯一（无重复）
- [ ] 编码格式符合类型码规范
- [ ] 交叉引用编码确实存在
- [ ] 引用方向正确（仅下层引用上层）

### 合规性检查

- [ ] 不存在代码或具体实现内容（核心约束）
- [ ] 目录结构和文件命名合规
- [ ] 文件名不含中文；文档内容使用中文撰写（二者不可混淆）
- [ ] 细项编码清单完整且与正文一致
- [ ] 变更记录存在且最新

### SSoT 一致性检查

- [ ] 细项编码全局无冲突（无重复编码）
- [ ] 交叉引用的编码确实存在于系统中
- [ ] 引用方向正确（仅下层引用上层）

### 逐层合规性检查

从上层向下层逐级检查，确保下层不违背上层：

- [ ] **L1 vs L0**：利益相关者需求是否对齐战略目标（GOL-xxx）
- [ ] **L2 vs L1**：功能/非功能需求是否覆盖利益相关者场景（SCN-xxx）
- [ ] **L3 vs L2**：架构方案是否满足所有需求（FR-xxx, NFR-xxx）
- [ ] **L4 vs L3**：系统设计是否遵循架构原则（PRN-xxx）和约束（CON-xxx）
- [ ] **L5 vs L4**：详细设计是否符合接口契约（IF-xxx）和领域模型（DOM-xxx）
- [ ] **L6 vs L2**：验证场景是否追溯到需求（FR-xxx）和验收标准（AC-xxx）

## 常见问题检查

### 内容合规性

1. **代码混入**：检查是否包含 SQL、配置文件、脚本等具体实现
2. **编码冲突**：验证细项编码的全局唯一性
3. **引用错误**：检查交叉引用的编码是否存在
4. **层级违背**：确保下层设计不违背上层约束

### 格式规范性

1. **元信息缺失**：检查文档头部的元信息表格
2. **编码清单**：验证文档末尾的细项编码清单
3. **变更记录**：确认变更记录的完整性
4. **命名规范**：检查文件名和目录名是否符合规范

### 质量保证

1. **可追溯性**：每个需求是否都有对应的验证方案
2. **完整性**：设计是否覆盖所有必需的场景
3. **一致性**：相关文档之间是否存在冲突
4. **可维护性**：文档结构是否清晰，便于后续维护
