# 设计文档审核规范

## 审核流程

审核设计文档时，按以下检查项逐层执行：

### 必选项检查（可执行验证）

#### 状态值合规性
- [ ] **L0-L6 文档状态** ∈ {草稿, 正式, 废弃}
- [ ] **ADR 状态** ∈ {提议, 采纳, 拒绝, 废弃, 取代}
- [ ] **REF 状态** ∈ {有效, 废弃}
- [ ] **需求追溯状态** ∈ {待验证, 已验证, 失败}

#### 元信息完整性
- [ ] 文档编码存在且格式正确（如 `CRM-L0-001`，含项目码与层级）
- [ ] 层级字段与文档类型匹配
- [ ] 版本号符合语义化规范（vX.Y）
- [ ] 状态值从标准集合中选择
- [ ] 作者字段非空
- [ ] 创建日期和最后更新日期存在

#### 编码体系合规性
- [ ] 细项编码全局唯一（无重复）
- [ ] 编码格式符合类型码规范
- [ ] 交叉引用编码确实存在
- [ ] 引用方向正确（仅下层引用上层）

### 合规性检查

- [ ] 不存在代码或具体实现内容（核心约束）
- [ ] 目录结构和文件命名合规
- [ ] 文件名不含中文；文档内容使用中文撰写（二者不可混淆）
- [ ] 细项编码清单完整且与正文一致
- [ ] 变更记录存在且最新
- [ ] 项目编码配置正确（ued/.doc-config.json 存在且格式正确）
- [ ] 编码生命周期规则遵守（永不删除、永不复用）
- [ ] 真值源优先级遵守（本 skill 包内 `references/` 专项文件优先于 `SKILL.md`）

### 项目编码合规性

- [ ] 项目编码长度为 2-4 位大写字母
- [ ] 项目编码只包含字母 A-Z
- [ ] 项目编码体现项目或产品线特征
- [ ] 跨项目引用包含项目码前缀
- [ ] 同项目内简写与完整形式符合 `coding-system.md` 约定（索引与跨文档须完整编码）
- [ ] 类型码 REF 的细项计数与引用方式符合 `coding-system.md`（含全局 `*` 行说明）

### 编码分配合规性

- [ ] 编码计数器按项目码+类型码独立计数
- [ ] 序号从 001 开始，单调递增
- [ ] 废弃编码保留编号，标注"已废弃"
- [ ] 编码分配前检查冲突
- [ ] 编码引用格式正确（如 CRM-FR-001 用户登录功能）
- [ ] **编码含义锁定检查**：已分配编码的含义和标题未被修改
- [ ] **编码修改防护**：没有违反"永不修改编码含义"原则
- [ ] **废弃流程合规**：废弃编码正确标注，未重用编号
- [ ] **废弃信息完整**：包含废弃时间、原因、替代方案、废弃人
- [ ] **引用更新完整**：废弃细项的引用已正确处理
- [ ] **索引同步更新**：全局索引和文档清单已更新废弃状态
- [ ] **变更记录完整**：废弃变更已记录在变更记录中
- [ ] **即时索引更新**：新编码分配后立即更新所有相关索引
- [ ] **标题命名规范**：编码标题适度宽泛，避免过于具体
- [ ] **全局唯一性验证**：新编码在 ued/ 目录下无重复定义
- [ ] **编码计数器同步**：新编码分配后，计数器已递增且备注已更新
- [ ] **文档清单完整性**：文档末尾细项编码清单包含所有正文中的编码
- [ ] **全局索引完整性**：ued/README.md 全局索引包含所有文档的编码
- [ ] **索引一致性**：文档清单与全局索引的编码信息完全一致

### 内容完整性检查

- [ ] L0 文档包含愿景声明、商业背景、成功标准、范围边界
- [ ] L1 文档包含利益相关者分析、场景描述、高阶约束
- [ ] L2 文档包含功能需求、非功能需求、用例描述、验收标准
- [ ] L3 文档包含架构目标与约束、架构原则、子系统划分
- [ ] L4 文档包含领域模型、接口契约、模块职责
- [ ] L5 文档包含详细设计、算法逻辑、异常策略
- [ ] L6 文档包含测试策略、需求追溯矩阵、缺陷管理策略

### 引用关系合规性

- [ ] 交叉引用编码确实存在
- [ ] 引用方向正确（仅下层引用上层）
- [ ] 引用格式包含必要标题信息
- [ ] REF 引用来源信息完整（类型、来源、链接、获取日期）
- [ ] ADR 文档引用相关约束（CON-xxx、NFR-xxx）
- [ ] 需求追溯到验收标准（FR-xxx → AC-xxx）
- [ ] **标准引用格式**：优先使用 Markdown 链接 `[编码（标题）](#锚点)`（见 `coding-system.md`）；纯文本场景再用统一分隔形式
- [ ] **标题一致性**：引用标题与索引完全一致
- [ ] **引用完整性**：重要引用都包含完整标题信息
- [ ] **格式统一性**：同一文档内引用格式保持一致
- [ ] **Markdown 兼容性**：引用格式在 Markdown 中渲染正常

### 格式规范检查

- [ ] 文档编码格式正确（如 CRM-L0-001）
- [ ] 层级目录命名规范（L{N}-{英文短名}/）
- [ ] 子系统目录小写英文，短横线分隔
- [ ] 文件名不含中文，描述部分使用英文或拼音
- [ ] 版本号符合语义化规范（vX.Y）
- [ ] 日期格式统一（YYYY-MM-DD）
- [ ] 表格格式标准，列名清晰

### 流程图格式合规性

- [ ] 单行流程使用简化箭头形式（无分支、无跳转、无循环）
- [ ] 复杂流程使用 Mermaid 图表（有分支、跳转或循环）
- [ ] 多步骤流程使用表格形式，步骤清晰列示
- [ ] Mermaid 语法正确，可渲染
- [ ] 箭头符号使用标准格式（推荐 →，可接受 ->）

### SSoT 一致性检查

- [ ] 细项编码全局无冲突（无重复编码）
- [ ] 交叉引用的编码确实存在于系统中
- [ ] 引用方向正确（仅下层引用上层）

### 逐层合规性检查

从上层向下层逐级检查，确保下层不违背上层：

- [ ] **L1 vs L0**：利益相关者需求是否对齐战略目标（GOL-xxx）
- [ ] **L2 vs L1**：功能/非功能需求是否覆盖利益相关者场景（SCN-xxx）
- [ ] **L3 vs L2**：架构方案是否满足所有需求（FR-xxx, NFR-xxx）
- [ ] **L4 vs L3**：系统设计是否遵循架构原则（PRN-xxx）和约束（CON-xxx）
- [ ] **L5 vs L4**：详细设计是否符合接口契约（IF-xxx）和领域模型（DOM-xxx）
- [ ] **L6 vs L2**：验证场景是否追溯到需求（FR-xxx）和验收标准（AC-xxx）

## 常见问题检查

### 内容合规性

1. **代码混入**：检查是否包含 SQL、配置文件、脚本等具体实现
2. **编码冲突**：验证细项编码的全局唯一性
3. **引用错误**：检查交叉引用的编码是否存在
4. **层级违背**：确保下层设计不违背上层约束
5. **流程图格式合规性**：
   - 单行流程（无分支、无跳转、无循环）：允许使用简化箭头形式
   - 复杂流程（有分支、跳转或循环）：必须使用 Mermaid 图表
   - 多步骤流程：可使用表格形式，步骤清晰列示
6. **项目编码错误**：项目编码不符合 2-4 位大写字母规范
7. **配置文件缺失**：ued/.doc-config.json 不存在或格式错误
8. **编码生命周期违规**：删除或复用已分配的编码
9. **编码含义被修改**：已分配编码的含义或标题被擅自修改
10. **废弃编码重用**：废弃的编码被重新分配给新内容
11. **索引更新延迟**：新编码分配后未及时更新索引
12. **标题过于具体**：编码标题包含过多实现细节
13. **引用格式不规范**：引用编码时缺少标题或分隔符不当
14. **引用格式不统一**：同一文档内使用了多种不同的引用格式
15. **Markdown 渲染问题**：引用格式在 Markdown 中显示异常
16. **废弃信息缺失**：废弃细项缺少完整的废弃信息
17. **废弃引用未处理**：废弃细项的引用未正确更新
18. **废弃标记不规范**：废弃标记格式不符合规范要求

### 格式规范性

1. **元信息缺失**：检查文档头部的元信息表格
2. **编码清单**：验证文档末尾的细项编码清单
3. **变更记录**：确认变更记录的完整性
4. **命名规范**：检查文件名和目录名是否符合规范
5. **日期格式**：统一使用 YYYY-MM-DD 格式
6. **版本号格式**：符合语义化版本规范 vX.Y
7. **表格格式**：Markdown 表格语法正确，列名清晰
8. **Mermaid 语法**：图表语法正确，可正常渲染

### 内容完整性

1. **层级内容缺失**：各层级必需章节不完整
2. **追溯链断裂**：需求未追溯到验收标准或测试场景
3. **约束遗漏**：未识别或记录重要约束条件
4. **风险未评估**：关键风险未识别或无应对策略
5. **假定未声明**：设计依赖的前提条件未明确
6. **度量指标缺失**：关键成功标准缺乏量化指标

### 引用关系问题

1. **无效引用**：引用的编码不存在
2. **循环引用**：出现循环依赖关系
3. **跨层引用**：上层引用下层内容（违规）
4. **REF 信息不全**：外部参考资料缺少来源、链接等信息
5. **引用格式错误**：编码格式不符合规范
6. **标题信息缺失**：引用时缺少必要的标题说明

### 质量保证问题

1. **可追溯性不足**：需求无法追溯到验证方案
2. **完整性缺陷**：设计未覆盖所有必需场景
3. **一致性冲突**：相关文档之间存在矛盾
4. **可维护性差**：文档结构混乱，难以维护
5. **表述模糊**：关键信息表述不清晰或有歧义
6. **过度设计**：包含不必要的复杂性
7. **范围蔓延**：超出项目边界的内容

## 编码索引完整性校验

### 校验原则

**核心要求**：每个编码必须在以下三个位置同时建立，缺一不可：
1. 文档正文中定义位置（带锚点的标题）
2. 文档末尾的细项编码清单
3. ued/README.md 的全局编码索引

### 校验步骤

#### 步骤 1：检查文档正文的编码定义

**方法**：在文档中搜索所有编码格式的标题（如 `## CRM-FR-001 用户登录功能 {#crm-fr-001}`）

```bash
# 提取文档中所有编码标题
grep -E "^## [A-Z]+-[A-Z]+-[0-9]{3}" document.md
```

**检查项**：
- [ ] 每个编码标题包含项目码、类型码、三位序号
- [ ] 每个编码标题带锚点（如 `{#crm-fr-001}`）
- [ ] 锚点命名符合规范（全小写，短横线分隔）

#### 步骤 2：检查文档末尾的细项编码清单

**方法**：找到文档末尾的"细项编码清单"或"编码清单"章节

**检查项**：
- [ ] 清单包含正文中的所有编码
- [ ] 每个条目格式正确（编码 | 类型 | 标题）
- [ ] 编码、类型、标题与正文完全一致
- [ ] 无遗漏、无重复

#### 步骤 3：检查 ued/README.md 的全局索引

**方法**：打开 ued/README.md，找到对应的类型码表格（如 FR 类型码表格）

**检查项**：
- [ ] 新编码已在对应的类型码表格中
- [ ] 编码、标题与文档完全一致
- [ ] 编码计数器已递增（如 CRM-FR 下一个编号从 015 → 016）
- [ ] 备注已更新（如"已使用 001-015"）

#### 步骤 4：验证一致性

**方法**：对比以下三个来源

| 来源 | 编码数量 | 最小编码 | 最大编码 |
|------|---------|---------|---------|
| 文档正文 | | | |
| 文档清单 | | | |
| 全局索引 | | | |

**检查项**：
- [ ] 三个来源的编码数量一致
- [ ] 三个来源的最小编码和最大编码一致
- [ ] 三个来源的编码列表完全相同（无遗漏、无多余）

### 编码冲突检查（防止重复定义）

**方法**：在 ued/ 目录下搜索目标编码

```bash
# 搜索 ued/ 目录下所有包含特定编码的文件
grep -r "CRM-FR-015" ued/
```

**检查项**：
- [ ] 新编码在 ued/ 目录下无重复定义
- [ ] 如发现重复，判断是复用现有编码还是冲突
- [ ] 冲突时需重新分配编码

### 编码计数器验证

**方法**：对比编码计数器与实际使用情况

```bash
# 提取 ued/README.md 中的计数器信息
# 提取所有 .md 文件中实际使用的编码
# 对比计数器是否正确
```

**检查项**：
- [ ] 计数器的"下一个编号"未被使用
- [ ] 计数器的"最大编号"与实际最大编码一致
- [ ] 备注中的"已使用 XXX-YYY"与实际使用情况一致
- [ ] 没有跳号（如 001-015 但缺少 010）

### 常见问题识别

| 问题 | 表现 | 处理方法 |
|------|------|---------|
| 编码重复定义 | 同一编码在多个文档中定义 | 检查是否为同一定义的多处引用，否则需重新分配 |
| 编码清单不完整 | 正文有编码但清单缺失 | 补充到清单中 |
| 全局索引缺失 | 编码在文档清单但全局索引缺失 | 补充到全局索引中 |
| 计数器未更新 | 新编码已使用但计数器未递增 | 更新计数器和备注 |
| 标题不一致 | 不同位置标题文字不同 | 统一为规范标题，不得修改已分配编码的标题 |
| 锚点缺失 | 编码标题不带锚点 | 补充锚点 `{#编码小写}` |
| 锚点命名错误 | 锚点大小写或格式错误 | 修正为全小写短横线格式 |

### 校验清单总结

#### 文档级别校验
- [ ] 正文所有编码都带锚点
- [ ] 清单包含正文所有编码
- [ ] 清单编码与正文完全一致
- [ ] 无遗漏、无重复

#### 全局级别校验
- [ ] 新编码已添加到 ued/README.md 对应类型码表格
- [ ] 全局索引信息与文档完全一致
- [ ] 编码计数器已正确递增
- [ ] 计数器备注已更新
- [ ] 全局编码无冲突

#### 一致性校验
- [ ] 正文、清单、索引三方一致
- [ ] 编号连续无跳号
- [ ] 项目码、类型码正确
- [ ] 锚点命名符合规范

## 审核流程总结

### 第一阶段：自动化检查（可执行验证）

1. **状态值合规性**：检查状态值是否在标准集合中
2. **元信息完整性**：验证必要字段是否存在
3. **编码格式检查**：验证编码格式是否符合规范
4. **文件命名检查**：确保文件名和目录名符合规范
5. **配置文件验证**：检查 ued/.doc-config.json 存在性和格式

### 第二阶段：内容审核（人工检查）

1. **代码混入检查**：确保无具体实现内容
2. **引用关系验证**：检查交叉引用的有效性和方向
3. **内容完整性**：验证各层级必需章节
4. **层级一致性**：确保下层不违背上层
5. **流程图格式**：验证图表格式符合复杂度要求

### 第三阶段：质量保证（综合评估）

1. **可追溯性**：验证需求到验证的完整链路
2. **一致性检查**：跨文档的一致性验证
3. **可维护性**：文档结构和表述的清晰度
4. **完整性评估**：设计覆盖范围的充分性

### 审核结果处理

| 审核结果 | 处理方式 | 责任人 |
|----------|----------|--------|
| 通过 | 文档状态更新为"正式" | 审核人 |
| 需修改 | 返回作者修改，明确修改项 | 审核人、作者 |
| 严重问题 | 重新设计，重大问题需讨论 | 审核人、架构师 |

### 审核工具支持

1. **编码冲突检查**：搜索全局编码重复
2. **引用有效性验证**：检查引用编码存在性
3. **格式规范检查**：自动化验证命名和格式
4. **内容完整性检查**：根据模板验证章节完整性

### 审核记录模板

```
审核记录
文档编号：CRM-L2-001
审核日期：2024-XX-XX
审核人：XXX
审核结果：通过/需修改/不通过

检查项：
✅ 必选项检查 - 全部通过
✅ 合规性检查 - 全部通过  
⚠️ 内容完整性 - 发现 2 个问题
✅ 引用关系 - 全部正确

问题列表：
1. 验收标准 AC-003 缺少量化指标
2. 风险 RSK-002 缺少应对策略

建议：
1. 补充 AC-003 的量化指标
2. 完善 RSK-002 的风险应对策略

后续行动：
□ 作者修改后重新审核
□ 直接通过
□ 需要架构师介入
```
