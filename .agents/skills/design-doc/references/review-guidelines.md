# 设计文档审核规范

## 审核流程

审核设计文档时，按以下检查项逐层执行：

同一含义的规则以最贴近的章节为主检查一次；后续章节只保留补充性检查点，避免重复罗列。

### 必选项检查（可自动化验证）

#### 状态值合规性
- [ ] **L0-L6 文档状态** ∈ {草稿, 正式, 废弃}
- [ ] **ADR 状态** ∈ {提议, 采纳, 拒绝, 废弃, 取代}
- [ ] **REF 状态** ∈ {有效, 废弃}
- [ ] **需求追溯状态** ∈ {待验证, 已验证, 失败}

#### 元信息完整性
- [ ] 文档编码存在且格式正确（如 `CRM-L0-001`，含项目码与层级）
- [ ] L0-L6 文档的层级字段与文档类型匹配；REF 文档的类别字段为 `REF - 外部参考资料`
- [ ] 版本字段检查：L0-L6 文档的版本号符合语义化规范（vX.Y）；ADR/REF 的变更记录版本号格式一致
- [ ] 状态字段检查：仅在存在状态字段时，状态值从对应标准集合中选择
- [ ] 作者或录入人字段非空
- [ ] 日期字段完整：L0-L6 需创建日期和最后更新；ADR 需日期；REF 需录入日期
- [ ] 当前作用域 `README.md` 含必要元信息，至少可识别 `doc_mode`、`project_name` 与 `project_code` 或其缺省状态

#### 编码体系合规性
- [ ] 细项编码全局唯一（无重复）
- [ ] 编码格式符合类型码规范
- [ ] 交叉引用编码确实存在
- [ ] 引用方向正确（仅下层引用上层）

### 合规性检查

- [ ] **术语一致性（可选触发）**：专有名词密集且易造成理解偏差时，是否具备项目词汇表、文档内「术语与概念」小节或文内首次定义（见 [glossary-conventions.md](glossary-conventions.md)）；无此类风险时可跳过
- [ ] 不存在可执行代码、脚本、SQL、配置片段等实现内容（核心约束）
- [ ] 目录结构和文件命名合规
- [ ] 文件名不含中文；文档内容使用中文撰写（二者不可混淆）
- [ ] 若正文定义了明确细项编码，则细项编码清单完整且与正文一致
- [ ] 变更记录存在且最新；仅示例片段或无需维护版本历史的清单型模板可按模板约定处理
- [ ] 项目元信息来源正确：使用作用域 `README.md` 作为配置源，且未再引入额外配置文件
- [ ] 编码生命周期规则遵守（永不删除、永不复用）
- [ ] 真值源优先级遵守（本 skill 包内 `references/` 专项文件优先于 `SKILL.md`）

### 项目编码合规性

- [ ] 项目编码长度为 2-4 位大写字母
- [ ] 项目编码只包含字母 A-Z
- [ ] 项目编码体现项目或产品线特征
- [ ] 跨项目引用包含项目码前缀
- [ ] 同项目内简写与完整形式符合 `coding-system.md` 约定（索引与跨文档须完整编码）
- [ ] 类型码 REF 的细项计数与引用方式符合 `coding-system.md`（含全局 `*` 行说明）
- [ ] 单应用模式下，`ued/README.md` 承担项目元信息、计数器和索引
- [ ] 多应用模式下，顶层 `ued/README.md` 包含应用注册表，目标应用 `ued/{app-name}/README.md` 承担该应用元信息、计数器和索引
- [ ] 多应用模式下，顶层 `ued/README.md` 未被误用为某个应用的细项索引真值源

### 编码分配合规性

- [ ] 编码计数器按项目码+类型码独立计数
- [ ] 序号从 001 开始，单调递增
- [ ] 废弃编码保留编号，标注"已废弃"
- [ ] 编码分配前检查冲突
- [ ] 编码引用格式正确（如 CRM-FR-001 用户登录功能）
- [ ] **编码含义锁定检查**：已分配编码的含义和标题未被修改
- [ ] **编码修改防护**：没有违反"永不修改编码含义"原则
- [ ] **废弃流程合规**：废弃编码正确标注，未重用编号
- [ ] **废弃信息完整**：包含细项状态、废弃时间、废弃原因、替代方案四个必须字段
- [ ] **引用更新完整**：废弃细项的引用已正确处理
- [ ] **索引同步更新**：全局索引和文档清单已更新废弃状态
- [ ] **变更记录完整**：废弃变更已记录在变更记录中
- [ ] **废弃标题标记**：废弃细项标题包含 `~~已废弃~~` 删除线
- [ ] **废弃必须字段**：包含细项状态、废弃时间、废弃原因、替代方案四个必须字段
- [ ] **废弃时间格式**：废弃时间格式为 YYYY-MM-DD
- [ ] **废弃原因规范**：废弃原因属于标准分类（需求变更、设计优化、技术升级等）
- [ ] **替代方案规范**：有替代方案填写编码，无替代方案填写"无"
- [ ] **即时索引更新**：新编码分配后立即更新所有相关索引
- [ ] **标题命名规范**：编码标题适度宽泛，避免过于具体
- [ ] **全局唯一性验证**：新编码在 ued/ 目录下无重复定义
- [ ] **编码计数器同步**：新编码分配后，计数器已递增且备注已更新
- [ ] **文档清单完整性**：若正文定义了明确细项编码，文档末尾细项编码清单包含所有正文中的编码
- [ ] **文档清单完整编码**：若存在细项编码清单，其编码均为完整编码形式
- [ ] **全局索引完整性**：当前作用域 `README.md` 的全局索引包含所有已正式分配并需纳入索引的编码
- [ ] **索引一致性**：对正文定义的细项编码，文档清单与全局索引的编码信息完全一致

### 内容完整性检查

- [ ] L0 文档包含愿景声明、商业背景、成功标准、范围边界
- [ ] L1 文档包含利益相关者分析、场景描述、高阶约束
- [ ] L2 文档包含功能需求、非功能需求、用例描述、验收标准
- [ ] L3 文档包含架构目标与约束、架构原则、子系统划分
- [ ] L4 文档包含领域模型、接口契约、模块职责
- [ ] L5 文档包含详细设计、算法逻辑、异常策略
- [ ] L6 文档包含测试策略、需求追溯矩阵、缺陷管理策略

### 细项定义清晰性检查（概念是否明确）

先按 `coding-system.md` 的“细项定义最小要素（清晰性基线）— 适用性边界（按类型分组）”判定检查强度，避免对 REF/GOL/STK 等类型机械套用同一标准。

- [ ] **一句话定义清晰**：每个新增/变更细项可用 1 句话解释“是什么”
- [ ] **边界明确**：包含适用范围，且至少有 1 条“非目标/不适用范围”
- [ ] **触发条件明确**：细项何时生效、由谁触发、前置条件是什么
- [ ] **可验证口径存在**：存在可验证标准（可直接写指标，或引用 AC/TC/MET）
- [ ] **追溯关系完整**：细项至少可追溯到一个上层依据（如 GOL/SCN/FR/NFR/PRN）
- [ ] **术语歧义可控**：高歧义术语在文内首次定义或引用项目词汇表

补充判定：
- [ ] **REF 例外正确处理**：REF 以来源说明、适用边界、被支撑细项追溯为主，不强制要求触发条件
- [ ] **目标类不误判**：GOL/STK/SCN/DOM/CMP 若无显式触发条件，不应仅因缺少 When 被判不合格

### 引用关系合规性

- [ ] 引用格式包含必要标题信息
- [ ] REF 引用来源信息完整（类型、来源、链接、获取日期）
- [ ] ADR 文档引用相关约束（CON-xxx、NFR-xxx）
- [ ] 需求追溯到验收标准（FR-xxx → AC-xxx）
- [ ] **标准引用格式**：优先使用 Markdown 链接 `[编码（标题）](#锚点)`（见 `coding-system.md`）；纯文本场景再用统一分隔形式
- [ ] **标题一致性**：引用标题与索引完全一致
- [ ] **引用完整性**：重要引用都包含完整标题信息
- [ ] **格式统一性**：同一文档内引用格式保持一致
- [ ] **Markdown 兼容性**：引用格式在 Markdown 中渲染正常

### IF / PLN 类型使用合理性检查

- [ ] **IF 不被误限于 HTTP API**：事件、Webhook、文件交换、内部服务边界等契约可使用 IF
- [ ] **API 分类一致**：对外/对内 API 使用 IF 编码并通过“接口风格”区分，不新增独立 API 类型码
- [ ] **IF/ACT 不重叠误用**：契约语义用 IF，动作语义用 ACT；若同一场景两者都需要，应双编码并建立追溯
- [ ] **PLN 使用边界正确**：仅用于阶段性计划对象，不替代已稳定的 FR/NFR/IF/ALG 定义
- [ ] **PLN 迁移闭环**：PLN 存在目标类型迁移记录（迁移为 FR/NFR/PRN/DEC/CMP/IF/ACT/ALG 等）

### 格式规范检查

- [ ] 层级目录命名规范（L{N}-{英文短名}/）
- [ ] 应用目录小写英文，短横线分隔
- [ ] 文件名不含中文，描述部分使用英文或拼音
- [ ] 日期格式统一（YYYY-MM-DD）
- [ ] 表格格式标准，列名清晰

### 废弃格式规范检查

#### 标题标记规范
- [ ] 废弃细项标题包含 `~~已废弃~~` 删除线（必须）
- [ ] 不使用粗体括号 `(已废弃)` 等其他格式

#### 必须字段检查
- [ ] 废弃细项包含 `**细项状态**：已废弃` 字段
- [ ] 废弃细项包含 `**废弃时间**：YYYY-MM-DD` 字段
- [ ] 废弃细项包含 `**废弃原因**：{具体原因}` 字段
- [ ] 废弃细项包含 `**替代方案**：{编码} 或 "无"`
- [ ] 废弃时间格式正确（YYYY-MM-DD）
- [ ] 废弃原因清晰明确（至少说明废弃的简要原因）

#### 索引更新检查
- [ ] 全局索引中该细项状态更新为"已废弃"
- [ ] 文档清单中该细项状态更新为"已废弃"
- [ ] 索引中包含废弃时间信息
- [ ] 索引备注中说明替代方案（如有）

### 流程图格式合规性

- [ ] 单行流程使用简化箭头形式（无分支、无跳转、无循环）
- [ ] 复杂流程使用 Mermaid 图表（有分支、跳转或循环）
- [ ] 多步骤流程使用表格形式，步骤清晰列示
- [ ] Mermaid 语法正确，可渲染
- [ ] 箭头符号使用标准格式（推荐 →，可接受 ->）

### SSoT 一致性检查

- [ ] 本文前述编码唯一性、引用存在性、引用方向、索引同步等检查项结果一致，且不存在跨章节结论冲突

### 逐层合规性检查

从上层向下层逐级检查，确保下层不违背上层：

- [ ] **L1 vs L0**：利益相关者需求是否对齐战略目标（GOL-xxx）
- [ ] **L2 vs L1**：功能/非功能需求是否覆盖利益相关者场景（SCN-xxx）
- [ ] **L3 vs L2**：架构方案是否满足所有需求（FR-xxx, NFR-xxx）
- [ ] **L4 vs L3**：系统设计是否遵循架构原则（PRN-xxx）和约束（CON-xxx）
- [ ] **L5 vs L4**：详细设计是否符合接口契约（IF-xxx）和领域模型（DOM-xxx）
- [ ] **L6 vs L2**：验证场景是否追溯到需求（FR-xxx）和验收标准（AC-xxx）

## 常见问题检查

- **内容违规**：混入代码、脚本、SQL、配置片段或其他实现内容；流程图形式与复杂度不匹配
- **编码违规**：项目编码不合法、细项编码冲突、编码含义被修改、废弃编码重用、标题过度具体
- **索引违规**：正文定义的细项编码与文档清单、当前作用域 `README.md` 全局索引不同步；计数器未递增；废弃信息缺失
- **引用违规**：引用编码不存在、引用方向错误、格式不统一、标题缺失、Markdown 锚点渲染异常、REF 信息不完整
- **结构违规**：元信息缺失、变更记录缺失、命名不规范、日期或版本格式错误、表格或 Mermaid 语法不合法
- **完整性缺陷**：层级必需章节缺失、追溯链断裂、约束/假定/风险/度量指标遗漏
- **质量问题**：文档间结论冲突、表述模糊、结构混乱、过度设计、范围蔓延

## 编码索引完整性校验

### 校验原则

**核心要求**：对**正文中定义的细项编码**，必须在以下三个位置同时建立，缺一不可：
1. 文档正文中定义位置（带锚点的标题）
2. 文档末尾的细项编码清单
3. 当前作用域 `README.md` 的全局编码索引

### 校验步骤

#### 步骤 1：检查文档正文的编码定义

**方法**：在文档中搜索所有编码格式的标题（如 `## CRM-FR-001 用户登录功能 {#crm-fr-001}`）

可使用编辑器搜索、仓库搜索工具或自定义脚本提取这些标题；本文不绑定特定平台命令。

**检查项**：
- [ ] 每个编码标题包含项目码、类型码、三位序号
- [ ] 每个编码标题带锚点（如 `{#crm-fr-001}`）
- [ ] 锚点命名符合规范（全小写，短横线分隔）

#### 步骤 2：检查文档末尾的细项编码清单

仅适用于正文中定义了明确细项编码的文档。

**方法**：找到文档末尾的"细项编码清单"或"编码清单"章节

**检查项**：
- [ ] 清单包含正文中的所有编码
- [ ] 每个条目格式正确（编码 | 类型 | 标题）
- [ ] 编码、类型、标题与正文完全一致
- [ ] 清单中的编码均为完整编码
- [ ] 无遗漏、无重复

#### 步骤 3：检查当前作用域 `README.md` 的全局索引

**方法**：打开当前作用域 `README.md`，找到对应的类型码表格（如 FR 类型码表格）

**检查项**：
- [ ] 新编码已在对应的类型码表格中
- [ ] 编码、标题与文档完全一致
- [ ] 编码计数器已递增（如 CRM-FR 下一个编号从 015 → 016）
- [ ] 备注已更新（如"已使用 001-015"）

#### 步骤 4：验证一致性

**方法**：对比以下三个来源

| 来源 | 编码数量 | 最小编码 | 最大编码 |
|------|---------|---------|---------|
| 文档正文 | | | |
| 文档清单 | | | |
| 全局索引 | | | |

**检查项**：
- [ ] 三个来源的编码数量一致
- [ ] 三个来源的最小编码和最大编码一致
- [ ] 三个来源的编码列表完全相同（无遗漏、无多余）

### 编码冲突检查（防止重复定义）

**方法**：在 ued/ 目录下搜索目标编码

可使用编辑器全文搜索、仓库搜索工具或自定义脚本检索目标编码。

**检查项**：
- [ ] 新编码在 ued/ 目录下无重复定义
- [ ] 如发现重复，判断是复用现有编码还是冲突
- [ ] 冲突时需重新分配编码

### 编码计数器验证

**方法**：对比编码计数器与实际使用情况

建议提取当前作用域 `README.md` 的计数器信息，再对比所有 Markdown 文档中的实际编码使用情况；实现方式可按团队环境选择。

**检查项**：
- [ ] 计数器的"下一个编号"未被使用
- [ ] 计数器的"最大编号"与实际最大编码一致
- [ ] 备注中的"已使用 XXX-YYY"与实际使用情况一致
- [ ] 没有跳号（如 001-015 但缺少 010）

### 常见问题识别

| 问题 | 表现 | 处理方法 |
|------|------|---------|
| 编码重复定义 | 同一编码在多个文档中定义 | 检查是否为同一定义的多处引用，否则需重新分配 |
| 编码清单不完整 | 正文有编码但清单缺失 | 补充到清单中 |
| 全局索引缺失 | 编码在文档清单但全局索引缺失 | 补充到全局索引中 |
| 计数器未更新 | 新编码已使用但计数器未递增 | 更新计数器和备注 |
| 标题不一致 | 不同位置标题文字不同 | 统一为规范标题，不得修改已分配编码的标题 |
| 锚点缺失 | 编码标题不带锚点 | 补充锚点 `{#编码小写}` |
| 锚点命名错误 | 锚点大小写或格式错误 | 修正为全小写短横线格式 |

### 校验清单总结

#### 文档级别校验
- [ ] 正文所有编码都带锚点
- [ ] 清单包含正文所有编码
- [ ] 清单编码与正文完全一致
- [ ] 无遗漏、无重复

#### 全局级别校验
- [ ] 新编码已添加到当前作用域 `README.md` 对应类型码表格
- [ ] 全局索引信息与文档完全一致
- [ ] 编码计数器已正确递增
- [ ] 计数器备注已更新
- [ ] 全局编码无冲突

#### 一致性校验
- [ ] 正文、清单、索引三方一致
- [ ] 编号连续无跳号
- [ ] 项目码、类型码正确
- [ ] 锚点命名符合规范

## 审核流程总结

### 第一阶段：自动化检查（可自动化验证）

1. **状态值合规性**：检查状态值是否在标准集合中
2. **元信息完整性**：验证必要字段是否存在
3. **编码格式检查**：验证编码格式是否符合规范
4. **文件命名检查**：确保文件名和目录名符合规范
5. **README 配置验证**：检查作用域 `README.md` 中的元信息区块、编码模式与应用注册信息

### 第二阶段：内容审核（人工检查）

1. **代码混入检查**：确保无可执行代码和实现片段，但允许算法、流程、状态机、逻辑与模型描述
2. **引用关系验证**：检查交叉引用的有效性和方向
3. **内容完整性**：验证各层级必需章节
4. **层级一致性**：确保下层不违背上层
5. **流程图格式**：验证图表格式符合复杂度要求

### 第三阶段：质量保证（综合评估）

1. **可追溯性**：验证需求到验证的完整链路
2. **一致性检查**：跨文档的一致性验证
3. **可维护性**：文档结构和表述的清晰度
4. **完整性评估**：设计覆盖范围的充分性

### 审核结果处理

| 审核结果 | 处理方式 | 责任人 |
|----------|----------|--------|
| 通过 | 文档状态更新为"正式" | 审核人 |
| 需修改 | 返回作者修改，明确修改项 | 审核人、作者 |
| 严重问题 | 重新设计，重大问题需讨论 | 审核人、架构师 |

### 审核工具支持

1. **编码冲突检查**：搜索全局编码重复
2. **引用有效性验证**：检查引用编码存在性
3. **格式规范检查**：自动化验证命名和格式
4. **内容完整性检查**：根据模板验证章节完整性
5. **平台建议**：优先采用仓库搜索、编辑器诊断、自定义脚本或 CI 检查，不把规范绑定到某一类 shell 命令
