# 废弃文档操作规范

## 概述

本文档规范了整份设计文档的废弃操作流程。废弃文档是项目演进过程中的正常现象，但需要规范的流程确保可追溯性和安全性。

**与「废弃细项」的区别**：
- **废弃细项**：指文档内某个编码细项（如 FR-001、RUL-005）的作废，参见 [deprecation-guide.md](deprecation-guide.md)
- **废弃文档**：指整份设计文档（如 L2-001 系统需求文档）的作废，是本文档规范的内容

## 核心原则

**1. 保留原则**：废弃的文档必须保留，不能直接删除
**2. 隔离原则**：废弃文档应移动到独立目录，避免与活跃文档混淆
**3. 标注原则**：清晰标注文档的废弃状态和废弃原因
**4. 追溯原则**：保留废弃历史，便于追溯和审计
**5. 清理原则**：定期清理过期的废弃文档，避免累积过多

## 废弃文档的判断标准

**建议废弃整份文档的场景**：

| 场景类型 | 说明 | 示例 |
|---------|------|------|
| 项目终止 | 项目被取消或暂停，相关文档不再维护 | CRM项目终止，废弃所有相关文档 |
| 版本替换 | 文档内容被新版本完全替代 | V1.0版本的需求文档被V2.0版本替代 |
| 架构重构 | 整体架构调整，原设计文档失效 | 从单体架构迁移到微服务架构，原L3架构文档失效 |
| 业务调整 | 业务方向重大调整，原设计文档不再适用 | 业务从B2C转向B2B，原用户画像文档失效 |
| 合并重组 | 多个文档合并为一个新文档 | 两个子系统的需求文档合并为统一文档 |

**不建议废弃整份文档的场景**：
- 仅修改部分内容（应该使用细项废弃流程）
- 文档中的少量细项失效（应该废弃细项，保留文档）
- 临时性调整（应该使用文档状态"草稿"）

## 废弃文档操作流程

### 第一步：评估废弃必要性

**检查清单**：
- [ ] 确认文档是否真的需要整体废弃，而非仅修改部分内容
- [ ] 确认是否有新文档替代（新文档编码）
- [ ] 确认废弃原因和影响范围
- [ ] 确认是否有其他文档引用此文档

**评估示例**：
```
文档编码：CRM-L2-001 用户管理系统需求
评估结果：需要废弃
原因：用户管理系统被整体重构，原需求不再适用
替代方案：CRM-L2-005 新用户管理系统需求
引用文档：CRM-L3-001、CRM-L4-002
```

### 第二步：准备废弃信息

**收集必要信息**：

| 信息项 | 说明 | 示例 |
|-------|------|------|
| 文档编码 | 待废弃的文档编码 | CRM-L2-001 |
| 文档标题 | 文档的完整标题 | 用户管理系统需求 |
| 废弃时间 | 废弃操作的日期 | 2024-03-23 |
| 废弃原因 | 废弃的详细原因 | 系统重构，需求完全变更 |
| 替代文档 | 新的替代文档编码 | CRM-L2-005 |
| 影响范围 | 受影响的其他文档列表 | CRM-L3-001, CRM-L4-002 |

### 第三步：移动文档到 deprecated 目录

**目录结构**：

```
ued/
├── active/                    # 活跃文档目录
│   ├── L0-001 战略愿景.md
│   ├── L1-001 利益相关者.md
│   └── ...
├── deprecated/                # 废弃文档目录
│   ├── 2024/
│   │   ├── 03/
│   │   │   ├── CRM-L2-001_用户管理系统需求.md
│   │   │   └── README.md
│   │   └── ...
│   └── README.md
└── README.md                  # 全局索引
```

**命名规范**：
- 文件名格式：`{文档编码}_{文档标题}.md`
- 日期目录：按年/月组织（如 `2024/03/`）
- 特殊字符处理：标题中的空格替换为下划线，移除特殊字符

**移动命令**：
```bash
# 创建目录结构（如果不存在）
mkdir -p ued/deprecated/2024/03

# 移动文档
mv ued/active/CRM-L2-001_用户管理系统需求.md ued/deprecated/2024/03/

# 重命名文档（添加废弃标记）
mv ued/deprecated/2024/03/CRM-L2-001_用户管理系统需求.md \
   ued/deprecated/2024/03/CRM-L2-001_用户管理系统需求_已废弃.md
```

### 第四步：添加废弃标记

**在废弃文档中添加标记**：

在文档开头添加废弃信息块：

```markdown
---
⚠️ **本文档已废弃**
- 废弃时间：2024-03-23
- 废弃原因：系统重构，需求完全变更
- 替代文档：[CRM-L2-005 新用户管理系统需求](../active/CRM-L2-005_新用户管理系统需求.md)
- 影响范围：CRM-L3-001, CRM-L4-002

请参考替代文档。本文档仅保留用于历史追溯。
---
```

**更新文档元信息**：

```markdown
| 文档编号 | CRM-L2-001 |
| 文档状态 | ~~草稿/正式~~ **已废弃** |
```

### 第五步：更新引用文档

**处理引用关系**：

1. **搜索引用**：在 `ued/active/` 目录中搜索对废弃文档的引用
2. **更新引用**：将引用更新为替代文档，或添加废弃标记
3. **更新索引**：更新全局索引中的文档状态

**引用处理示例**：

```markdown
# 废弃前
参见 [CRM-L2-001 用户管理系统需求](../active/CRM-L2-001_用户管理系统需求.md)

# 废弃后（有替代文档）
参见 ~~[CRM-L2-001 用户管理系统需求](../deprecated/2024/03/CRM-L2-001_用户管理系统需求.md)~~（已废弃）
和 [CRM-L2-005 新用户管理系统需求](../active/CRM-L2-005_新用户管理系统需求.md)

# 废弃后（无替代文档）
参见 ~~[CRM-L2-001 用户管理系统需求](../deprecated/2024/03/CRM-L2-001_用户管理系统需求.md)~~（已废弃，无替代）
```

### 第六步：更新全局索引

**更新 ued/README.md**：

```markdown
## 废弃文档

| 文档编码 | 文档标题 | 废弃日期 | 废弃原因 | 替代文档 | 影响范围 |
|---------|---------|---------|---------|---------|---------|
| CRM-L2-001 | 用户管理系统需求 | 2024-03-23 | 系统重构 | CRM-L2-005 | CRM-L3-001, CRM-L4-002 |
```

### 第七步：记录废弃操作

**在废弃日志中记录**：

```markdown
## 文档废弃记录

| 日期 | 文档编码 | 文档标题 | 废弃原因 | 替代文档 | 操作人 |
|------|---------|---------|---------|---------|--------|
| 2024-03-23 | CRM-L2-001 | 用户管理系统需求 | 系统重构 | CRM-L2-005 | 张三 |
```

## 定期清理策略

### 清理周期建议

| 文档类型 | 保留期限 | 清理策略 |
|---------|---------|---------|
| 临时文档 | 3个月 | 直接删除 |
| 项目文档 | 6个月 | 评估后删除或归档 |
| 架构文档 | 1年 | 归档到长期存储 |
| 核心文档 | 2年 | 归档到永久存储 |

### 清理操作流程

**1. 评估清理清单**：
```bash
# 查找过期的废弃文档
find ued/deprecated -name "*.md" -mtime +180
```

**2. 确认清理必要性**：
- [ ] 确认文档已超过保留期限
- [ ] 确认无历史追溯需求
- [ ] 确认相关项目已结束或文档已归档

**3. 执行清理或归档**：
```bash
# 删除过期的临时文档
rm ued/deprecated/2023/09/TEMP-001_临时设计.md

# 归档重要的废弃文档
tar -czf archive/2024_q1_deprecated.tar.gz ued/deprecated/2024/01 ued/deprecated/2024/02 ued/deprecated/2024/03
```

**4. 更新清理记录**：
```markdown
## 文档清理记录

| 清理日期 | 文档数量 | 清理方式 | 操作人 |
|---------|---------|---------|--------|
| 2024-04-01 | 5 | 删除 | 李四 |
| 2024-04-01 | 10 | 归档 | 李四 |
```

## 废弃文档恢复

**恢复流程**（如果需要恢复废弃的文档）：

1. **定位文档**：在 `deprecated` 目录中找到文档
2. **评估恢复必要性**：确认文档是否需要恢复到活跃状态
3. **移动回 active 目录**：
   ```bash
   mv ued/deprecated/2024/03/CRM-L2-001_用户管理系统需求_已废弃.md \
      ued/active/CRM-L2-001_用户管理系统需求.md
   ```
4. **移除废弃标记**：删除文档开头的废弃信息块
5. **更新文档状态**：将状态改为"草稿"或"正式"
6. **更新引用**：恢复或更新引用关系
7. **更新索引**：从废弃文档索引中移除，添加到活跃文档索引

## 目录管理最佳实践

### 1. 目录组织

```
ued/
├── active/                    # 活跃文档（默认目录）
├── deprecated/                # 废弃文档
│   ├── 2024/
│   │   ├── 01/
│   │   ├── 02/
│   │   └── ...
│   └── README.md              # 废弃文档索引
└── archive/                   # 长期归档（可选）
    ├── 2023/
    └── ...
```

### 2. README.md 维护

**deprecated/README.md** 模板：

```markdown
# 废弃文档索引

本目录存放已废弃的设计文档，按废弃日期组织。

## 按日期索引

### 2024年
- [03月](./2024/03/) - 5个文档
- [02月](./2024/02/) - 3个文档
- [01月](./2024/01/) - 2个文档

### 2023年
- [12月](./2023/12/) - 8个文档

## 按类型索引

|---------|---------|---------|---------|

### 专项文档
| 文档编码 | 文档标题 | 废弃日期 | 替代文档 |
|---------|---------|---------|---------|
| CRM-ADR-005 | 缓存策略决策 | 2024-02-15 | CRM-ADR-008 |

## 清理计划

| 季度 | 计划清理文档数 | 预计清理日期 | 负责人 |

| 2024-Q2 | 15 | 2024-07-01 | 李四 |

## 注意事项

- 废弃文档仅用于历史追溯，不应继续维护
- 需要恢复文档时，请参考"废弃文档恢复"流程
- 定期清理过期的废弃文档，避免占用过多存储空间
```

## 注意事项

**1. 禁止直接删除**：
- ❌ 直接删除文档
- ✅ 先移动到 deprecated 目录，再根据清理策略处理

**2. 保持引用完整**：
- ❌ 废弃文档后不管引用关系
- ✅ 更新所有引用文档，确保引用关系清晰

**3. 记录详细信息**：
- ❌ 废弃时只改文档状态
- ✅ 记录废弃时间、原因、替代方案、影响范围

**4. 定期清理**：
- ❌ 废弃文档无限期堆积
- ✅ 按清理策略定期清理或归档

**5. 恢复流程规范**：
- ❌ 随意从 deprecated 目录复制文档
- ✅ 按照恢复流程正式恢复文档

## 工具支持

**使用文档检查脚本**：

```bash
# 检查废弃文档的引用完整性
python scripts/check_docs.py -p ued/deprecated

# 检查活跃文档中对废弃文档的引用
python scripts/check_docs.py -p ued/active
```

**清理脚本建议**（可扩展）：

```bash
# 查找可清理的废弃文档
find ued/deprecated -name "*.md" -mtime +180

# 统计废弃文档数量
find ued/deprecated -name "*.md" | wc -l

# 生成废弃文档报告
find ued/deprecated -name "*.md" -exec ls -lh {} \; > deprecated_report.txt
```
