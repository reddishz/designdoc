# 废弃文档操作规范

## 概述

本文档规范了整份设计文档的废弃操作流程。废弃文档是项目演进过程中的正常现象，但需要规范的流程确保可追溯性和安全性。

**与「废弃细项」的区别**：
- **废弃细项**：指文档内某个编码细项（如 FR-001、RUL-005）的作废，参见 [deprecation-guide.md](deprecation-guide.md)
- **废弃文档**：指整份设计文档（如 L2-001 系统需求文档）的作废，是本文档规范的内容

## 核心原则

**1. 保留原则**：废弃的文档必须保留，不能直接删除
**2. 隔离原则**：废弃文档应移动到独立目录，避免与活跃文档混淆
**3. 标注原则**：清晰标注文档的废弃状态和废弃原因
**4. 追溯原则**：保留废弃历史，便于追溯和审计
**5. 清理原则**：定期清理过期的废弃文档，避免累积过多

## 废弃文档的判断标准

**建议废弃整份文档的场景**：

| 场景类型 | 说明 | 示例 |
|---------|------|------|
| 项目终止 | 项目被取消或暂停，相关文档不再维护 | CRM项目终止，废弃所有相关文档 |
| 版本替换 | 文档内容被新版本完全替代 | V1.0版本的需求文档被V2.0版本替代 |
| 架构重构 | 整体架构调整，原设计文档失效 | 从单体架构迁移到微服务架构，原L3架构文档失效 |
| 业务调整 | 业务方向重大调整，原设计文档不再适用 | 业务从B2C转向B2B，原用户画像文档失效 |
| 合并重组 | 多个文档合并为一个新文档 | 两个子系统的需求文档合并为统一文档 |

**不建议废弃整份文档的场景**：
- 仅修改部分内容（应该使用细项废弃流程）
- 文档中的少量细项失效（应该废弃细项，保留文档）
- 临时性调整（应该使用文档状态"草稿"）

## 废弃文档操作流程

### 第一步：评估废弃必要性

**检查清单**：
- [ ] 确认文档是否真的需要整体废弃，而非仅修改部分内容
- [ ] 确认是否有新文档替代（新文档编码）
- [ ] 确认废弃原因和影响范围
- [ ] 确认是否有其他文档引用此文档

**评估示例**：
```
文档编码：CRM-L2-001 用户管理系统需求
评估结果：需要废弃
原因：用户管理系统被整体重构，原需求不再适用
替代方案：CRM-L2-005 新用户管理系统需求
引用文档：CRM-L3-001、CRM-L4-002
```

### 第二步：准备废弃信息

**收集必要信息**：

| 信息项 | 说明 | 示例 |
|-------|------|------|
| 文档编码 | 待废弃的文档编码 | CRM-L2-001 |
| 文档标题 | 文档的完整标题 | 用户管理系统需求 |
| 废弃时间 | 废弃操作的日期 | 2024-03-23 |
| 废弃原因 | 废弃的详细原因 | 系统重构，需求完全变更 |
| 替代文档 | 新的替代文档编码 | CRM-L2-005 |
| 影响范围 | **预评估的受影响文档列表** | **CRM-L3-001, CRM-L4-002** |
| **建议删除日期** | **根据文档类型计算的建议清理日期** | **2024-09-23（6个月后）** |

### 第三步：引用检查（MANDATORY - 移动前必须执行）

**在移动文档到 deprecated 目录之前，必须先排查引用关系**：

**引用排查命令**：
```bash
# 在 active 目录中搜索对废弃文档的引用
grep -r "CRM-L2-001" ued/active/

# 搜索文档编码和文档标题
grep -r "用户管理系统需求" ued/active/

# 使用文档检查脚本进行更全面的检查
python scripts/check_docs.py -p ued/active
```

**引用排查清单**：
- [ ] 搜索文档编码的全量引用（如 `CRM-L2-001`）
- [ ] 搜索文档编码的简写引用（如 `L2-001`）
- [ ] 搜索文档标题的引用（如 `用户管理系统需求`）
- [ ] 记录所有引用文档及其引用位置
- [ ] 评估引用影响范围

**引用影响评估**：
```bash
# 发现引用后，评估影响
$ grep -r "CRM-L2-001" ued/active/
ued/active/CRM-L3-001_架构设计.md:参见 [CRM-L2-001 用户管理系统需求](./CRM-L2-001_用户管理系统需求.md)
ued/active/CRM-L4-002_接口设计.md:根据 [CRM-L2-001 用户管理系统需求](./CRM-L2-001_用户管理系统需求.md) 的要求

# 记录引用信息到准备信息表
影响范围: CRM-L3-001, CRM-L4-002
引用位置: 
  - CRM-L3-001 架构设计.md: 第15行
  - CRM-L4-002 接口设计.md: 第8行
```

**引用处理策略**：

| 检查结果 | 处理方式 | 说明 |
|---------|---------|------|
| 无引用 | ✅ 继续移动 | 安全移动到 deprecated |
| 有引用 | ✅ 先处理引用，再移动 | 必须在移动前处理引用 |

**重要提醒**：
- **MANDATORY**：移动前必须完成引用检查
- 不要假设没有引用就跳过检查
- 发现引用后必须记录，在后续步骤中处理

### 第五步：添加废弃标记

**在废弃文档中添加标准化的废弃元信息**：

在文档开头添加完整的废弃信息块：

```markdown
---
⚠️ **本文档已废弃**

**废弃元信息**
- 文档编码：CRM-L2-001
- 文档标题：用户管理系统需求
- 废弃时间：2024-03-23
- 废弃原因：系统重构，需求完全变更
- 替代文档：[CRM-L2-005 新用户管理系统需求](../active/CRM-L2-005_新用户管理系统需求.md)
- 影响范围：CRM-L3-001, CRM-L4-002
- 建议删除日期：2024-09-23（6个月后）

**文档类型保留期限**：项目文档（6个月）
**当前状态**：保留期中，请勿删除

请参考替代文档。本文档仅保留用于历史追溯。
---
```

**废弃元信息字段说明**：

| 字段 | 必填 | 说明 | 示例 |
|------|------|------|------|
| 文档编码 | ✅ | 废弃文档的完整编码 | CRM-L2-001 |
| 文档标题 | ✅ | 文档的完整标题 | 用户管理系统需求 |
| 废弃时间 | ✅ | 废弃操作的日期 | 2024-03-23 |
| 废弃原因 | ✅ | 废弃的详细原因 | 系统重构，需求完全变更 |
| 替代文档 | ⚠️ | 新的替代文档编码（如有） | CRM-L2-005，无则填写"无" |
| 影响范围 | ✅ | 受影响的其他文档列表 | CRM-L3-001, CRM-L4-002 |
| **建议删除日期** | ✅ | **根据文档类型计算的清理日期** | **2024-09-23** |

**建议删除日期计算规则**：

| 文档类型 | 保留期限 | 计算方式 | 示例 |
|---------|---------|---------|------|
| 临时文档 | 3个月 | 废弃时间 + 3个月 | 2024-03-23 → 2024-06-23 |
| 项目文档 | 6个月 | 废弃时间 + 6个月 | 2024-03-23 → 2024-09-23 |
| 架构文档 | 1年 | 废弃时间 + 1年 | 2024-03-23 → 2025-03-23 |
| 核心文档 | 2年 | 废弃时间 + 2年 | 2024-03-23 → 2026-03-23 |

**文档类型判断标准**：

| 文档类型 | 典型文档 | 特征 |
|---------|---------|------|
| 临时文档 | TEMP-*, 试验性设计 | 临时性、试验性、短期需求 |
| 项目文档 | L2-需求, L4-设计 | 具体项目的需求设计文档 |
| 架构文档 | L3-架构, ADR | 系统架构、架构决策记录 |
| 核心文档 | L0-战略, L1-利益相关者 | 战略级、影响范围大的核心文档 |

**计算脚本示例**：

```bash
#!/bin/bash
# 计算建议删除日期

deprecated_date="2024-03-23"
doc_type="project"  # 临时文档/temp, 项目文档/project, 架构文档/architecture, 核心文档/core

case $doc_type in
  temp)
    days=$((3 * 30))
    ;;
  project)
    days=$((6 * 30))
    ;;
  architecture)
    days=$((365))
    ;;
  core)
    days=$((2 * 365))
    ;;
  *)
    days=$((6 * 30))  # 默认6个月
    ;;
esac

suggested_delete_date=$(date -d "$deprecated_date +$days days" +%Y-%m-%d)
echo "建议删除日期: $suggested_delete_date"
```

### 第六步：更新引用文档

**根据第三步的引用检查结果，更新所有引用文档**：

**处理引用关系**：

1. **引用源确认**：基于第三步记录的引用信息
2. **更新引用**：将引用更新为替代文档，或添加废弃标记
3. **路径修正**：更新相对路径引用（文档已移动到 deprecated 目录）
4. **更新索引**：更新全局索引中的文档状态

## 定期清理策略

### 清理周期建议

| 文档类型 | 保留期限 | 清理策略 |
|---------|---------|---------|
| 临时文档 | 3个月 | 直接删除 |
| 项目文档 | 6个月 | 评估后删除或归档 |
| 架构文档 | 1年 | 归档到长期存储 |
| 核心文档 | 2年 | 归档到永久存储 |

### 清理操作流程

**0. 引用检查（MANDATORY - 清理前必须执行）**：

**在执行清理前，必须排查现有设计文档中是否还有对该废弃文档的引用**：

```bash
# 在 active 目录中搜索对废弃文档的引用
grep -r "CRM-L2-001" ued/active/

# 搜索文档编码和文档标题
grep -r "用户管理系统需求" ued/active/

# 使用文档检查脚本进行更全面的检查
python scripts/check_docs.py -p ued/active
```

**引用排查清单**：
- [ ] 搜索文档编码的全量引用（如 `CRM-L2-001`）
- [ ] 搜索文档编码的简写引用（如 `L2-001`）
- [ ] 搜索文档标题的引用（如 `用户管理系统需求`）
- [ ] 检查是否有相对路径引用（如 `../deprecated/2024/03/CRM-L2-001_*.md`）
- [ ] 检查全局索引中的引用
- [ ] 检查其他废弃文档的引用（防止废弃文档互相引用）

**引用处理策略**：

| 检查结果 | 处理方式 | 说明 |
|---------|---------|------|
| 无引用 | ✅ 可以清理 | 安全清理或归档 |
| 有引用，引用方也废弃 | ✅ 可以清理 | 一并处理引用方文档 |
| 有引用，引用方仍活跃 | ❌ 禁止清理 | 必须先更新或移除引用 |

**引用处理示例**：

```bash
# 发现活跃文档中有引用
$ grep -r "CRM-L2-001" ued/active/
ued/active/CRM-L3-001_架构设计.md:参见 [CRM-L2-001 用户管理系统需求](../deprecated/2024/03/CRM-L2-001_用户管理系统需求.md)

# 处理方案1：更新引用（如果有替代文档）
# 将引用更新为替代文档，或添加废弃标记

# 处理方案2：评估引用方是否也应废弃
# 如果引用方的内容完全依赖该废弃文档，考虑同时废弃引用方

# 处理方案3：延长保留期
# 如果引用方短期内无法更新，延长该废弃文档的保留期
```

**1. 评估清理清单**：

```bash
# 查找过期的废弃文档（超过建议删除日期）
# 方法1：使用文档头部元信息检查
find ued/deprecated -name "*.md" -exec grep -l "建议删除日期" {} \; | while read file; do
  suggested_date=$(grep "建议删除日期" "$file" | awk '{print $2}')
  if [ "$(date -d "$suggested_date" +%s)" -lt "$(date +%s)" ]; then
    echo "可清理: $file (建议删除日期: $suggested_date)"
  fi
done

# 方法2：使用文件修改时间（参考）
find ued/deprecated -name "*.md" -mtime +180
```

**2. 确认清理必要性**：
- [ ] 确认文档已超过建议删除日期
- [ ] 确认无活跃文档引用该废弃文档（MANDATORY）
- [ ] 确认无历史追溯需求
- [ ] 确认相关项目已结束或文档已归档

**3. 执行清理或归档**：
```bash
# 删除过期的临时文档
rm ued/deprecated/2023/09/TEMP-001_临时设计.md

# 归档重要的废弃文档
tar -czf archive/2024_q1_deprecated.tar.gz ued/deprecated/2024/01 ued/deprecated/2024/02 ued/deprecated/2024/03
```

**4. 更新清理记录**：
```markdown
## 文档清理记录

| 清理日期 | 文档数量 | 清理方式 | 操作人 |
|---------|---------|---------|--------|
| 2024-04-01 | 5 | 删除 | 李四 |
| 2024-04-01 | 10 | 归档 | 李四 |
```

## 废弃文档恢复

**恢复流程**（如果需要恢复废弃的文档）：

1. **定位文档**：在 `deprecated` 目录中找到文档
2. **评估恢复必要性**：确认文档是否需要恢复到活跃状态
3. **移动回 active 目录**：
   ```bash
   mv ued/deprecated/2024/03/CRM-L2-001_用户管理系统需求_已废弃.md \
      ued/active/CRM-L2-001_用户管理系统需求.md
   ```
4. **移除废弃标记**：删除文档开头的废弃信息块
5. **更新文档状态**：将状态改为"草稿"或"正式"
6. **更新引用**：恢复或更新引用关系
7. **更新索引**：从废弃文档索引中移除，添加到活跃文档索引

```

**批量排查脚本示例**：

```bash
#!/bin/bash
# check-deprecated-refs.sh
# 批量检查所有废弃文档的引用

DEPRECATED_DIR="ued/deprecated"
ACTIVE_DIR="ued/active"

echo "=== 废弃文档引用检查报告 ==="
echo "检查时间: $(date)"
echo ""

# 遍历所有废弃文档
find "$DEPRECATED_DIR" -name "*.md" | while read -r deprecated_file; do
    # 提取文档编码（假设文件名格式：CRM-L2-001_标题.md）
    doc_code=$(basename "$deprecated_file" | sed 's/_.*//' | sed 's/_已废弃$//')
    doc_title=$(basename "$deprecated_file" | sed 's/^[^_]*_//' | sed 's/.md$//' | sed 's/_已废弃$//')

    echo "文档: $doc_code - $doc_title"

    # 搜索引用
    ref_count=$(grep -r "$doc_code" "$ACTIVE_DIR" 2>/dev/null | wc -l)
    title_count=$(grep -r "$doc_title" "$ACTIVE_DIR" 2>/dev/null | wc -l)

    # 检查建议删除日期
    suggested_date=$(grep "建议删除日期" "$deprecated_file" 2>/dev/null | awk '{print $2}')
    if [ -n "$suggested_date" ]; then
      current_timestamp=$(date +%s)
      suggested_timestamp=$(date -d "$suggested_date" +%s 2>/dev/null || echo 0)
      is_expired="保留期中"
      if [ "$suggested_timestamp" -gt 0 ] && [ "$suggested_timestamp" -lt "$current_timestamp" ]; then
        is_expired="⏰ 已过建议删除日期"
      fi
    else
      is_expired="未设置建议删除日期"
    fi

    if [ "$ref_count" -gt 0 ] || [ "$title_count" -gt 0 ]; then
        echo "  ⚠️  发现引用:"
        [ "$ref_count" -gt 0 ] && echo "     - 文档编码引用: $ref_count 处"
        [ "$title_count" -gt 0 ] && echo "     - 标题引用: $title_count 处"
        echo "  建议处理: 先更新引用再清理"
        echo "  $is_expired"
    else
        echo "  ✅ 无引用"
        echo "  $is_expired"
        if [ "$is_expired" = "⏰ 已过建议删除日期" ]; then
          echo "  建议操作: 可以清理"
        else
          echo "  建议操作: 保留至建议删除日期"
        fi
    fi
    echo ""
done

echo "=== 检查完成 ==="
```
