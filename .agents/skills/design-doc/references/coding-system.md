# 编码体系规范

## 需求级别说明

本文档使用 RFC2119 定义的词汇表示需求级别：

- **MUST**：绝对要求，必须遵守
- **SHOULD**：强烈推荐，除非有充分理由否则应遵守  
- **MAY**：可选，可根据具体情况决定

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", 
"SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" 
in this document are to be interpreted as described in RFC 2119.

## 项目编码管理

### 项目编码获取

项目编码从配置文件中获取，优先级如下：

1. **项目配置文件** `ued/.doc-config.json` 中的 `project_code` 字段
2. **环境变量** `DESIGN_DOC_PROJECT_CODE`
3. **默认值**：`DEFAULT`

**配置文件格式**（`ued/.doc-config.json`）：
```json
{
  "author": "张三",
  "project_code": "CRM"
}
```

**项目编码规则**：
- 长度：2-4位大写字母
- 只能包含字母 A-Z
- 建议体现项目或产品线特征
- 企业内统一分配管理

**有效示例**：`CRM`、`ERP`、`CMS`、`COM`、`HRS`

## 文档编码

格式：`{项目码}-L{层级}-{三位序号}` 或 `{项目码}-ADR-{三位序号}` 或 `{项目码}-REF-{三位序号}`，全系统唯一。

示例：`CRM-L0-001`、`ERP-L2-003`、`CMS-ADR-005`、`COM-REF-001`

### 同仓库内简写与完整形式（统一约定）

- **唯一真值源与跨边界引用**（`ued/README.md` 全局索引、跨文档引用、跨项目引用）：细项与文档编码**一律使用完整形式** `{项目码}-{类型码}-{三位序号}` 或 `{项目码}-L{n}-{三位序号}`。
- **同一文档内部**（元信息已注明项目码时）：小节标题中可写 `FR-001` 等简写以节省篇幅，但**同一文档内**须前后一致，且文末**细项编码清单**、**全局索引**中仍须写完整编码。
- **同项目引用**（仅正文叙述）：可省略项目码前缀（如 `FR-001`），若目标读者可能跨项目阅读，**必须**写全项目码。

## 细项编码

格式：`{项目码}-{类型码}-{三位序号}`，全系统唯一，不依赖文档编码。

### 编码生命周期规则

- 序号从 001 开始，在同一项目码+类型码内单调递增
- 编码一经分配，**MUST NOT** 被删除或复用
- 不再使用的编码 **SHOULD** 将其状态标注为"已废弃"，编码本身 **MUST** 保留
- 新增细项始终使用该项目码+类型码下的下一个可用序号
- 细项废弃的标记方式、引用处理与索引同步：见 [deprecation-guide.md](deprecation-guide.md)

### 类型码表

| 类型码 | 含义 | 典型层级 | 说明 |
|--------|------|----------|------|
| GOL | 目标（Goal） | L0, L1 | 项目要达成的业务目标 |
| STK | 利益相关者（Stakeholder） | L1 | 项目的参与方和影响方 |
| SCN | 场景（Scenario） | L1, L2 | 用户使用场景和业务场景 |
| FR | 功能需求（Functional Req.） | L2 | 系统必须提供的功能 |
| NFR | 非功能需求（Non-functional Req.） | L2 | 性能、安全、可用性等质量属性 |
| UC | 用例（Use Case） | L2 | 具体的用户操作流程 |
| PRN | 原则（Principle） | L3 | 必须遵循的设计规则和约束，无选择余地 |
| DEC | 决策（Decision） | L3/ADR | 从多个可选方案中选择的具体决策，有取舍过程 |
| CMP | 组件/子系统（Component） | L3, L4 | 系统组成模块 |
| IF | 接口契约（Interface） | L4, L5 | 模块间的接口定义 |
| FLW | 流程（Flow） | L4, L5 | 业务流程或系统流程，通常以流程图形式 |
| ALG | 算法/逻辑（Algorithm） | L5 | 具体算法、计算逻辑、公式或引用的标准规范 |
| DOM | 领域对象（Domain Object） | L4 | 业务领域的核心实体 |
| TC | 测试场景（Test Case） | L6 | 测试用例和测试场景 |
| AC | 验收标准（Acceptance Criteria） | L2, L6 | 功能验收的具体标准 |
| CON | 约束（Constraint） | 所有层级 | 技术或业务约束 |
| ASM | 假定（Assumption） | 所有层级 | 设计依赖的前提条件 |
| RUL | 规则（Rule） | 所有层级 | 业务规则或计算规则 |
| RSK | 风险（Risk） | 所有层级 | 项目风险和应对策略 |
| MET | 度量指标（Metric） | L2, L6 | 可量化的性能或质量指标 |
| REF | 外部参考（Reference） | L2-L6 / `ued/references/` | 外部资料细项；对应文档在宿主 `ued/references/` 下，勿与本 skill 的 `references/` 规范目录混淆 |

类型码可按项目需要扩展，扩展时需更新此表。

### 编码与文档的关联

每份文档在末尾维护**细项编码清单**，列出该文档定义的所有细项编码。
`ued/README.md` 维护**全局编码索引**，可按编码查到所属文档。

### 编码分配管理

#### 全局编码计数器

在 `ued/README.md` 中维护各类别编码的分配计数器：

```markdown
## 编码计数器

| 项目码 | 类型码 | 下一个编号 | 最大编号 | 备注 |
|--------|--------|------------|----------|------|
| CRM | GOL | 001 | 999 | |
| CRM | STK | 001 | 999 | |
| CRM | FR | 015 | 999 | 已使用 001-014 |
| CRM | NFR | 008 | 999 | 已使用 001-007 |
| ERP | GOL | 001 | 999 | |
| ERP | FR | 005 | 999 | 已使用 001-004 |
| COM | PRN | 003 | 999 | 已使用 001-002 |
| * | REF | 025 | 999 | 全局共享，跨项目 |
```

**说明**：
- `*` 表示全局共享（如 REF 类型码）
- 每个项目码+类型码组合独立计数
- 跨项目引用时必须包含项目码前缀

#### 编码分配流程

1. **获取项目编码**：从配置文件或环境变量读取当前项目码
2. **新增编码时**：从对应项目码+类型码的"下一个编号"获取，使用后递增计数器
3. **批量分配时**：可预分配一段连续编号，避免频繁更新计数器
4. **废弃处理**：编码废弃时保留编号，计数器不变，在备注中说明
5. **定期检查**：每月检查编码使用情况，必要时调整最大编号

#### 编码分配即时更新机制

**核心要求**：新生成编码 ID 时，**MUST** 立即更新所有相关索引。

**更新清单**：
1. **全局编码索引**（ued/README.md）：
   - **MUST** 添加新编码到对应类型码表格
   - **MUST** 更新编码计数器（递增下一个编号）
   
2. **文档细项清单**（当前文档末尾）：
   - **MUST** 添加新编码到细项编码清单
   - **SHOULD** 确保编码、类型、标题信息完整

3. **跨项目引用索引**（如适用）：
   - 如涉及跨项目引用，**SHOULD** 更新引用关系表

**更新时机**：
- ✅ **立即更新**：编码分配完成后立即更新索引
- ❌ **禁止延迟**：不允许"稍后更新"或"批量更新"
- ✅ **双重检查**：更新后验证索引一致性

#### 编码标题命名原则

**核心原则**：编码 ID 对应的标题，**一旦新建也不得修改**。

**命名指导**：
1. **适度宽泛**：标题应具有一定的概括性，避免过于具体
2. **避免细节**：不要包含具体的实现细节或参数
3. **稳定表述**：使用相对稳定的业务概念或技术术语

**命名示例**：

| 类型 | ❌ 不推荐（过于具体） | ✅ 推荐（适度宽泛） |
|------|---------------------|-------------------|
| FR | 用户使用手机号+密码登录系统 | 用户身份认证功能 |
| NFR | 系统响应时间小于 500ms | 系统响应性能要求 |
| RUL | 账号长度 6-20 位，包含字母数字 | 账号格式规范 |
| PRN | 使用 Spring Security 框架 | 安全架构设计原则 |
| CON | 必须支持 MySQL 8.0 数据库 | 数据存储技术约束 |

**标题修改规则**：
- **绝对禁止**：修改已分配编码的标题
- **允许调整**：仅在发现明显错误时，且需走废弃+新增流程
- **最佳实践**：初始命名时考虑未来扩展性

#### 编码修改防护机制

**核心原则**：编码一经分配，**永不修改**其含义和标题。

**防护措施**：

1. **编码含义锁定**：
   - 编码分配后，其含义和标题永久固定
   - 任何 AI 修改请求必须拒绝：`"编码 CRM-RUL-016 已分配给'账号检查'，不能修改为其他含义"`

2. **修改操作限制**：
   - **禁止修改**：已分配编码的含义、标题、类型
   - **允许修改**：在**不改变编码含义与标题**的前提下，可对细项**正文说明**做澄清、措辞修订；`ued/` 内仍禁止写入代码与具体实现。
   - **含义变化**：须走废弃+新编码，不得借「修改描述」变相改编码语义
   - **废弃处理**：不再使用时标注"已废弃"，不删除不重用

3. **AI 操作约束**：
   ```
   当 AI 收到修改已分配编码的请求时：
   1. 检查编码是否已存在于全局索引
   2. 如已存在，拒绝修改并说明原因
   3. 建议分配新的编码给新内容
   4. 如原内容确实错误，走废弃+新增流程
   ```

4. **正确处理流程**：
   ```
   场景：CRM-RUL-016 原为"账号检查"，需要改为"定时器规则"
   
   ❌ 错误做法：
   - 直接修改 CRM-RUL-016 的含义为"定时器规则"
   
   ✅ 正确做法：
   - CRM-RUL-016 标注为"已废弃"，原因：需求变更
   - 分配新编码 CRM-RUL-017 给"定时器规则"
   - 如需保留账号检查语义，另分配新编码（如 CRM-RUL-018）
   - 更新相关引用和索引
   ```

5. **冲突检测机制**：
   - 修改前检查编码是否已被引用
   - 如有引用，提醒修改影响范围
   - 建议先处理引用关系再修改

#### 编码冲突检查

手动检查编码是否已存在：
- 搜索 `ued/` 目录下的所有 `.md` 文件
- 查找目标编码（包含项目码前缀）的出现位置
- 确认编码未被使用后再分配

### 交叉引用规则

#### 标准引用格式

**核心要求**：引用细项编码时，**必须使用链接格式**，支持跳转和导航。

**标准格式**：
```
[{项目码}-{类型码}-{编号}（{标题}）](#{编码锚点})
```

**引用示例**（须带项目码；锚点为全小写）：
- `[CRM-FR-001（用户身份认证功能）](#crm-fr-001)`
- `[ERP-NFR-002（系统响应性能要求）](#erp-nfr-002)`
- `[COM-PRN-003（安全架构设计原则）](#com-prn-003)`
- `[CRM-RUL-004（账号格式规范）](#crm-rul-004)`

**优势**：
- ✅ 可点击跳转到编码定义处
- ✅ 适合长文档的导航和查阅
- ✅ 交互性强，用户体验好
- ✅ Markdown 原生支持，兼容性最佳

#### 锚点命名规则

**编码定义处（目标）**：
```markdown
## CRM-FR-001 用户身份认证功能 {#crm-fr-001}
```

**编码引用处（源）**：
```markdown
根据 [CRM-FR-001（用户身份认证功能）](#crm-fr-001) 的要求...
```

**锚点命名规范**：
- **格式**：`#{项目码小写}-{类型码小写}-{编号}`
- **示例**：`#crm-fr-001`、`#erp-nfr-002`
- **规则**：全小写，短横线分隔，与编码格式对应

#### 引用场景和格式

**1. 文档正文中的引用**：
```
根据 [CRM-FR-001（用户身份认证功能）](#crm-fr-001) 的要求，系统需要...
遵循 [CRM-RUL-004（账号格式规范）](#crm-rul-004) 的原则，...
```

**2. 表格中的引用**：
| 需求 | 关联规则 | 验收标准 |
|------|----------|----------|
| [CRM-FR-001（用户身份认证功能）](#crm-fr-001) | [CRM-RUL-004（账号格式规范）](#crm-rul-004) | [CRM-AC-001（登录成功标准）](#crm-ac-001) |

**3. 列表中的引用**：
- [CRM-FR-001（用户身份认证功能）](#crm-fr-001)
- [ERP-NFR-002（系统响应性能要求）](#erp-nfr-002)
- [COM-PRN-003（安全架构设计原则）](#com-prn-003)

#### 跨文档引用处理

**同文档内引用**：
```markdown
[CRM-FR-001（用户身份认证功能）](#crm-fr-001)
```

**跨文档引用**：
```markdown
[CRM-FR-001（用户身份认证功能）](../L2-requirements/crm-fr-001-user-auth.md#crm-fr-001)
```

**跨项目引用**：
```markdown
[CRM-FR-001（用户身份认证功能）](../../crm/ued/L2-requirements/crm-fr-001-user-auth.md#crm-fr-001)
```

#### 引用格式检查清单

**引用前检查**：
- [ ] 编码格式正确（项目码-类型码-编号）
- [ ] 标题与索引完全一致
- [ ] 锚点命名符合规范（小写+短横线）
- [ ] 链接路径正确（同文档/跨文档/跨项目）

**引用后验证**：
- [ ] 被引用编码确实存在
- [ ] 目标位置有对应锚点
- [ ] 引用方向正确（下层引用上层）
- [ ] 标题拼写准确
- [ ] 链接可正常跳转

#### 引用错误示例

**❌ 错误格式**：
```
CRM-FR-001                    // 缺少链接和标题
[CRM-FR-001](#crm-fr-001)        // 缺少标题
[CRM-FR-001 用户身份认证功能](#crm-fr-001)  // 缺少括号
[crm-fr-001（用户身份认证功能）](#CRM-FR-001)  // 锚点大小写错误
```

**✅ 正确格式**：
```
[CRM-FR-001（用户身份认证功能）](#crm-fr-001)
```

#### 自动化引用检查

实现时可扫描 `[编码（标题）](#锚点)`，校验锚点是否为**完整编码**的全小写形式（如 `CRM-FR-001` → `#crm-fr-001`）。以下为**思路示意**，正则需按实际编码格式（多段、多类型码）调整，不保证复制即用。

```text
对每条链接：解析 [] 内编码 → 转小写 → 与 # 后锚点比较；不一致则报错。
```
