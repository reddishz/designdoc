# 编码体系规范

## 需求级别说明

本 skill 在规范正文与 AI 约束里混用 **中文**（必须、应当、可以、禁止）与 **英文**（MUST、SHOULD、MAY、MUST NOT），仅用于区分**约束松紧**，语义上**借鉴** [RFC 2119](https://www.rfc-editor.org/rfc/rfc2119) 的常见解释；**本 skill 不以 RFC 合规为交付目标**，也不要求宿主 `ued/` 文档逐条对照 RFC 或附加英文声明。

| 常见写法 | 含义（简要） |
|----------|----------------|
| **必须 / MUST** | 绝对要求，违反即视为不符合本规范 |
| **禁止 / MUST NOT** | 绝对不允许 |
| **应当 / SHOULD** | 强烈建议，除非有充分理由可不照做 |
| **不应当 / SHOULD NOT** | 强烈不建议 |
| **可以 / MAY** | 可选，由实现或项目裁量 |

**对业务文档的建议**：`ued/` 内设计文档一般用中文「必须、应当、可以」即可；若需对外或跨团队对齐，再视情况在文首加一句「关键词按 RFC 2119 理解」。

## 编码核心原则

1. **简洁优先**：默认忽略项目前缀。这是本系统的**标准模式**，适用于 95% 以上的项目。
2. **全局唯一**：在同一仓库范围内，编码必须唯一。
3. **不可变更**：编码一旦分配，其含义与标题不得修改（详见防护机制）。

### 编码锁定约束（MANDATORY）
- **MUST NOT**：修改已分配编码的ID、类型、标题或业务含义
- **MUST NOT**：接受任何修改编码含义的请求
- **MUST**：拒绝此类请求并引导用户走废弃+新增流程
- **MUST**：检测并阻止任何违反编码锁定原则的操作

## 标准编码体系（推荐/默认）

### 细项编码

格式：`{类型码}-{三位序号}`，全系统唯一。

示例：`FR-001`、`NFR-003`、`PRN-005`、`RUL-010`

### 类型码表

| 类型码 | 含义 | 典型层级 | 说明 | 典型示例 |
|--------|------|----------|------|----------|
| GOL | 目标（Goal） | L0, L1 | 项目要达成的业务目标 | `GOL-001`: 实现用户月活增长20% |
| STK | 利益相关者（Stakeholder） | L1 | 项目的参与方和影响方 | `STK-001`: 普通个人用户 |
| SCN | 场景（Scenario） | L1, L2 | 用户使用场景和业务场景 | `SCN-001`: 用户在弱网环境下支付 |
| FR | 功能需求（Functional Req.） | L2 | 系统必须提供的功能 | `FR-001`: 支持第三方微信登录 |
| NFR | 非功能需求（Non-functional Req.） | L2 | 性能、安全、可用性等质量属性 | `NFR-001`: 核心接口响应时间 < 200ms |
| UC | 用例（Use Case） | L2 | 具体的用户操作流程 | `UC-001`: 用户找回密码流程 |
| PRN | 原则（Principle） | L3 | 必须遵循的设计规则和约束 | `PRN-001`: 无状态设计原则 |
| DEC | 决策（Decision） | L3/ADR | 从多个可选方案中选择的具体决策 | `DEC-001`: 使用 PostgreSQL 存储业务数据 |
| CMP | 组件/子系统（Component） | L3, L4 | 系统组成模块 | `CMP-001`: 订单处理子系统 |
| IF | 接口契约（Interface） | L4, L5 | 模块间的接口定义 | `IF-001`: 获取用户信息 API |
| FLW | 流程（Flow） | L4, L5 | 业务流程或系统流程 | `FLW-001`: 订单取消状态流转图 |
| ALG | 算法/逻辑（Algorithm） | L5 | 具体算法、计算逻辑或公式 | `ALG-001`: 阶梯价格计算公式 |
| DOM | 领域对象（Domain Object） | L4 | 业务领域的核心实体 | `DOM-001`: 订单（Order）实体模型 |
| ASM | 假设（Assumption） | L1, L2 | 项目设计时依赖的前提假设 | `ASM-001`: 假设用户手机均支持 HTTPS |
| RSK | 风险（Risk） | L1, L2 | 可能影响项目成功的风险因素 | `RSK-001`: 第三方支付接口可能无法及时上线 |
| MET | 指标（Metric/KPI） | L0, L1 | 用于衡量系统表现的关键指标 | `MET-001`: 系统年可用性达到 99.9% |
| TC | 测试场景（Test Case） | L6 | 测试用例和测试场景 | `TC-001`: 并发 1000 下的下单压力测试 |
| AC | 验收标准（Acceptance Criteria） | L2, L6 | 功能验收的具体标准 | `AC-001`: 支付成功后 5s 内发送短信通知 |
| CON | 约束（Constraint） | 所有层级 | 技术或业务约束 | `CON-001`: 必须兼容 IE11 浏览器 |
| RUL | 规则（Rule） | 所有层级 | 业务规则或计算规则 | `RUL-001`: 只有会员才能参与秒杀活动 |
| ACT | 动作定义（Action） | L4, L5 | 系统执行的具体动作和行为，包括消息、事件、命令、菜单等交互类型 | ACT-001: 用户注册动作定义 |
| PLAN | 计划项（Plan Item） | L2-L5 | 具体可执行的计划项，可迁移为其他类型 | PLAN-001: 用户注册流程优化计划 |
| REF | 外部参考（Reference） | L2-L6 | 外部资料细项 | REF-001: 《行业支付安全协议 v2.0》 |

### 编码生命周期规则

- 序号从 001 开始，在同一类型码内单调递增。
- 编码一经分配，**MUST NOT** 被删除或复用。
- 不再使用的编码 **SHOULD** 标注为"已废弃"，编码本身 **MUST** 保留。

### 编码含义锁定（MANDATORY）
- **MUST NOT**：修改已分配编码的ID、类型、标题或业务含义
- **MUST**：废弃不再使用的编码并分配新编码
- **MUST**：保留所有已分配编码的完整历史记录

## 文档编码

格式：`L{层级}-{三位序号}` 或 `ADR-{三位序号}` 或 `REF-{三位序号}`。

示例：`L0-001`（战略文档）、`L2-003`（需求文档）、`ADR-005`（架构决策记录）。

## 扩展用法：多项目管理（仅高级需求启用）

在 99% 的场景下，**MUST NOT** 启用此功能。仅当满足以下**所有**条件，且用户明确要求时，才考虑启用：

1. **多项目共存**：在同一个 `ued/` 目录下管理完全独立的、不同产品线的项目。
2. **企业级合规**：公司明确规定所有文档、所有设计细项必须带有全局统一的 ID 标识前缀。
3. **已显式配置**：`ued/.doc-config.json` 中配置了 `project_code` 字段。

### 项目编码规则
- **格式要求**：2-4 位大写字母（A-Z）。
- **获取优先级**：
  1. 配置文件 `ued/.doc-config.json` 中的 `project_code`
  2. 环境变量 `DESIGN_DOC_PROJECT_CODE`
  3. 默认值：`DEFAULT`（当启用扩展模式但未配置时）
- **分配原则**：
  - **全局唯一**：每个项目编码在全局范围内唯一。
  - **语义化**：应体现项目或产品线特征（如 `CRM`, `ERP`）。
  - **不可变更**：项目一旦启动并分配编码，禁止中途修改。

### 扩展编码格式
- 格式：`{项目码}-{类型码}-{序号}`
- 示例：`CRM-FR-001`、`ERP-L2-003`

### 获取方式
1. 默认情况下，AI 应假设项目码为空，直接生成标准格式编码（如 `FR-001`）。
2. 若用户在 `ued/.doc-config.json` 中定义了 `project_code`，AI 应当询问用户：“检测到项目编码配置，后续生成 ID 是否启用该前缀？”
3. 只有在用户明确确认后，才开始在所有新生成的 ID 中加入该前缀。

### 编码分配即时更新机制

**核心要求**：新生成编码 ID 时，**MUST** 立即更新所有相关索引。

1. **全局编码索引**（ued/README.md）：在对应类型码表格中添加新编码条目。
2. **文档细项清单**（当前文档末尾）：在表格中列出该文档定义的所有编码。

## 编码防护与修改机制

### 核心原则：编码含义锁定（MANDATORY）
编码一经分配，**永不修改**其含义和标题。

### 修改操作限制（MUST COMPLY）
- **禁止修改（MUST NOT）**：已分配编码的 ID、类型、标题。
- **允许修改（MAY）**：在不改变含义的前提下，完善正文描述或澄清细节。
- **含义变更处理（MANDATORY）**：若业务需求发生根本性变化，**MUST** 废弃原编码并分配新编码。

### 违规检测与纠正（REQUIRED）
AI **MUST** 能够：
- 识别修改编码含义的违规请求
- 自动拒绝此类请求
- 提供正确的废弃+新增流程指导
- 使用标准化的拒绝响应模板

## ACT类型详细说明

### 概念范围
ACT类型用于定义系统执行的具体动作和行为，涵盖以下交互类型：

**消息类型（Message）**：
- Actor间的消息传递
- 系统组件间的通信消息
- 异步消息定义

**事件类型（Event）**：
- 领域事件（Domain Event）
- 系统事件（System Event）
- 集成事件（Integration Event）

**命令类型（Command）**：
- 用户命令（User Command）
- 系统命令（System Command）
- 管理命令（Admin Command）

**菜单类型（Menu）**：
- 用户界面菜单动作
- 功能菜单定义
- 操作菜单行为

**其他交互类型**：
- 查询动作（Query Action）
- 通知动作（Notification Action）
- 调用动作（Invocation Action）

### 命名规范
```
ACT-{序号}: {动作类型} - {具体动作描述}
```

**示例**：
- `ACT-001: UserRegistrationCommand - 用户注册命令`
- `ACT-002: OrderCreatedEvent - 订单创建事件`
- `ACT-003: SendEmailMessage - 发送邮件消息`
- `ACT-004: MainMenuAction - 主菜单动作`
- `ACT-005: UserQueryAction - 用户查询动作`

### 与其他类型的关系
- **ACT vs IF**：ACT定义行为动作，IF定义数据接口
- **ACT vs FLW**：ACT是单个动作，FLW是完整流程
- **ACT vs ALG**：ACT是业务动作，ALG是计算逻辑

## PLAN类型详细说明

### 概念范围
PLAN类型用于定义具体可执行的计划项，是连接路线图和具体需求/设计的桥梁：

**层级适配**：
- **L2-PLAN**：产品功能计划 → 迁移为FR/NFR
- **L3-PLAN**：架构改进计划 → 迁移为PRN/DEC/CMP
- **L4-PLAN**：接口设计计划 → 迁移为IF/ACT
- **L5-PLAN**：算法优化计划 → 迁移为ALG/ACT

### 计划项生命周期
1. **规划中**：在路线图中识别，创建PLAN项
2. **细化中**：详细分析，准备迁移
3. **执行中**：开始转化为具体需求/设计
4. **已迁移**：成功转化为其他类型码
5. **已完成**：对应需求/设计实现完成

### 命名规范
```
PLAN-{序号}: {计划类型} - {具体计划描述}
```

**示例**：
- `PLAN-001: UserRegistrationOptimization - 用户注册流程优化计划`
- `PLAN-002: PerformanceImprovement - 性能改进计划`
- `PLAN-003: CacheArchitectureUpgrade - 缓存架构升级计划`

### PLAN项结构
```markdown
### PLAN-{序号} {计划标题}

**计划状态**：规划中 → 细化中 → 执行中 → 已迁移 → 已完成
**优先级**：高/中/低
**预估工作量**：{人天}
**负责团队**：{团队名称}

**路线图关联**：
- 关联路线图：[L0-XXX 产品路线图](#l0-xxx)
- 所属主题：{路线图主题}
- 支撑目标：[GOL-XXX 战略目标](#gol-xxx)

**详细描述**：
1. 现状分析
2. 改进目标
3. 关键步骤
4. 成功标准

**迁移记录**：
- {日期}：迁移为 [FR-XXX 功能需求](#fr-xxx)
- {日期}：迁移为 [NFR-XXX 非功能需求](#nfr-xxx)
```

### 与其他类型的关系
- **PLAN vs FR/NFR**：PLAN是预备阶段，FR/NFR是最终需求
- **PLAN vs PRN/DEC**：PLAN是改进计划，PRN/DEC是架构决策
- **PLAN vs ACT**：PLAN是规划层面，ACT是具体动作
