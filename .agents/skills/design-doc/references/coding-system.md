# 编码体系规范

## 需求级别说明

本 skill 在规范正文与 AI 约束里混用 **中文**（必须、应当、可以、禁止）与 **英文**（MUST、SHOULD、MAY、MUST NOT），仅用于区分**约束松紧**，语义上**借鉴** [RFC 2119](https://www.rfc-editor.org/rfc/rfc2119) 的常见解释；**本 skill 不以 RFC 合规为交付目标**，也不要求宿主 `ued/` 文档逐条对照 RFC 或附加英文声明。

| 常见写法 | 含义（简要） |
|----------|----------------|
| **必须 / MUST** | 绝对要求，违反即视为不符合本规范 |
| **禁止 / MUST NOT** | 绝对不允许 |
| **应当 / SHOULD** | 强烈建议，除非有充分理由可不照做 |
| **不应当 / SHOULD NOT** | 强烈不建议 |
| **可以 / MAY** | 可选，由实现或项目裁量 |

**对业务文档的建议**：`ued/` 内设计文档一般用中文「必须、应当、可以」即可；若需对外或跨团队对齐，再视情况在文首加一句「关键词按 RFC 2119 理解」。

## 编码核心原则

1. **简洁优先**：默认忽略项目前缀。这是本系统的**标准模式**，适用于单应用或单作用域项目。
2. **全局唯一**：在同一仓库范围内，编码必须唯一。
3. **不可变更**：编码一旦分配，其含义与标题不得修改（详见防护机制）。

### 编码锁定约束（MANDATORY）
- **MUST NOT**：修改已分配编码的ID、类型、标题或业务含义
- **MUST NOT**：接受任何修改编码含义的请求
- **MUST**：拒绝此类请求并引导用户走废弃+新增流程
- **MUST**：检测并阻止任何违反编码锁定原则的操作

## 标准编码体系（推荐/默认）

### 细项编码

格式：`{类型码}-{三位序号}`，全系统唯一。

示例：`FR-001`、`NFR-003`、`PRN-005`、`RUL-010`

### 类型码表

| 类型码 | 含义 | 典型层级 | 说明 | 典型示例 |
|--------|------|----------|------|----------|
| GOL | 目标（Goal） | L0, L1 | 项目要达成的业务目标 | `GOL-001`: 实现用户月活增长20% |
| STK | 利益相关者（Stakeholder） | L1 | 项目的参与方和影响方 | `STK-001`: 普通个人用户 |
| SCN | 场景（Scenario） | L1, L2 | 用户使用场景和业务场景 | `SCN-001`: 用户在弱网环境下支付 |
| FR | 功能需求（Functional Req.） | L2 | 系统必须提供的功能 | `FR-001`: 支持第三方微信登录 |
| NFR | 非功能需求（Non-functional Req.） | L2 | 性能、安全、可用性等质量属性 | `NFR-001`: 核心接口响应时间 < 200ms |
| UC | 用例（Use Case） | L2 | 具体的用户操作流程 | `UC-001`: 用户找回密码流程 |
| PRN | 原则（Principle） | L3 | 必须遵循的设计规则和约束 | `PRN-001`: 无状态设计原则 |
| DEC | 决策（Decision） | L3/ADR | 从多个可选方案中选择的具体决策 | `DEC-001`: 使用 PostgreSQL 存储业务数据 |
| CMP | 组件/子系统（Component） | L3, L4 | 系统组成模块 | `CMP-001`: 订单处理子系统 |
| IF | 接口契约（Interface） | L4, L5 | 模块间的接口定义 | `IF-001`: 获取用户信息 API |
| FLW | 流程（Flow） | L4, L5 | 业务流程或系统流程 | `FLW-001`: 订单取消状态流转图 |
| ALG | 算法/逻辑（Algorithm） | L5 | 具体算法、计算逻辑或公式 | `ALG-001`: 阶梯价格计算公式 |
| DOM | 领域对象（Domain Object） | L4 | 业务领域的核心实体 | `DOM-001`: 订单（Order）实体模型 |
| ASM | 假设（Assumption） | L1, L2 | 项目设计时依赖的前提假设 | `ASM-001`: 假设用户手机均支持 HTTPS |
| RSK | 风险（Risk） | L1, L2 | 可能影响项目成功的风险因素 | `RSK-001`: 第三方支付接口可能无法及时上线 |
| MET | 指标（Metric/KPI） | L0, L1 | 用于衡量系统表现的关键指标 | `MET-001`: 系统年可用性达到 99.9% |
| TC | 测试场景（Test Case） | L6 | 测试用例和测试场景 | `TC-001`: 并发 1000 下的下单压力测试 |
| AC | 验收标准（Acceptance Criteria） | L2, L6 | 功能验收的具体标准 | `AC-001`: 支付成功后 5s 内发送短信通知 |
| CON | 约束（Constraint） | 所有层级 | 技术或业务约束 | `CON-001`: 必须兼容 IE11 浏览器 |
| RUL | 规则（Rule） | 所有层级 | 业务规则或计算规则 | `RUL-001`: 只有会员才能参与秒杀活动 |
| ACT | 动作定义（Action） | L4, L5 | 系统执行的具体动作和行为，包括消息、事件、命令、菜单等交互类型 | ACT-001: 用户注册动作定义 |
| PLN | 计划项（Plan Item） | L2-L5 | 具体可执行的计划项，可迁移为其他类型 | PLN-001: 用户注册流程优化计划 |
| REF | 外部参考（Reference） | L2-L6 | 外部资料细项 | REF-001: 《行业支付安全协议 v2.0》 |

### 细项定义最小要素（清晰性基线）

为避免“有编码、无定义”或“标题可读但不可执行”的情况，新增或修订细项时，**SHOULD** 至少满足以下要素：

1. **一句话定义（What）**：这个细项到底约束/声明了什么。
2. **边界说明（Scope）**：明确适用范围与不适用范围（至少说明一条非目标）。
3. **触发与条件（When）**：在什么场景、前置条件下生效。
4. **可验证性（Verify）**：给出可检查的验收口径、指标或判定方式（可引用 AC/TC/MET）。
5. **来源与追溯（Trace）**：至少关联一个上层依据（如 GOL/SCN/FR/NFR/PRN）。

#### 适用性边界（按类型分组）

不是所有类型码都要“等强度”满足 5 要素。评审时应按类型语义做差异化判断：

| 类型分组 | 典型类型码 | 建议判定 | 说明 |
|----------|------------|----------|------|
| 目标/对象类 | GOL, STK, SCN, DOM, CMP | What + Scope + Trace 为主；When/Verify 可弱化 | 这类更偏“定义对象/目标”，不一定存在明确触发条件 |
| 需求/规则类 | FR, NFR, RUL, CON, AC, UC | 5 要素完整性要求最高 | 这类直接驱动设计与验收，必须可执行、可验证 |
| 设计/行为类 | IF, FLW, ALG, ACT, DEC, PRN | What + Scope + When + Trace 为主；Verify 通过关联 AC/TC/MET 满足 | 允许把验证口径放在下游验证文档，不要求都写在本细项正文 |
| 管理/演进类 | PLN, RSK, ASM, MET | What + Scope + Trace 必需；When/Verify 视场景可选 | 计划、风险、假设更强调管理语义，触发和验证粒度可按项目裁量 |
| 外部参考类 | REF | What（来源说明）+ Scope（适用边界）必需；Trace 应指向被支撑细项；When/Verify 通常不作硬性要求 | REF 是“证据/依据”而非内部需求细项，不应机械套用触发条件 |

> 说明：以上是“定义质量”的基线与适用边界，不改变编码分配、索引同步、废弃处理等既有强制流程。

### 编码生命周期规则

- 序号从 001 开始，在同一类型码内单调递增。
- 编码一经分配，**MUST NOT** 被删除或复用。
- 不再使用的编码 **SHOULD** 标注为"已废弃"，编码本身 **MUST** 保留。

### 编码含义锁定（MANDATORY）
- **MUST NOT**：修改已分配编码的ID、类型、标题或业务含义
- **MUST**：废弃不再使用的编码并分配新编码
- **MUST**：保留所有已分配编码的完整历史记录

## 扩展用法：多应用管理（仅多应用模式启用）

在 99% 的场景下，**MUST NOT** 启用此功能。仅当满足以下**所有**条件，且请求发起方明确要求时，才考虑启用：

1. **多应用共存**：在同一个 `ued/` 目录下管理完全独立的应用。
2. **企业级合规**：公司明确规定所有文档、所有设计细项必须带有全局统一的 ID 标识前缀。
3. **已显式登记**：顶层 `ued/README.md` 的应用注册表与应用级 `README.md` 中均声明了 `project_code`。

### 强制提示启用场景
- 当 AI 识别到 `ued/` 下已经存在多个应用、多个独立业务域或明显分属不同产品线的设计文档时，**MUST** 提示切换到多应用目录结构，并为每个应用配置独立 `project_code`。
- 当 AI 识别到仓库内已存在某一应用的设计文档，而当前任务是在其基础上新增另一应用的设计文档时，**MUST** 在分配新编码前提示创建应用子目录、补齐应用级 `README.md`，并启用项目编码前缀。
- 此类提示的目的，是保证多应用并行维护时，编码、索引、跨文档引用和废弃追溯仍然保持全局唯一且可辨识。

### 项目编码规则
- **格式要求**：2-4 位大写字母（A-Z）。
- **获取优先级**：
  1. 当前作用域 `README.md` 中的 `project_code`
  2. 顶层 `ued/README.md` 应用注册表中的 `project_code`（多应用模式下交叉校验）
  3. 默认值：空（即不启用前缀，回退简洁编码格式）
- **分配原则**：
  - **全局唯一**：每个项目编码在全局范围内唯一。
  - **语义化**：应体现项目或产品线特征（如 `CRM`, `ERP`）。
  - **不可变更**：项目一旦启动并分配编码，禁止中途修改。

### 扩展编码格式
- 格式：`{项目码}-{类型码}-{序号}`
- 示例：`CRM-FR-001`、`ERP-L2-003`

### 获取方式
1. 默认情况下，AI 应假设项目码为空，直接生成标准格式编码（如 `FR-001`）。
2. 若当前作用域 `README.md` 中定义了 `project_code`，AI 应当发起确认：“检测到项目编码配置，后续生成 ID 是否启用该前缀？”
3. 只有在请求发起方明确确认后，才开始在所有新生成的 ID 中加入该前缀。
4. 若 AI 检测到“多应用并存”或“在已有应用基础上新增另一应用”这两类场景，而当前作用域 `README.md` 尚未配置 `project_code`，则 **MUST** 先提示补充配置，再继续后续编码分配流程。

## 编码分配管理

### 作用域级全局编码索引

在当前作用域的 `README.md` 中维护各类别编码的全局索引：
- 单应用模式：`ued/README.md`
- 多应用模式：`ued/{app-name}/README.md`

顶层 `ued/README.md` 在多应用模式下 **MUST NOT** 承担每个应用细项级编码索引的真值源，只维护应用注册表与跨应用入口。

```markdown
## 全局编码索引

### 按类型码索引

#### FR - 功能需求
| 编码 | 标题 | 所属文档 | 状态 |
|------|------|----------|------|
| FR-001 | 用户身份认证功能 | L2-001 | 草稿 |
| FR-002 | 商品搜索功能 | L2-001 | 已确认 |
```

### 编码分配即时更新机制

**核心要求**：新生成编码 ID 时，**MUST** 立即更新所有相关索引。

1. **作用域级全局编码索引**（当前作用域 `README.md`）：在对应类型码表格中添加新编码条目。
2. **文档细项清单**（当前文档末尾）：在表格中列出该文档定义的所有编码。

### AI 编码分配强制流程

新增细项时 **MUST** 严格遵循以下流程：

```
⚠️ 警告：编码的全局唯一性是体系核心，AI 不得随意在文档内自增编号！

1. 确定编码模式（简洁优先原则）
   默认一律采用简洁模式（如 `FR-015`）。
   只有当当前作用域 `README.md` 中声明了 `project_code` 且请求发起方在 Checkpoint 中明确要求使用时，才启用高级模式（如 `CRM-FR-015`）。

2. 读取全局编码索引
   打开当前作用域 `README.md`，找到"全局编码索引"章节。
   在对应类型码的表格中查找已使用的最大编号。

3. 搜索现有编码（强制步骤，不可跳过）
   在 ued/ 目录下搜索所有 .md 文件。
   确认目标编码（含前缀，如有）是否已被使用。

4. 获取最新编码
   从全局编码索引确定下一个可用编号（如 015），构造编码（如 FR-015）。

5. 外部确认 (Checkpoint)
   如果是批量新增或涉及核心业务规则变更，
   MUST 暂停并请求确认编码标题与定义是否准确。

6. 即时更新索引（编码分配后立即执行，**MUST NOT** 延迟）
   a) **MUST** 在当前文档末尾的"细项编码清单"中添加新编码
   b) **MUST** 在当前作用域 `README.md` 的全局索引中添加新编码条目

7. 验证索引一致性
   **SHOULD** 确认新编码在文档清单和全局索引中都正确建立。
```

## 编码防护与修改机制

### 核心原则：编码含义锁定（MANDATORY）
编码一经分配，**永不修改**其含义和标题。

### 修改操作限制（MUST COMPLY）
- **禁止修改（MUST NOT）**：已分配编码的 ID、类型、标题。
- **允许修改（MAY）**：在不改变含义的前提下，完善正文描述或澄清细节。
- **含义变更处理（MANDATORY）**：若业务需求发生根本性变化，**MUST** 废弃原编码并分配新编码。

### 违规检测与纠正（REQUIRED）
AI **MUST** 能够：
- 识别修改编码含义的违规请求
- 自动拒绝此类请求
- 提供正确的废弃+新增流程指导
- 使用标准化的拒绝响应模板

## IF类型详细说明

### 概念范围
IF（Interface）用于定义**系统边界上的契约**，不仅包含 HTTP API，还包括其他可被调用、可被集成、可被校验的交互边界。

### 常见用途（按实践频次）
1. **HTTP/RPC API 契约**（最常见）：路径、方法、入参/出参、错误语义、幂等性等。
2. **事件/消息契约**：Topic、事件字段、版本兼容、投递与重试语义。
3. **回调/Webhook 契约**：回调触发条件、签名校验、重试策略。
4. **批处理/文件交换契约**：文件格式、字段约束、时序、交付窗口。
5. **内部模块服务边界**：BFF 与领域服务、领域服务之间的调用契约。

### IF 与 API 的关系建议
- **结论**：**不建议**单独新增 `API` 类型码。
- **原因**：
  1. API 本质是 IF 的子集，新增 `API` 易造成语义重叠与分配分叉（同一接口到底用 IF 还是 API）。
  2. 现有 IF 已覆盖“接口契约”主语义，便于统一检索、追溯与治理。
  3. 可通过“接口风格标签”表达差异，不必增加类型码复杂度。

### 推荐做法（不新增类型码）
- 统一使用 `IF-xxx` 编码。
- 在 IF 正文增加标签：`接口风格：HTTP API / RPC / Event / Webhook / FileExchange / InternalService`。
- 对“对外 HTTP API”可在标题中显式标注，如：`IF-012 用户查询 API（HTTP）`。

### IF 与 ACT 的边界判定（防重叠）

**判定核心**：
- 若关注“交互边界的约束与兼容性”（输入输出、版本、错误码、幂等、时序/重试约定）→ **IF**
- 若关注“系统执行了什么业务动作”（命令触发、状态推进、业务语义）→ **ACT**

**可并存场景（推荐双编码）**：
- 同一业务交互同时需要“动作语义 + 接口契约”时，允许 `ACT-xxx` 与 `IF-xxx` 并存
- 约束要求：ACT 必须引用对应 IF；IF 也应关联触发该契约的 ACT（至少单向可追溯）

**快速判定问题**（二选一优先）：
1. 这条定义是否主要用于“调用方/集成方对接”？
   - 是 → IF
2. 这条定义是否主要用于“业务状态/流程推进”？
   - 是 → ACT

若两者都明显成立，应拆分为 IF + ACT，避免单条细项承载双重语义。

## ACT类型详细说明

### 概念范围
ACT类型用于定义系统执行的具体动作和行为，涵盖以下交互类型：

**消息类型（Message）**：
- Actor间的消息传递
- 系统组件间的通信消息
- 异步消息定义

**事件类型（Event）**：
- 领域事件（Domain Event）
- 系统事件（System Event）
- 集成事件（Integration Event）

**命令类型（Command）**：
- 用户命令（User Command）
- 系统命令（System Command）
- 管理命令（Admin Command）

**菜单类型（Menu）**：
- 用户界面菜单动作
- 功能菜单定义
- 操作菜单行为

**其他交互类型**：
- 查询动作（Query Action）
- 通知动作（Notification Action）
- 调用动作（Invocation Action）

### 命名规范
```
ACT-{序号}: {动作类型} - {具体动作描述}
```

**示例**：
- `ACT-001: UserRegistrationCommand - 用户注册命令`
- `ACT-002: OrderCreatedEvent - 订单创建事件`
- `ACT-003: SendEmailMessage - 发送邮件消息`
- `ACT-004: MainMenuAction - 主菜单动作`
- `ACT-005: UserQueryAction - 用户查询动作`

### 与其他类型的关系
- **ACT vs IF**：ACT定义业务动作语义，IF定义交互契约语义（可双编码但不得语义混写）
- **ACT vs FLW**：ACT是单个动作，FLW是完整流程
- **ACT vs ALG**：ACT是业务动作，ALG是计算逻辑

## PLN类型详细说明

### 概念范围
PLN类型用于定义具体可执行的计划项，是连接路线图和具体需求/设计的桥梁：

**层级适配**：
- **L2-PLN**：产品功能计划 → 迁移为FR/NFR
- **L3-PLN**：架构改进计划 → 迁移为PRN/DEC/CMP
- **L4-PLN**：接口设计计划 → 迁移为IF/ACT
- **L5-PLN**：算法优化计划 → 迁移为ALG/ACT

### 适用与不适用边界

**适用场景（建议使用 PLN）**：
- 跨阶段推进、需要逐步拆解的中期工作包（通常 2 周以上）。
- 暂未稳定到单一 FR/NFR/IF/ALG，但已明确方向与价值目标。
- 需要与路线图主题、里程碑、跨团队协作进行显式关联。

**不适用场景（不建议使用 PLN）**：
- 可直接定义为稳定需求/设计细项的内容（直接用 FR/NFR/IF/ALG 等）。
- 仅任务执行层面的工程事项（应放在研发任务系统，而非设计编码体系）。
- 外部资料登记（应使用 REF）。

### 与其他类型的判别规则
- **PLN vs FR/NFR**：PLN 是“计划承诺”，FR/NFR 是“需求定义”。
- **PLN vs DEC/PRN**：PLN 描述推进路径，DEC/PRN 描述已确认决策/原则。
- **PLN vs REF**：PLN 是内部计划对象，REF 是外部依据对象。

### 计划项生命周期
1. **规划中**：在路线图中识别，创建PLN项
2. **细化中**：详细分析，准备迁移
3. **执行中**：开始转化为具体需求/设计
4. **已迁移**：成功转化为其他类型码
5. **已完成**：对应需求/设计实现完成

### 命名规范
```
PLN-{序号}: {计划类型} - {具体计划描述}
```

**示例**：
- `PLN-001: UserRegistrationOptimization - 用户注册流程优化计划`
- `PLN-002: PerformanceImprovement - 性能改进计划`
- `PLN-003: CacheArchitectureUpgrade - 缓存架构升级计划`

### PLN项结构
```markdown
### PLN-{序号} {计划标题}

**计划状态**：规划中 → 细化中 → 执行中 → 已迁移 → 已完成
**优先级**：高/中/低
**预估工作量**：{人天}
**负责团队**：{团队名称}

**路线图关联**：
- 关联路线图：[L0-XXX 产品路线图](#l0-xxx)
- 所属主题：{路线图主题}
- 支撑目标：[GOL-XXX 战略目标](#gol-xxx)

**详细描述**：
1. 现状分析
2. 改进目标
3. 关键步骤
4. 成功标准

**迁移记录**：
- {日期}：迁移为 [FR-XXX 功能需求](#fr-xxx)
- {日期}：迁移为 [NFR-XXX 非功能需求](#nfr-xxx)
```

### 与其他类型的关系
- **PLN vs FR/NFR**：PLN是预备阶段，FR/NFR是最终需求
- **PLN vs PRN/DEC**：PLN是改进计划，PRN/DEC是架构决策
- **PLN vs ACT**：PLN是规划层面，ACT是具体动作
