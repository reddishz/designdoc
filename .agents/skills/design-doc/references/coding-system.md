# 编码体系详细规范

## 项目编码管理

### 项目编码获取

项目编码从配置文件中获取，优先级如下：

1. **项目配置文件** `ued/.doc-config.json` 中的 `project_code` 字段
2. **环境变量** `DESIGN_DOC_PROJECT_CODE`
3. **默认值**：`DEFAULT`

**配置文件格式**（`ued/.doc-config.json`）：
```json
{
  "author": "张三",
  "project_code": "CRM"
}
```

**项目编码规则**：
- 长度：2-4位大写字母
- 只能包含字母 A-Z
- 建议体现项目或产品线特征
- 企业内统一分配管理

**有效示例**：`CRM`、`ERP`、`CMS`、`COM`、`HRS`

## 文档编码

格式：`{项目码}-L{层级}-{三位序号}` 或 `{项目码}-ADR-{三位序号}` 或 `{项目码}-REF-{三位序号}`，全系统唯一。

示例：`CRM-L0-001`、`ERP-L2-003`、`CMS-ADR-005`、`COM-REF-001`

## 细项编码

格式：`{项目码}-{类型码}-{三位序号}`，全系统唯一，不依赖文档编码。

### 编码生命周期规则

- 序号从 001 开始，在同一项目码+类型码内单调递增
- 编码一经分配，**永不删除、永不复用**
- 不再使用的编码将其状态标注为"已废弃"，编码本身保留
- 新增细项始终使用该项目码+类型码下的下一个可用序号

### 类型码表

| 类型码 | 含义 | 典型层级 | 说明 |
|--------|------|----------|------|
| GOL | 目标（Goal） | L0, L1 | 项目要达成的业务目标 |
| STK | 利益相关者（Stakeholder） | L1 | 项目的参与方和影响方 |
| SCN | 场景（Scenario） | L1, L2 | 用户使用场景和业务场景 |
| FR | 功能需求（Functional Req.） | L2 | 系统必须提供的功能 |
| NFR | 非功能需求（Non-functional Req.） | L2 | 性能、安全、可用性等质量属性 |
| UC | 用例（Use Case） | L2 | 具体的用户操作流程 |
| PRN | 原则（Principle） | L3 | 必须遵循的设计规则和约束，无选择余地 |
| DEC | 决策（Decision） | L3/ADR | 从多个可选方案中选择的具体决策，有取舍过程 |
| CMP | 组件/子系统（Component） | L3, L4 | 系统组成模块 |
| IF | 接口契约（Interface） | L4, L5 | 模块间的接口定义 |
| FLW | 流程（Flow） | L4, L5 | 业务流程或系统流程，通常以流程图形式 |
| ALG | 算法/逻辑（Algorithm） | L5 | 具体算法、计算逻辑、公式或引用的标准规范 |
| DOM | 领域对象（Domain Object） | L4 | 业务领域的核心实体 |
| TC | 测试场景（Test Case） | L6 | 测试用例和测试场景 |
| AC | 验收标准（Acceptance Criteria） | L2, L6 | 功能验收的具体标准 |
| CON | 约束（Constraint） | 所有层级 | 技术或业务约束 |
| ASM | 假定（Assumption） | 所有层级 | 设计依赖的前提条件 |
| RUL | 规则（Rule） | 所有层级 | 业务规则或计算规则 |
| RSK | 风险（Risk） | 所有层级 | 项目风险和应对策略 |
| MET | 度量指标（Metric） | L2, L6 | 可量化的性能或质量指标 |
| REF | 外部参考（Reference） | references/ | 外部文档、标准、链接 |

类型码可按项目需要扩展，扩展时需更新此表。

### 编码与文档的关联

每份文档在末尾维护**细项编码清单**，列出该文档定义的所有细项编码。
`ued/README.md` 维护**全局编码索引**，可按编码查到所属文档。

### 编码分配管理

#### 全局编码计数器

在 `ued/README.md` 中维护各类别编码的分配计数器：

```markdown
## 编码计数器

| 项目码 | 类型码 | 下一个编号 | 最大编号 | 备注 |
|--------|--------|------------|----------|------|
| CRM | GOL | 001 | 999 | |
| CRM | STK | 001 | 999 | |
| CRM | FR | 015 | 999 | 已使用 001-014 |
| CRM | NFR | 008 | 999 | 已使用 001-007 |
| ERP | GOL | 001 | 999 | |
| ERP | FR | 005 | 999 | 已使用 001-004 |
| COM | PRN | 003 | 999 | 已使用 001-002 |
| * | REF | 025 | 999 | 全局共享，跨项目 |
```

**说明**：
- `*` 表示全局共享（如 REF 类型码）
- 每个项目码+类型码组合独立计数
- 跨项目引用时必须包含项目码前缀

#### 编码分配流程

1. **获取项目编码**：从配置文件或环境变量读取当前项目码
2. **新增编码时**：从对应项目码+类型码的"下一个编号"获取，使用后递增计数器
3. **批量分配时**：可预分配一段连续编号，避免频繁更新计数器
4. **废弃处理**：编码废弃时保留编号，计数器不变，在备注中说明
5. **定期检查**：每月检查编码使用情况，必要时调整最大编号

#### 编码分配即时更新机制

**核心要求**：新生成编码 ID 时，**必须立即更新所有相关索引**。

**更新清单**：
1. **全局编码索引**（ued/README.md）：
   - 添加新编码到对应类型码表格
   - 更新编码计数器（递增下一个编号）
   
2. **文档细项清单**（当前文档末尾）：
   - 添加新编码到细项编码清单
   - 确保编码、类型、标题信息完整

3. **跨项目引用索引**（如适用）：
   - 如涉及跨项目引用，更新引用关系表

**更新时机**：
- ✅ **立即更新**：编码分配完成后立即更新索引
- ❌ **禁止延迟**：不允许"稍后更新"或"批量更新"
- ✅ **双重检查**：更新后验证索引一致性

#### 编码标题命名原则

**核心原则**：编码 ID 对应的标题，**一旦新建也不得修改**。

**命名指导**：
1. **适度宽泛**：标题应具有一定的概括性，避免过于具体
2. **避免细节**：不要包含具体的实现细节或参数
3. **稳定表述**：使用相对稳定的业务概念或技术术语

**命名示例**：

| 类型 | ❌ 不推荐（过于具体） | ✅ 推荐（适度宽泛） |
|------|---------------------|-------------------|
| FR | 用户使用手机号+密码登录系统 | 用户身份认证功能 |
| NFR | 系统响应时间小于 500ms | 系统响应性能要求 |
| RUL | 账号长度 6-20 位，包含字母数字 | 账号格式规范 |
| PRN | 使用 Spring Security 框架 | 安全架构设计原则 |
| CON | 必须支持 MySQL 8.0 数据库 | 数据存储技术约束 |

**标题修改规则**：
- **绝对禁止**：修改已分配编码的标题
- **允许调整**：仅在发现明显错误时，且需走废弃+新增流程
- **最佳实践**：初始命名时考虑未来扩展性

#### 编码修改防护机制

**核心原则**：编码一经分配，**永不修改**其含义和标题。

**防护措施**：

1. **编码含义锁定**：
   - 编码分配后，其含义和标题永久固定
   - 任何 AI 修改请求必须拒绝：`"编码 RUL-016 已分配给'账号检查'，不能修改为其他含义"`

2. **修改操作限制**：
   - **禁止修改**：已分配编码的含义、标题、类型
   - **允许修改**：编码的描述内容、实现细节（非设计文档范围）
   - **废弃处理**：不再使用时标注"已废弃"，不删除不重用

3. **AI 操作约束**：
   ```
   当 AI 收到修改已分配编码的请求时：
   1. 检查编码是否已存在于全局索引
   2. 如已存在，拒绝修改并说明原因
   3. 建议分配新的编码给新内容
   4. 如原内容确实错误，走废弃+新增流程
   ```

4. **正确处理流程**：
   ```
   场景：RUL-016 原为"账号检查"，需要改为"定时器规则"
   
   ❌ 错误做法：
   - 直接修改 RUL-016 的含义为"定时器规则"
   
   ✅ 正确做法：
   - RUL-016 标注为"已废弃"，原因：需求变更
   - 分配新编码 RUL-017 给"定时器规则"
   - RUL-018 给"账号检查"（如仍需要）
   - 更新相关引用和索引
   ```

5. **冲突检测机制**：
   - 修改前检查编码是否已被引用
   - 如有引用，提醒修改影响范围
   - 建议先处理引用关系再修改

#### 编码冲突检查

手动检查编码是否已存在：
- 搜索 `ued/` 目录下的所有 `.md` 文件
- 查找目标编码（包含项目码前缀）的出现位置
- 确认编码未被使用后再分配

### 交叉引用规则

#### 标准引用格式

**核心要求**：引用细项编码时，**除非特殊原因，否则要连同标题一起带上**。

**推荐格式**（按优先级排序）：

### 1. 括号格式（推荐）
```
{项目码}-{类型码}-{编号}（{标题}）
```

**引用示例**：
- `CRM-FR-001（用户身份认证功能）`
- `ERP-NFR-002（系统响应性能要求）`
- `COM-PRN-003（安全架构设计原则）`
- `RUL-004（账号格式规范）`

**优势**：
- ✅ Markdown 原生支持，渲染效果好
- ✅ 中文括号与内容自然融合
- ✅ 视觉上编码和标题有明确区分
- ✅ 兼容所有 Markdown 编辑器

### 2. 冒号格式（备选）
```
{项目码}-{类型码}-{编号}：{标题}
```

**引用示例**：
- `CRM-FR-001：用户身份认证功能`
- `ERP-NFR-002：系统响应性能要求`
- `COM-PRN-003：安全架构设计原则`

**优势**：
- ✅ 简洁明了
- ✅ 冒号有明确的分隔作用
- ✅ 中文环境下显示效果好

### 3. 斜杠格式（特殊场景）
```
{项目码}-{类型码}-{编号}/{标题}
```

**引用示例**：
- `CRM-FR-001/用户身份认证功能`
- `ERP-NFR-002/系统响应性能要求`

**使用场景**：
- 表格中空间有限时
- 需要紧凑显示的场合

**注意**：斜杠可能在某些 Markdown 解析器中被误识别为路径

### 4. 表格格式（引用清单）
```
| 编码 | 标题 |
|------|------|
| CRM-FR-001 | 用户身份认证功能 |
| ERP-NFR-002 | 系统响应性能要求 |
```

**使用场景**：
- 文档末尾的引用清单
- 需要对齐显示的表格
- 批量引用多个编码

### 5. 链接格式（可跳转）
```
[{项目码}-{类型码}-{编号}（{标题}）](#{编码锚点})
```

**引用示例**：
- `[CRM-FR-001（用户身份认证功能）](#crm-fr-001)`
- `[ERP-NFR-002（系统响应性能要求）](#erp-nfr-002)`

**优势**：
- ✅ 可点击跳转到编码定义处
- ✅ 适合长文档的导航
- ✅ 交互性强

**注意**：需要目标位置有对应的锚点

## 格式选择指南

### 推荐使用场景

| 场景 | 推荐格式 | 原因 |
|------|----------|------|
| 正文引用 | 括号格式 | 视觉效果好，兼容性强 |
| 表格引用 | 冒号格式 | 紧凑，对齐美观 |
| 列表引用 | 括号格式 | 一致性好 |
| 链接跳转 | 链接格式 | 可导航，交互性强 |
| 空间受限 | 斜杠格式 | 紧凑显示 |

### 统一性要求

**同一文档内应保持格式统一**：
- 选择一种主要格式后，全文保持一致
- 特殊场景可使用其他格式，但应有明确理由
- 在文档开头或风格指南中说明所使用的格式

### 兼容性说明

**所有格式都完全兼容**：
- ✅ GitHub / GitLab Markdown
- ✅ VS Code Markdown Preview
- ✅ Typora
- ✅ Obsidian
- ✅ Notion
- ✅ 飞书文档
- ✅ 语雀
- ✅ 石墨文档

#### 引用场景和格式

**1. 文档正文中的引用**：
```
根据用户身份认证功能（CRM-FR-001）的要求，系统需要...
系统响应性能要求（ERP-NFR-002）规定了...
```

**2. 表格中的引用**：
| 需求 | 关联规则 | 验收标准 |
|------|----------|----------|
| CRM-FR-001 用户身份认证功能 | RUL-004 账号格式规范 | AC-001 登录成功标准 |

**3. 列表中的引用**：
- 用户身份认证功能（CRM-FR-001）
- 系统响应性能要求（ERP-NFR-002）
- 安全架构设计原则（COM-PRN-003）

#### 特殊情况处理

**可以省略标题的情况**：
1. **同一文档内多次引用**：首次引用带标题，后续可省略
   ```
   首次：用户身份认证功能（CRM-FR-001）
   后续：CRM-FR-001
   ```

2. **表格列名限制**：表格列名空间不足时，可使用编码列+标题列分离
   ```
   | 编码 | 标题 | 描述 |
   |------|------|------|
   | CRM-FR-001 | 用户身份认证功能 | ... |
   ```

3. **图表标注**：图表中空间有限，可使用纯编码
   ```
   [CRM-FR-001] → [CRM-FR-002] → [CRM-FR-003]
   ```

**必须包含标题的情况**：
- 跨文档引用
- 重要决策点引用
- 验收标准引用
- 需求追溯引用

#### 引用最佳实践

**1. 标题一致性**：
- 引用时的标题必须与编码索引中的标题完全一致
- 不得缩写或简化标题
- 不得添加额外的说明文字

**2. 位置合适性**：
- 引用放在相关内容的前面或后面
- 避免在句子中间插入长引用
- 重要引用可单独成行

**3. 频率控制**：
- 同一编码在同一章节中不宜过度重复引用
- 可使用"上述"、"前述"等词语指代

#### 引用格式检查清单

**引用前检查**：
- [ ] 编码格式正确（项目码-类型码-编号）

- [ ] 分隔符使用正确（括号/冒号/斜杠）
- [ ] 特殊情况有合理理由
- [ ] 同一文档内格式统一

**引用后验证**：
- [ ] 被引用编码确实存在
- [ ] 引用方向正确（下层引用上层）
- [ ] 标题拼写准确
- [ ] 全文引用格式一致
- [ ] Markdown 渲染效果正常

#### 自动化引用检查

**检查脚本示例**：
```bash
#!/bin/bash
# 检查引用格式合规性
check_reference_format() {
    local file=$1
    # 查找所有编码引用
    grep -o '[A-Z]\+-[A-Z]\+-[0-9]\+' "$file" | while read coding; do
        # 检查是否有合适的分隔符和标题
        if grep -q "$coding（" "$file" || grep -q "$coding：" "$file" || grep -q "$coding/" "$file"; then
            echo "✅ $coding 引用格式正确"
        else
            echo "❌ $coding 可能缺少标题或分隔符不当"
        fi
    done
}
```

**Python 检查脚本**：
```python
import re

def check_reference_format(text):
    """检查引用格式合规性"""
    # 匹配各种引用格式
    patterns = [
        r'([A-Z]+-[A-Z]+-\d+)[（](.+?)[）]',  # 括号格式
        r'([A-Z]+-[A-Z]+-\d+)[:：](.+)',      # 冒号格式
        r'([A-Z]+-[A-Z]+-\d+)[/](.+)',        # 斜杠格式
    ]
    
    issues = []
    for pattern in patterns:
        matches = re.findall(pattern, text)
        for coding, title in matches:
            if not title.strip():
                issues.append(f"{coding} 标题为空")
    
    return issues
```
