# RFC2119 设计文档模板示例

## 模板说明

本文档展示如何在各类设计文档中正确使用 RFC2119 需求级别词汇。

---

## 示例 1：L2 需求文档模板

```markdown
# CRM 系统用户管理需求文档

## 需求级别说明

本文档使用 RFC2119 定义的词汇表示需求级别：

- **必须**：绝对要求，无任何例外情况
- **应当**：强烈推荐，除非有充分且合理的理由
- **可以**：真正可选，可根据具体情况自由选择

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", 
"SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" 
in this document are to be interpreted as described in RFC 2119.

## 功能需求

### CRM-FR-001 用户注册 {#crm-fr-001}
**状态**：正式

用户 **必须** 能够通过邮箱注册账号：
- **必须** 提供有效的邮箱地址
- **必须** 设置密码，密码长度不少于 8 位
- **必须** 验证邮箱地址的有效性
- **应当** 支持手机号注册（可选）
- **可以** 支持第三方账号登录（如微信、QQ）

### CRM-FR-002 用户登录 {#crm-fr-002}
**状态**：正式

用户 **必须** 能够通过用户名密码登录：
- **必须** 验证用户名和密码的正确性
- **必须** 在连续失败 5 次后锁定账号 30 分钟
- **应当** 记录登录失败的 IP 和时间
- **可以** 提供记住登录状态的功能
- **禁止** 在客户端存储明文密码

### CRM-FR-003 密码重置 {#crm-fr-003}
**状态**：草稿

用户 **必须** 能够重置忘记的密码：
- **必须** 通过注册邮箱验证身份
- **必须** 发送重置链接到注册邮箱
- **应当** 设置重置链接的有效期为 24 小时
- **可以** 支持短信验证重置密码
- **不应当** 在重置链接中包含敏感信息

## 非功能需求

### CRM-NFR-001 性能要求 {#crm-nfr-001}
**状态**：正式

- 登录响应时间 **必须** 小于 2 秒
- 页面加载时间 **应当** 小于 3 秒
- 系统 **必须** 支持 1000 并发用户
- **可以** 使用 CDN 加速静态资源

### CRM-NFR-002 安全要求 {#crm-nfr-002}
**状态**：正式

- **必须** 使用 HTTPS 传输所有数据
- **必须** 对密码进行加盐哈希
- **应当** 实施双因子认证
- **禁止** 在日志中记录密码等敏感信息
- **必须** 定期进行安全审计

## 约束条件

### CRM-CON-001 技术约束 {#crm-con-001}
**状态**：正式

- **必须** 使用 Java 8+ 开发
- **必须** 支持 MySQL 5.7+ 数据库
- **应当** 使用 Redis 作为缓存
- **禁止** 使用已过时的框架版本

### CRM-CON-002 业务约束 {#crm-con-002}
**状态**：正式

- **必须** 遵循 GDPR 数据保护要求
- **应当** 遵循 PCI DSS 支付安全标准
- **禁止** 未经用户同意收集个人信息
- **必须** 提供数据导出功能
```

## 示例 2：L3 架构文档模板

```markdown
# CRM 系统架构设计文档

## 需求级别说明

**需求级别说明**：本文档中"必须"表示绝对要求，"应当"表示强烈推荐，"可以"表示可选。

## 架构原则

### CRM-PRN-001 安全原则 {#crm-prn-001}
**状态**：正式

- **必须** 实施纵深防御策略
- **必须** 遵循最小权限原则
- **应当** 实施零信任架构
- **禁止** 在网络边界信任内部流量
- **必须** 定期更新安全补丁

### CRM-PRN-002 可扩展性原则 {#crm-prn-002}
**状态**：正式

- **必须** 支持水平扩展
- **应当** 使用微服务架构
- **可以** 使用容器化部署
- **不应当** 使用单体架构
- **必须** 设计无状态的服务

### CRM-PRN-003 可维护性原则 {#crm-prn-003}
**状态**：草稿

- **必须** 遵循 SOLID 原则
- **应当** 实施代码审查
- **可以** 使用自动化测试
- **禁止** 在生产环境直接修改代码
- **应当** 维护完整的文档

## 系统架构

### 核心组件

#### 用户服务 (CRM-CMP-001)
- **必须** 提供用户注册、登录、权限管理
- **应当** 支持多种认证方式
- **可以** 集成第三方身份提供商
- **禁止** 暴露内部数据结构

#### 订单服务 (CRM-CMP-002)
- **必须** 处理订单创建、查询、更新
- **应当** 支持订单状态追踪
- **可以** 实现订单推荐功能
- **必须** 保证订单数据的一致性

#### 支付服务 (CRM-CMP-003)
- **必须** 集成主流支付渠道
- **应当** 支持退款和争议处理
- **可以** 提供分期付款功能
- **禁止** 存储完整的支付卡信息

## 接口设计

### RESTful API 规范

#### 认证接口 (CRM-API-001)
- **必须** 使用 JWT Token 认证
- **应当** 实现 Token 刷新机制
- **可以** 支持 OAuth 2.0
- **禁止** 在 URL 中传递敏感信息

#### 数据格式 (CRM-API-002)
- **必须** 使用 JSON 格式
- **应当** 遵循 RESTful 设计原则
- **可以** 支持 GraphQL 查询
- **必须** 统一错误响应格式

## 数据架构

### 数据库设计

#### 用户表 (CRM-DB-001)
- **必须** 包含用户基本信息
- **应当** 分离敏感信息到独立表
- **可以** 使用读写分离
- **禁止** 在表中存储明文密码

#### 订单表 (CRM-DB-002)
- **必须** 记录订单完整生命周期
- **应当** 使用合适的索引优化查询
- **可以** 实现分库分表
- **必须** 保证数据的一致性
```

## 示例 3：L6 测试文档模板

```markdown
# CRM 系统测试策略文档

## 需求级别说明

本文档使用 RFC2119 定义的词汇表示需求级别：

- **必须**：绝对要求，无任何例外情况
- **应当**：强烈推荐，除非有充分且合理的理由
- **可以**：真正可选，可根据具体情况自由选择

## 测试策略

### CRM-TC-001 功能测试 {#crm-tc-001}
**状态**：正式

- **必须** 对所有核心功能进行测试
- **必须** 达到 80% 以上的代码覆盖率
- **应当** 包含边界值测试
- **可以** 进行探索性测试
- **禁止** 在生产环境进行破坏性测试

### CRM-TC-002 性能测试 {#crm-tc-002}
**状态**：正式

- **必须** 验证系统在预期负载下的性能
- **必须** 测试系统的最大并发能力
- **应当** 进行压力测试和稳定性测试
- **可以** 使用真实的业务数据
- **不应当** 在测试期间影响生产环境

### CRM-TC-003 安全测试 {#crm-tc-003}
**状态**：草稿

- **必须** 进行渗透测试
- **必须** 验证身份认证和授权机制
- **应当** 进行数据加密测试
- **可以** 进行社会工程学测试
- **禁止** 未经授权的攻击行为

## 验收标准

### 功能验收 (CRM-AC-001)
- **必须** 所有需求功能正常工作
- **应当** 用户体验流畅
- **可以** 支持多种浏览器
- **禁止** 存在严重级别的缺陷

### 性能验收 (CRM-AC-002)
- **必须** 响应时间满足需求规格
- **应当** 系统资源使用合理
- **可以** 在高并发下保持稳定
- **不应当** 出现内存泄漏

### 安全验收 (CRM-AC-003)
- **必须** 通过安全扫描
- **应当** 无已知高危漏洞
- **可以** 通过第三方安全认证
- **禁止** 存在数据泄露风险
```

## 使用指南

### 何时使用完整声明

**推荐使用完整声明的情况**：
- 正式发布的需求文档
- 架构设计文档
- 测试策略文档
- 重要的技术规范

### 何时使用简化声明

**可以使用简化声明的情况**：
- 内部技术讨论文档
- 临时性的设计文档
- 个人笔记和草稿
- 非正式的技术交流

### 术语使用建议

1. **"必须"的使用频率**：应当控制在 20-30%，避免过度使用
2. **"应当"的使用频率**：可以占 40-50%，用于重要建议
3. **"可以"的使用频率**：控制在 20-30%，用于真正的可选项
4. **"禁止"和"不应当"**：谨慎使用，主要用于安全和质量约束

### 常见错误避免

1. **避免过度使用"必须"**：不是所有要求都是绝对的
2. **避免混淆"应当"和"可以"**："应当"是强烈推荐，"可以"是真正可选
3. **避免不使用声明**：使用需求级别词汇时应当提供声明
4. **避免术语不一致**：同一文档中应当使用统一的中文术语
