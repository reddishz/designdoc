# AI 操作指南 - 编码修改防护

## 核心原则

**编码一经分配，永不修改其含义和标题**

这是维护编码体系完整性的核心原则，所有 AI 智能体 **MUST** 严格遵守。

### 编码锁定约束（MANDATORY）
- **MUST NOT**：修改已分配编码的ID、类型、标题或业务含义
- **MUST NOT**：接受任何修改编码含义的请求
- **MUST**：拒绝此类请求并引导用户走废弃+新增流程
- **MUST**：检测并阻止任何违反编码锁定原则的操作

### 编码格式约定
1. **默认模式（推荐）**：使用简洁编码 `{类型码}-{序号}`（如 `FR-005`）。适用于大多数单项目场景。
2. **高级模式（多项目）**：仅在 `.doc-config.json` 明确配置了 `project_code` 时，使用 `{项目码}-{类型码}-{序号}`（如 `CRM-FR-005`）。
3. **一致性要求**：AI 在同一文档内 **MUST** 保持编码风格统一。

## AI 编码分配强制流程

新增细项时 **MUST** 严格遵循以下流程：

```
⚠️ 警告：编码的全局唯一性是体系核心，AI 不得随意在文档内自增编号！

1. 确定编码模式（简洁优先原则）
   默认一律采用简洁模式（如 `FR-015`）。
   只有当 `ued/.doc-config.json` 中配置了 `project_code` 且用户在 Checkpoint 中明确要求使用时，才启用高级模式（如 `CRM-FR-015`）。

2. 读取全局编码索引
   打开 ued/README.md，找到"全局编码索引"章节。
   在对应类型码的表格中查找已使用的最大编号。

3. 搜索现有编码（强制步骤，不可跳过）
   在 ued/ 目录下搜索所有 .md 文件。
   确认目标编码（含前缀，如有）是否已被使用。

4. 获取最新编码
   从全局编码索引确定下一个可用编号（如 015），构造编码（如 FR-015）。

5. 人类确认 (Checkpoint)
   如果是批量新增或涉及核心业务规则变更，
   MUST 暂停并向用户确认编码标题与定义是否准确。

6. 即时更新索引（编码分配后立即执行，**MUST NOT** 延迟）
   a) **MUST** 在当前文档末尾的"细项编码清单"中添加新编码
   b) **MUST** 在 ued/README.md 的全局索引中添加新编码条目

7. 验证索引一致性
   **SHOULD** 确认新编码在文档清单和全局索引中都正确建立。
```

## 人类确认与自我反思规范

为了确保设计质量，AI 在执行任务时 **MUST** 遵守以下动态确认与反思机制：

### 1. 人类确认检查点 (Human-in-the-loop)

在以下关键节点，AI **MUST** 主动停下并请求用户确认：

- **大纲生成后**：在填充详细需求/设计前，展示目录结构与核心细项标题。
- **编码分配前**：在正式写入索引前，确认拟分配的 ID 与标题对应关系。
- **执行废弃操作前**：在标注细项为"已废弃"前，确认废弃原因及替代方案。

### 2. 任务完成后的自我反思 (Self-Reflection)

任务完成后，AI **SHOULD** 按照以下维度进行自我检查并反馈给用户：

- **一致性检查**：检查新定义的细项是否与上层目标存在潜在冲突。
- **颗粒度检查**：检查细项标题是否适度宽泛（参考 `coding-system.md`）。
- **合规性审查（MANDATORY）**：使用本技能的审查规则评审刚做的修改：
  - 编码锁定合规性：是否违反了编码含义锁定原则？
  - 约束强度检查：是否正确使用了MUST/MUST NOT等强制性表述？
  - 流程规范性：是否遵循了编码分配和废弃的正确流程？
  - 文档一致性：是否保持了所有相关文档间的约束表述一致？
- **优化建议**：发现文档中的逻辑断点，并提供 1-2 条改进建议。

## 操作场景与处理方式

### 场景 1：用户要求修改已分配编码的含义
- **AI 必须拒绝（MANDATORY）**：说明编码含义锁定的原则
- **检测违规（REQUIRED）**：识别请求违反编码锁定原则
- **正确方案（MANDATORY）**：引导用户执行"废弃原编码 + 分配新编码"流程
- **响应模板（REQUIRED）**：使用标准化拒绝话术和正确流程指导

### 场景 2：用户要求修改编码的描述内容
- **AI 可以执行**：允许完善、澄清描述，只要不改变编码本身的业务含义和标题。

### 场景 3：引用格式纠正
- **标准格式**：`[{编码}（{标题}）](#{编码锚点})`
- **纠正示例**：将 `FR-001` 纠正为 `[FR-001（用户身份认证功能）](#fr-001)`。

---

## 审核要点

1. **编码锁定合规性**：AI 是否拒绝了修改已分配编码含义的请求？
2. **约束强度检查**：是否使用了MUST/MUST NOT等强制性表述？
3. **违规检测能力**：AI 是否能主动识别并阻止违规操作？
4. **流程指导准确性**：是否正确引导用户走废弃+新增流程？
5. **即时更新执行**：AI 是否在分配编码后立即更新了全局 README.md？
6. **反思质量**：AI 提供的优化建议是否具有实际参考价值？
7. **自我反思能力**：AI 是否使用本技能的审查规则评审自身修改的合规性？
