# 术语与概念（词汇表）约定

本 skill **不强制**每份文档都写术语表；当章节或全项目需要**统一指称**（避免「一词多义、前后不一致」）时，按本节安排即可。

## 与细项编码的关系

| 内容 | 典型做法 |
|------|----------|
| **普通业务/领域名词** | 用语义表（术语 \| 定义 \| 备注）即可，**不必**为每个词分配 `DOM-xxx` |
| **关键术语/核心概念** | 可上升为 `DOM` 细项作为结构化概念 SSOT（见触发条件），并在术语表中保留可读定义 |
| **可验证的规则、约束** | 适合用既有类型码（如 `RUL`、`CON`）写成细项，见 [coding-system.md](coding-system.md) |
| **缩写（API、部门代号等）** | 在术语表或首次出现处写全称 + 缩写 |

**原则**：术语表解决**语言一致（语言层 SSOT）**；`DOM` 解决**结构化概念引用（结构层 SSOT）**；编码体系整体解决**可追溯与索引**。三者可并存，不要为「起名字」强行造编码。

## 术语何时上升为 `DOM`

满足任一条件时，**应当**将术语/概念定义为 `DOM` 细项并纳入全局索引：

1. **跨文档复用**：同一术语在 3 篇及以上文档被稳定复用；
2. **被规则化引用**：该术语被 `RUL`、`CON`、`IF`、`AC` 等细项直接引用；
3. **存在歧义风险**：出现同义词并存、上下文含义漂移或跨团队理解不一致。

**不建议上升为 `DOM` 的场景**：一次性描述词、仅本段落局部有效的表达、纯修辞性措辞。

## 三层放置方式（择一或组合）

### 1. 项目级（推荐在有多文档、多角色协同时使用）

在当前作用域下维护**一份**词汇表（文件名团队自定；单应用模式常见为 `ued/glossary.md`，多应用模式常见为 `ued/{app-name}/glossary.md`），并在对应作用域的 `README.md` 中登记链接。

- **定义一次**：核心概念、产品名、领域对象、对外统一用语。
- **其它文档**：首次使用可写「见《项目词汇表》」或「见 `ued/glossary.md`」，避免重复长定义。

### 2. 文档级（单份文档内易混、或项目尚未建总表）

在该文档靠前位置设 **「术语与概念（可选）」** 小节（模板已预留）。适用于：

- 本稿引入较多新名词；
- 与日常用语易混的专业含义（如「订单」在本文特指 B2B 大单）。

### 3. 首次出现定义（轻量）

名词不多时，可在正文**第一次出现**用括号或脚注给一句定义，**不写独立章节**。

## 与各层级的建议关系

| 层级 | 常见用途 |
|------|----------|
| L0 | 战略、市场用语；可选列产品/赛道专有名词 |
| L1 | 角色名、场景用语与业务缩写 |
| L2 | 需求域核心对象、与外部系统约定名称 |
| L3 | 架构组件名、上下文边界用语（与 L2 对齐） |
| L4/L5 | 模块/接口命名与业务术语对齐说明（可极短） |

**下层引用上层**：若 L2 已定义，L4 只写「同 L2-{编号} 文档中的术语表」或引用项目词汇表，**避免**在多处重复长定义（与 SSoT 一致）。

## 审核提示（可选）

若文档出现大量专有名词且无项目词汇表、无文档内术语小节、无首次定义，审阅人可要求作者**择一**补齐，但不作为所有文档的硬性门槛。
