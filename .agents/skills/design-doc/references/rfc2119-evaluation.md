# RFC2119 需求级别词汇引入评估

## RFC2119 核心概念

RFC2119 定义了在规范文档中用于表示需求级别的关键词：

### 需求级别词汇

| 词汇 | 含义 | 强制性 |
|------|------|--------|
| **MUST** / **REQUIRED** / **SHALL** | 绝对要求 | 绝对必须 |
| **MUST NOT** / **SHALL NOT** | 绝对禁止 | 绝对禁止 |
| **SHOULD** / **RECOMMENDED** | 推荐但可例外 | 强烈推荐 |
| **SHOULD NOT** / **NOT RECOMMENDED** | 不推荐但可例外 | 强烈不推荐 |
| **MAY** / **OPTIONAL** | 真正可选 | 可选 |

### 使用原则

1. **谨慎使用**：必须在真正需要互操作性或限制有害行为时使用
2. ** sparingly**：节约使用，不要过度使用
3. **安全考虑**：特别关注安全影响

## 在设计文档规范中的应用评估

### ✅ 适用场景

#### 1. 编码体系规则
```markdown
编码分配 **MUST** 遵循全局唯一性原则
已分配编码的含义 **MUST NOT** 被修改
废弃细项 **SHOULD** 保留并标注废弃状态
```

#### 2. 文档结构要求
```markdown
L0-L6 文档 **MUST** 包含对应的元信息表格
细项编码 **SHOULD** 使用标准引用格式
文档状态 **MUST** 从标准状态值中选择
```

#### 3. AI 操作约束
```markdown
AI **MUST** 拒绝修改已分配编码的含义
AI **SHOULD** 检查编码引用关系
AI **MUST NOT** 在文档内自增编码编号
```

#### 4. 流程图规范
```markdown
复杂流程 **MUST** 使用 Mermaid 图表
单行流程 **MAY** 使用简化箭头形式
多步骤流程 **SHOULD** 考虑表格形式
```

### ⚠️ 需要考虑的问题

#### 1. 文档复杂度增加
- 需要在文档开头添加 RFC2119 声明
- 可能增加学习成本
- 与现有风格可能不一致

#### 2. 过度规范化风险
- 可能导致文档过于僵化
- 限制灵活性
- 增加理解难度

#### 3. 上下文适配
- RFC2119 主要用于技术规范
- 设计文档更注重指导和最佳实践
- 需要平衡规范性和实用性

## 建议的引入方案

### 方案 1：完全引入（推荐指数：⭐⭐⭐）

**优点**：
- 规范性最强
- 需求级别明确
- 与国际标准接轨

**缺点**：
- 学习成本高
- 可能过度复杂
- 与现有风格差异大

**实施方式**：
```markdown
# 在每个文档开头添加声明
The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", 
"SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" 
in this document are to be interpreted as described in RFC 2119.
```

### 方案 2：选择性引入（推荐指数：⭐⭐⭐⭐⭐）

**优点**：
- 平衡规范性和实用性
- 渐进式引入
- 保持文档可读性

**缺点**：
- 规范性不如完全引入
- 需要明确使用边界

**实施方式**：
- 只在关键约束性规则中使用
- 保持现有文档风格
- 在规范文档中说明使用原则

### 方案 3：不引入（推荐指数：⭐⭐）

**优点**：
- 保持现有风格
- 无学习成本
- 简单直接

**缺点**：
- 规范性不足
- 需求级别不明确
- 与国际标准不一致

## 推荐方案：选择性引入

### 引入原则

1. **关键约束使用**：在编码体系、AI 操作等关键约束中使用
2. **保持简洁**：不在所有地方使用，避免过度复杂化
3. **渐进引入**：先在核心规范中试用，逐步扩展
4. **明确说明**：在规范文档中说明使用原则和范围

### 具体应用场景

#### 1. 编码体系（high priority）
```markdown
- 编码分配 **MUST** 遵循全局唯一性
- 已分配编码的含义 **MUST NOT** 被修改
- 废弃细项 **SHOULD** 保留并标注废弃状态
```

#### 2. AI 操作约束（high priority）
```markdown
- AI **MUST** 拒绝修改已分配编码的含义
- AI **SHOULD** 检查编码引用关系
- AI **MUST NOT** 在文档内自增编码编号
```

#### 3. 文档结构（medium priority）
```markdown
- 文档 **MUST** 包含元信息表格
- 细项编码 **SHOULD** 使用标准引用格式
- 状态值 **MUST** 从标准状态值中选择
```

#### 4. 流程图规范（low priority）
```markdown
- 复杂流程 **SHOULD** 使用 Mermaid 图表
- 单行流程 **MAY** 使用简化箭头形式
```

### 实施步骤

1. **第一步**：在 `references/coding-system.md` 中引入使用
2. **第二步**：在 `references/ai-operations.md` 中引入使用
3. **第三步**：在 `references/review-guidelines.md` 中引入使用
4. **第四步**：评估效果，决定是否扩展到其他文档

### 文档声明模板

```markdown
## 需求级别说明

本文档使用 RFC2119 定义的词汇表示需求级别：

- **MUST**：绝对要求，必须遵守
- **SHOULD**：强烈推荐，除非有充分理由否则应遵守
- **MAY**：可选，可根据具体情况决定

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", 
"SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" 
in this document are to be interpreted as described in RFC 2119.
```

## 预期效果

### 优势
1. **需求级别明确**：清晰区分必须、推荐、可选的要求
2. **规范性提升**：与国际标准接轨，提升专业性
3. **约束力增强**：关键规则的约束力更明确
4. **可读性改善**：读者能快速识别重要程度

### 风险
1. **学习成本**：团队需要理解 RFC2119 概念
2. **过度依赖**：可能过度依赖形式化表达
3. **灵活性降低**：可能限制创造性解决方案

### 缓解措施
1. **培训支持**：提供 RFC2119 概念培训
2. **使用指南**：制定详细的使用指南
3. **定期评估**：定期评估使用效果，调整策略

## 结论

**建议采用选择性引入方案**，在关键约束性规则中使用 RFC2119 词汇，既提升规范性，又保持实用性。重点在编码体系、AI 操作约束等核心领域引入，逐步扩展到其他领域。
