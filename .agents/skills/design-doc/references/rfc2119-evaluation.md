# RFC2119 需求级别词汇引入评估与规范

## RFC2119 核心概念与中文术语定义

### 需求级别词汇与中文术语对照

| RFC2119 词汇 | 中文术语 | 精确定义 | 强制级别 | 适用场景 |
|-------------|----------|----------|----------|----------|
| **MUST** | **必须** | 绝对要求，无任何例外情况，违反将导致系统无法正常工作或产生严重后果 | 绝对必须 | 关键约束、安全要求、核心规则 |
| **MUST NOT** | **禁止** | 绝对禁止，无任何例外情况，违反将导致系统故障或安全风险 | 绝对禁止 | 危险行为、错误操作、破坏性行为 |
| **SHOULD** | **应当** | 强烈推荐，除非有充分且合理的理由，否则应当遵守 | 强烈推荐 | 最佳实践、重要建议、质量要求 |
| **SHOULD NOT** | **不应当** | 强烈不推荐，除非有充分且合理的理由，否则不应当这样做 | 强烈不推荐 | 不推荐的做法、潜在风险 |
| **MAY** | **可以** | 真正可选，可根据具体情况、需求或偏好自由选择 | 可选 | 可选功能、增强特性、灵活选择 |

### 中文术语使用规范

**1. 术语一致性原则**：
- 在整个 skill 中，每个 RFC2119 词汇 **必须** 对应唯一的中文术语
- **禁止** 一个 RFC2119 词汇对应多个中文术语
- **禁止** 多个 RFC2119 词汇对应同一个中文术语

**2. 上下文适用性**：
- **必须**：用于描述不可违反的核心约束
- **应当**：用于描述重要的最佳实践
- **可以**：用于描述真正的可选选项

**3. 术语强度递减**：
```
必须 > 应当 > 可以
禁止 > 不应当 > (无对应)
```

### 使用原则

1. **谨慎使用**：**必须** 在真正需要互操作性或限制有害行为时使用
2. **节约使用**：**应当** 节约使用，避免过度规范化
3. **安全考虑**：**必须** 特别关注安全影响
4. **术语一致**：**必须** 保持中文术语的一致性

## 在设计文档规范中的应用评估

### ✅ 适用场景

#### 1. 编码体系规则
```markdown
编码分配 **MUST** 遵循全局唯一性原则
已分配编码的含义 **MUST NOT** 被修改
废弃细项 **SHOULD** 保留并标注废弃状态
```

#### 2. 文档结构要求
```markdown
L0-L6 文档 **MUST** 包含对应的元信息表格
细项编码 **SHOULD** 使用标准引用格式
文档状态 **MUST** 从标准状态值中选择
```

#### 3. AI 操作约束
```markdown
AI **MUST** 拒绝修改已分配编码的含义
AI **SHOULD** 检查编码引用关系
AI **MUST NOT** 在文档内自增编码编号
```

#### 4. 流程图规范
```markdown
复杂流程 **MUST** 使用 Mermaid 图表
单行流程 **MAY** 使用简化箭头形式
多步骤流程 **SHOULD** 考虑表格形式
```

### ⚠️ 需要考虑的问题

#### 1. 文档复杂度增加
- 需要在文档开头添加 RFC2119 声明
- 可能增加学习成本
- 与现有风格可能不一致

#### 2. 过度规范化风险
- 可能导致文档过于僵化
- 限制灵活性
- 增加理解难度

#### 3. 上下文适配
- RFC2119 主要用于技术规范
- 设计文档更注重指导和最佳实践
- 需要平衡规范性和实用性

### 设计文档中的使用场景

#### 1. 需求文档（L2）
```markdown
## 功能需求

### 用户认证
- 系统 **必须** 支持用户名密码登录
- 密码 **必须** 加密存储
- 登录失败 **应当** 限制重试次数
- 系统 **可以** 支持第三方登录

## 非功能需求

### 性能要求
- 登录响应时间 **必须** 小于 2 秒
- 并发用户数 **应当** 支持 1000+
- 系统 **可以** 使用缓存优化性能
```

#### 2. 架构文档（L3-L4）
```markdown
## 架构原则

### 安全原则
- 所有敏感数据 **必须** 加密传输
- 用户权限 **必须** 基于角色控制
- 日志记录 **应当** 包含关键操作
- 系统 **可以** 集成外部认证服务

### 设计约束
- **禁止** 在数据库中存储明文密码
- **不应当** 使用硬编码配置
- **必须** 遵循数据最小化原则
```

#### 3. 测试文档（L6）
```markdown
## 测试要求

### 功能测试
- 核心功能 **必须** 100% 覆盖
- 边界条件 **应当** 充分测试
- 异常场景 **可以** 适当简化

### 性能测试
- 关键接口 **必须** 满足性能指标
- 压力测试 **应当** 模拟真实负载
- 兼容性测试 **可以** 覆盖主流浏览器
```

本文档使用 RFC2119 定义的词汇表示需求级别：

- **必须**：绝对要求，无任何例外情况
- **应当**：强烈推荐，除非有充分且合理的理由

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", 
"SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL"

```

**简化声明**（用于一般文档）：

**需求级别说明**：本文档中"必须"表示绝对要求，"应当"表示强烈推荐，"可以"表示可选。
```

### 使用指南

#### 何时使用 **必须**
- 安全相关的关键要求
- 法律法规的强制要求
- 系统核心功能的必要条件
- 会导致系统故障的约束

#### 何时使用 **应当**
- 行业最佳实践
- 重要的质量要求
- 推荐的设计模式
- 用户体验的重要考虑

#### 何时使用 **可以**
- 可选的增强功能
- 灵活的实现选择
- 扩展性考虑
- 优化性建议

#### 何时使用 **禁止**
- 安全风险行为
- 系统破坏操作
- 违法违规行为
- 导致数据丢失的操作

#### 何时使用 **不应当**
- 潜在风险的做法
- 不推荐的实现方式
- 可能影响性能的操作
- 降低可维护性的设计

### 方案 1：完全引入（推荐指数：⭐⭐⭐）

**优点**：
- 规范性最强
- 需求级别明确
- 与国际标准接轨

**缺点**：
- 学习成本高
- 可能过度复杂
- 与现有风格差异大

**实施方式**：
```markdown
# 在每个文档开头添加声明
The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", 
"SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" 
in this document are to be interpreted as described in RFC 2119.
```

### 方案 2：选择性引入（推荐指数：⭐⭐⭐⭐⭐）

**优点**：
- 平衡规范性和实用性
- 渐进式引入
- 保持文档可读性

**缺点**：
- 规范性不如完全引入
- 需要明确使用边界

**实施方式**：
- 只在关键约束性规则中使用
- 保持现有文档风格
- 在规范文档中说明使用原则

### 方案 3：不引入（推荐指数：⭐⭐）

**优点**：
- 保持现有风格
- 无学习成本
- 简单直接

**缺点**：
- 规范性不足
- 需求级别不明确
- 与国际标准不一致

3. **渐进引入**：先在核心规范中试用，逐步扩展
4. **明确说明**：在规范文档中说明使用原则和范围
