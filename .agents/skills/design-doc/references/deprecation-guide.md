# 废弃细项处理规范

## 概述

废弃细项是设计文档演进过程中的正常现象。本规范确保废弃过程的规范性和一致性。

**与「文档状态」的区别**：`status-definitions.md` 中的 **草稿 / 正式 / 废弃** 主要针对**整份设计文档**在元信息表中的生命周期。本文中的 **「细项状态：已废弃」** 指某一 **FR / RUL / PRN 等编码细项** 的作废标注（标题删除线、`状态` 元数据等），二者**不要混用同一套枚举**——细项废弃时，承载它的文档仍可为「正式」，除非整份文档一并废止。

**与编码体系的关系**：全文示例使用项目码 **CRM**；`ued/README.md`、跨文档引用、锚点须与 [coding-system.md](coding-system.md) **全量编码**一致（如 `CRM-RUL-004`、`#crm-rul-004`）。同文档内叙述可沿用简写，但索引与对外引用须写全。

## 核心原则

**1. 保留原则**：废弃的细项必须保留，不能删除  
**2. 标注原则**：通过状态字段和标题标注废弃状态  
**3. 追溯原则**：反向追溯所有引用，同步标注废弃  
**4. 传播原则**：若某细项被废弃，依赖它的内容可能失效；**优先**将引用**更新为替代细项**（替代方案），仅在引用方语义不再成立时，再将引用方细项**标注废弃**。避免无必要的级联废弃。

## 废弃流程

### 第一步：标记细项废弃

**在细项中添加废弃标记**：

```markdown
### CRM-RUL-004 账号格式规范 ~~已废弃~~

**细项状态**：已废弃
**废弃时间**：2024-03-23
**废弃原因**：需求变更，采用新的账号格式规范
**替代方案**：CRM-RUL-018 新账号格式规范

原内容：账号长度 6-20 位，包含字母数字
```

**标记方式**：
- **标题标记**：在细项标题后添加 `~~已废弃~~`
- **状态字段**：在细项中明确标注 `细项状态：已废弃`（勿与文档元信息中的「文档状态」混淆）
- **保留原内容**：保留原细项内容，不删除

**字段说明**：
- **细项状态**：必须标注为"已废弃"
- **废弃时间**：格式为 YYYY-MM-DD，追溯废弃时间
- **废弃原因**：说明废弃的具体原因
- **替代方案**：如有替代细项，填写编码；如无替代方案，填写"无"

### 第二步：反向追溯引用

**搜索所有引用**（须同时搜简写与全量编码，避免漏网）：

在 `ued/` 目录中同时检索 `CRM-RUL-004` 与 `RUL-004`。

**追溯范围**：
- 同文档内的引用
- 其他文档的引用
- 跨项目的引用
- 索引中的引用

### 第三步：评估并处理引用

对每个引用，按以下决策树处理：

```
1. 是否有替代方案？
   ├─ 有 → 更新引用为替代编码，标注原引用废弃
   └─ 无 → 进入 2

2. 引用方是否仍成立？
   ├─ 仍有其他有效引用 → 标注本条引用废弃，保留引用方
   ├─ 可更新为替代方案 → 更新为替代编码，标注原引用废弃
   └─ 完全依赖废弃细项 → 标注引用方废弃
```

**处理示例**：

**情况 1：有替代方案（推荐）**

```markdown
# 废弃前
根据 [CRM-RUL-004（账号格式规范）](#crm-rul-004) 的要求，系统需要...

# 废弃后
根据 ~~[CRM-RUL-004（账号格式规范）](#crm-rul-004)~~（已废弃）和 [CRM-RUL-018（新账号格式规范）](#crm-rul-018) 的要求，系统需要...
```

**情况 2：无替代方案，引用方仍成立**

```markdown
### CRM-FR-001 用户认证需求
**文档状态**：正式
根据 ~~[CRM-RUL-004（账号格式规范）](#crm-rul-004)~~（已废弃）和 [CRM-RUL-005（密码安全规范）](#crm-rul-005) 的要求，系统需要...
```

**情况 3：无替代方案，引用方也废弃**

```markdown
### CRM-FR-002 登录功能 ~~已废弃~~

**细项状态**：已废弃
**废弃时间**：2024-03-23
**废弃原因**：依赖的 CRM-RUL-004（账号格式规范）已废弃，且无替代方案
**替代方案**：无
```

### 第四步：更新索引

**全局编码索引更新**：

```markdown
| CRM-FR-001 | FR | 用户身份认证功能 | 已废弃 | 2024-03-23 | 依赖规则废弃，无替代方案 |
```

**文档细项清单更新**：


## 废弃标记规范

### 标准标记格式

**细项标题标记**：

```markdown
### {项目码}-{类型码}-{序号} 细项标题 ~~已废弃~~
```

**细项内容标记**（全部为必须字段）：

```markdown
**细项状态**：已废弃
**废弃时间**：YYYY-MM-DD
**废弃原因**：{具体原因}
**替代方案**：{替代细项编码}（如无替代方案，填写"无"）
```

**字段说明**：
- **细项状态**：必须标注为"已废弃"，便于机器识别
- **废弃时间**：格式为 YYYY-MM-DD，追溯废弃时间，便于版本管理（AI 自动使用当前日期）
- **废弃原因**：说明废弃原因，便于决策追溯
- **替代方案**：如有替代细项，填写编码；如无替代方案，填写"无"

**引用标记**：

```markdown
~~[{项目码}-{类型码}-{序号}（细项标题）](#锚点)~~（已废弃）
```

### 废弃原因分类

| 废弃原因 | 说明 | 示例 |
|----------|------|------|
| 需求变更 | 业务需求发生变化 | 业务模式调整，原有需求不再适用 |
| 设计优化 | 设计方案被更好的方案替代 | 采用新的架构设计，原设计被淘汰 |
| 技术升级 | 技术栈升级导致规则变更 | 升级到新版本，技术规范发生变化 |
| 业务重构 | 业务流程重新设计 | 组织架构调整，业务流程重组 |
| 约束解除 | 原有约束条件不再存在 | 监管政策变化，合规要求解除 |
| 依赖废弃 | 依赖的细项已废弃 | 依赖的规则或需求已废弃，导致本细项无效 |
| 重复冗余 | 与其他细项重复或冗余 | 发现功能重复，合并处理 |

## 特殊情况处理

### 批量废弃

**场景**：业务重构导致大量细项废弃

**推荐方案：按依赖层级分批废弃**

**批次划分**：
1. **底层细项**：RUL、PRN、CON 等基础规则和约束
2. **中层细项**：FR、NFR、IF 等功能和接口
3. **上层细项**：GOL、SCN、STK 等目标和场景

**处理流程**（每批）：

1. **批量标记废弃**：为该批所有细项添加4个必须字段
2. **批量追溯引用**：搜索该批细项的所有引用（含简写和全量编码）
3. **批量评估引用方**：按决策树判断每个引用方是否需要废弃
4. **批量更新索引**：该批处理完成后，立即更新全局索引和文档清单

**关键注意事项**：
- 按从下到上的顺序处理，避免引用方在引用源还未处理时就被评估
- 每批处理完成后，立即更新索引，不要等到所有批次都处理完
- 保持详细的废弃记录，便于追溯

### 循环依赖废弃

**场景**：A 依赖 B，B 依赖 A，其中之一需要废弃

**处理流程**：
1. **同时废弃**：将循环依赖的细项同时标记废弃
2. **说明原因**：在废弃原因中说明循环依赖问题
3. **统一处理**：同时处理所有相关引用

### 紧急废弃

**场景**：发现重大问题需要立即废弃

**处理流程**：
1. **立即标记**：先标记废弃，后续完善信息
2. **快速追溯**：快速处理主要引用
3. **后续完善**：补充完整的废弃信息

## 工具支持建议

### 推荐工具与命令

**1. 搜索引用**

```bash
# 搜索完整编码
grep -r "CRM-RUL-004" ued/

# 搜索简写编码
grep -r "RUL-004" ued/

# 组合搜索（排除已处理的废弃引用）
(grep -r "CRM-RUL-004" ued/; grep -r "RUL-004" ued/) | grep -v "已废弃"
```

**2. 区分已处理引用**

```bash
# 已处理引用：包含 ~~已废弃~~ 或已改为替代编码
grep -r "CRM-RUL-004" ued/ | grep "已废弃\|已替代"

# 未处理引用：包含原编码，但不包含废弃标记
grep -r "CRM-RUL-004" ued/ | grep -v "已废弃\|已替代"
```

**3. 批量处理脚本思路**

```bash
# 伪代码示例，根据实际需求实现
for item in $(cat deprecation-list.txt); do
  # 1. 标记废弃（添加4个必须字段）
  # 2. 追溯引用（使用 grep 搜索）
  # 3. 处理引用（按决策树评估）
  # 4. 更新索引（全局 + 文档清单）
done

# 汇总需人工判断的项
echo "以下引用方需人工判断是否废弃：" > manual-review.txt
```

## 审核要点

### 废弃处理审核

**审核清单**：
- [ ] 废弃细项已标记 `~~已废弃~~`
- [ ] 废弃细项状态字段已更新
- [ ] 所有引用已追溯并标注废弃或已改为替代编码
- [ ] 引用方状态已正确评估（优先替代，避免过度级联废弃）
- [ ] 全局索引已更新废弃状态
- [ ] 文档清单已更新废弃状态

### 常见问题

**问题1：遗漏引用**
- 检查方法：使用 `grep` 同时搜索完整编码和简写编码
- 解决方案：补充标注遗漏的引用

**问题2：引用方状态评估错误**
- 检查方法：按决策树重新评估每个引用方
- 解决方案：修正评估结果，更新引用方状态

**问题3：索引未同步**
- 检查方法：对比细项状态和索引状态
- 解决方案：立即更新所有索引（全局 + 文档清单）

## 最佳实践

### 预防措施

1. **谨慎创建**：避免创建可能很快废弃的细项
2. **充分评审**：创建前充分评审必要性和稳定性
3. **依赖分析**：分析细项间的依赖关系

### 处理原则

1. **完整性**：确保废弃处理完整，无遗漏
2. **一致性**：遵循统一的废弃标记格式与全量编码（见 coding-system）
3. **及时性**：及时处理废弃和引用更新
4. **可追溯性**：保留完整的废弃历史

通过这套清晰的废弃处理规范，可以确保废弃细项得到规范、完整的处理。
