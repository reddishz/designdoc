# 废弃细项处理规范

## 概述

废弃细项是设计文档演进过程中的正常现象。本规范确保废弃过程的规范性和一致性。

## 核心原则

**1. 保留原则**：废弃的细项必须保留，不能删除
**2. 标注原则**：通过状态字段和标题标注废弃状态
**3. 追溯原则**：反向追溯所有引用，同步标注废弃
**4. 传播原则**：废弃状态可以传播，导致引用方也废弃

## 废弃流程

### 第一步：标记细项废弃

**在细项中添加废弃标记**：

```markdown
### RUL-004 账号格式规范 ~~已废弃~~

**状态**：已废弃  
**废弃时间**：2024-03-23  
**废弃原因**：需求变更，采用新的账号格式规范  
**替代方案**：RUL-018 新账号格式规范  
**废弃人**：张三  

原内容：账号长度 6-20 位，包含字母数字
```

**标记方式**：
- **标题标记**：在细项标题后添加 `~~已废弃~~`
- **状态字段**：在细项中明确标注 `状态：已废弃`
- **保留原内容**：保留原细项内容，不删除

### 第二步：反向追溯引用

**搜索所有引用**：
```bash
# 搜索所有引用该细项的文档
grep -r "RUL-004" ued/ --include="*.md"
```

**追溯范围**：
- 同文档内的引用
- 其他文档的引用
- 跨项目的引用
- 索引中的引用

### 第三步：处理引用关系

**统一处理方式**：所有引用都保留，同步标注废弃

**引用方处理**：
```markdown
# 废弃前
根据 [RUL-004（账号格式规范）](#rul-004) 的要求，系统需要...

# 废弃后
根据 ~~[RUL-004（账号格式规范）](#rul-004)~~（已废弃）的要求，系统需要...
```

**引用方状态判断**：
1. **可以继续工作**：引用方标注废弃，但内容保留
2. **彻底无效**：引用方也标注为废弃状态

### 第四步：评估引用方状态

**评估标准**：

| 情况 | 处理方式 | 示例 |
|------|----------|------|
| 引用方仍有其他有效引用 | 标注废弃，内容保留 | FR-001 引用 RUL-004，但还有其他有效规则 |
| 引用方完全依赖废弃细项 | 标注引用方也废弃 | FR-001 只引用 RUL-004，且 RUL-004 废弃了 |
| 引用方可以更新后工作 | 更新引用，标注原引用废弃 | FR-001 更新为引用 RUL-018，原 RUL-004 引用标注废弃 |

**引用方废弃示例**：
```markdown
### FR-001 用户身份认证功能 ~~已废弃~~

**状态**：已废弃  
**废弃时间**：2024-03-23  
**废弃原因**：依赖的 RUL-004（账号格式规范）已废弃，导致本需求无效  
**废弃人**：张三  

原内容：用户使用手机号+密码登录系统
```

### 第五步：更新索引

**全局编码索引更新**：
```markdown
| 编码 | 类型 | 标题 | 状态 | 废弃时间 | 废弃原因 | 替代方案 |
|------|------|------|------|----------|----------|----------|
| CRM-RUL-004 | RUL | 账号格式规范 | 已废弃 | 2024-03-23 | 需求变更 | CRM-RUL-018 |
| CRM-FR-001 | FR | 用户身份认证功能 | 已废弃 | 2024-03-23 | 依赖规则废弃 | 无 |
```

**文档细项清单更新**：
```markdown
## 细项编码清单

| 编码 | 类型 | 标题 | 状态 | 备注 |
|------|------|------|------|------|
| CRM-RUL-004 | RUL | 账号格式规范 | 已废弃 | 2024-03-23 废弃，替代：RUL-018 |
| CRM-FR-001 | FR | 用户身份认证功能 | 已废弃 | 2024-03-23 废弃，依赖规则废弃 |
```

## 废弃标记规范

### 标准标记格式

**细项标题标记**：
```markdown
### 编码-编号 细项标题 ~~已废弃~~
```

**细项内容标记**：
```markdown
**状态**：已废弃
**废弃时间**：YYYY-MM-DD
**废弃原因**：{具体原因}
**替代方案**：{替代细项编码}（如有）
**废弃人**：{废弃人姓名}
```

**引用标记**：
```markdown
~~[编码-编号（细项标题）](#锚点)~~（已废弃）
```

### 废弃原因分类

| 废弃原因 | 说明 | 示例 |
|----------|------|------|
| 需求变更 | 业务需求发生变化 | 业务模式调整，原有需求不再适用 |
| 设计优化 | 设计方案被更好的方案替代 | 采用新的架构设计，原设计被淘汰 |
| 技术升级 | 技术栈升级导致规则变更 | 升级到新版本，技术规范发生变化 |
| 业务重构 | 业务流程重新设计 | 组织架构调整，业务流程重组 |
| 约束解除 | 原有约束条件不再存在 | 监管政策变化，合规要求解除 |
| 依赖废弃 | 依赖的细项已废弃 | 依赖的规则或需求已废弃，导致本细项无效 |
| 重复冗余 | 与其他细项重复或冗余 | 发现功能重复，合并处理 |

## 引用追溯规范

### 追溯流程

**1. 全面搜索**：
```bash
# 搜索所有引用
grep -r "RUL-004" ued/ --include="*.md" | grep -v "已废弃"
```

**2. 分类处理**：
- **直接引用**：直接引用该细项的文档
- **间接引用**：通过其他细项间接引用的文档
- **索引引用**：在索引中的引用

**3. 逐个标注**：
- 在每个引用位置添加废弃标记
- 评估引用方是否需要废弃
- 更新相关索引

### 引用方状态判断

**判断流程**：
```
1. 检查引用方是否有其他有效引用
   - 有：标注废弃，内容保留
   - 无：检查是否可以更新后工作

2. 检查是否可以更新后工作
   - 可以：更新引用，标注原引用废弃
   - 不可以：标注引用方也废弃
```

**判断示例**：
```markdown
# 情况1：引用方有其他有效引用
### FR-001 用户认证需求
**状态**：正式
根据 ~~[RUL-004（账号格式规范）](#rul-004)~~（已废弃）和 [RUL-005（密码安全规范）](#rul-005) 的要求...

# 情况2：引用方完全依赖废弃细项
### FR-002 登录功能 ~~已废弃~~
**状态**：已废弃
**废弃原因**：依赖的 RUL-004（账号格式规范）已废弃
根据 ~~[RUL-004（账号格式规范）](#rul-004)~~（已废弃）的要求...

# 情况3：引用方可以更新后工作
### FR-003 用户注册
**状态**：正式
根据 ~~[RUL-004（账号格式规范）](#rul-004)~~（已废弃）（原引用）和 [RUL-018（新账号格式规范）](#rul-018) 的要求...
```

## 特殊情况处理

### 批量废弃

**场景**：业务重构导致大量细项废弃

**处理流程**：
1. **制定废弃计划**：列出所有待废弃细项
2. **分批处理**：按依赖关系分批废弃
3. **统一标注**：批量添加废弃标记
4. **全面追溯**：批量处理引用关系

### 循环依赖废弃

**场景**：A 依赖 B，B 依赖 A，其中之一需要废弃

**处理流程**：
1. **同时废弃**：将循环依赖的细项同时标记废弃
2. **说明原因**：在废弃原因中说明循环依赖问题
3. **统一处理**：同时处理所有相关引用

### 紧急废弃

**场景**：发现重大问题需要立即废弃

**处理流程**：
1. **立即标记**：先标记废弃，后续完善信息
2. **快速追溯**：快速处理主要引用
3. **后续完善**：补充完整的废弃信息

## 工具支持

### 废弃处理脚本

**检查脚本**：
```bash
#!/bin/bash
# 检查废弃细项处理完整性
check_deprecated_items() {
    local coding=$1
    
    # 查找所有引用
    echo "=== 查找 $coding 的引用 ==="
    grep -r "$coding" ued/ --include="*.md" | while read line; do
        file=$(echo "$line" | cut -d: -f1)
        if [[ "$line" != *"已废弃"* ]]; then
            echo "❌ $file 中有未标注废弃的引用：$line"
        else
            echo "✅ $file 中的引用已标注废弃"
        fi
    done
}
```

**批量废弃脚本**：
```bash
#!/bin/bash
# 批量处理废弃
batch_deprecate() {
    local deprecated_list=$1
    
    while read coding; do
        echo "处理废弃：$coding"
        # 1. 标记细项废弃
        # 2. 查找引用
        # 3. 标注引用废弃
        # 4. 更新索引
    done < "$deprecated_list"
}
```

## 审核要点

### 废弃处理审核

**审核清单**：
- [ ] 废弃细项已标记 `~~已废弃~~`
- [ ] 废弃细项状态字段已更新
- [ ] 所有引用已追溯并标注废弃
- [ ] 引用方状态已正确评估
- [ ] 全局索引已更新废弃状态
- [ ] 文档清单已更新废弃状态

### 常见问题

**问题1：遗漏引用**
- 检查方法：全面搜索编码引用
- 解决方案：补充标注遗漏的引用

**问题2：引用方状态错误**
- 检查方法：评估引用方的实际状态
- 解决方案：重新评估并更新引用方状态

**问题3：索引未同步**
- 检查方法：对比细项状态和索引状态
- 解决方案：同步更新所有索引

## 最佳实践

### 预防措施

1. **谨慎创建**：避免创建可能很快废弃的细项
2. **充分评审**：创建前充分评审必要性和稳定性
3. **依赖分析**：分析细项间的依赖关系

### 处理原则

1. **完整性**：确保废弃处理完整，无遗漏
2. **一致性**：遵循统一的废弃标记格式
3. **及时性**：及时处理废弃和引用更新
4. **可追溯性**：保留完整的废弃历史

通过这套简化的废弃处理规范，可以确保废弃细项得到统一、完整的处理。
