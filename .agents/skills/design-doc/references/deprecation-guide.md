# 废弃细项处理规范

## 概述

废弃细项是设计文档演进过程中的正常现象。本规范确保废弃过程的规范性、可追溯性和一致性。

## 废弃定义

**废弃细项**：指因需求变更、设计调整、业务重构等原因，不再需要或被替代的细项编码。

**废弃类型**：
- **需求废弃**：业务需求不再适用
- **设计废弃**：设计方案被更好的方案替代
- **规则废弃**：业务规则或约束不再有效
- **流程废弃**：业务流程被重新设计
- **约束废弃**：技术或业务约束被解除

## 废弃流程

### 第一步：废弃评估

**评估清单**：
- [ ] 确认细项确实不再需要
- [ ] 检查是否有其他文档引用此细项
- [ ] 评估废弃的影响范围
- [ ] 确认是否有替代方案

**评估责任人**：
- **细项创建者**：负责初始评估
- **架构师**：负责技术影响评估
- **产品经理**：负责业务影响评估

### 第二步：废弃标记

**在原细项位置添加废弃标记**：

```markdown
### RUL-004 账号格式规范 ~~已废弃~~

**废弃时间**：2024-03-23  
**废弃原因**：需求变更，采用新的账号格式规范  
**替代方案**：RUL-018 新账号格式规范  
**废弃人**：张三  

~~原内容：账号长度 6-20 位，包含字母数字~~

**引用影响**：
- CRM-FR-001 用户身份认证功能 → 已更新引用 RUL-018
- ERP-NFR-002 系统响应性能要求 → 无影响
```

**废弃标记格式**：
- 使用 `~~已废弃~~` 标记
- 包含完整的废弃信息
- 保留原内容供参考

### 第三步：引用更新

**引用更新流程**：
1. **搜索所有引用**：
   ```bash
   grep -r "RUL-004" ued/ --include="*.md"
   ```

2. **逐个处理引用**：
   - **直接替代**：用新细项替代
   - **移除引用**：如无替代方案则移除
   - **保留引用**：特殊情况需说明理由

3. **更新引用格式**：
   ```markdown
   # 废弃前
   根据 RUL-004（账号格式规范）的要求...
   
   # 废弃后
   ~~根据 RUL-004（账号格式规范）的要求~~（已废弃，现使用 RUL-018）
   ```

### 第四步：索引更新

**全局编码索引更新**：
```markdown
| 编码 | 类型 | 标题 | 状态 | 废弃时间 | 废弃原因 | 替代方案 |
|------|------|------|------|----------|----------|----------|
| CRM-RUL-004 | RUL | 账号格式规范 | 已废弃 | 2024-03-23 | 需求变更 | CRM-RUL-018 |
```

**文档细项清单更新**：
```markdown
## 细项编码清单

| 编码 | 类型 | 标题 | 状态 | 备注 |
|------|------|------|------|------|
| CRM-RUL-004 | RUL | 账号格式规范 | ~~已废弃~~ | 2024-03-23 废弃，替代：RUL-018 |
| CRM-RUL-018 | RUL | 新账号格式规范 | 正式 | - |
```

### 第五步：变更记录

**在相关文档中记录废弃变更**：
```markdown
## 变更记录

| 版本 | 日期 | 变更类型 | 变更内容 | 变更人 |
|------|------|----------|----------|--------|
| v1.2 | 2024-03-23 | 废弃 | 废弃 RUL-004 账号格式规范，采用 RUL-018 | 张三 |
| v1.2 | 2024-03-23 | 新增 | 新增 RUL-018 新账号格式规范 | 张三 |
```

## 废弃标记规范

### 标准废弃标记

**在细项标题中**：
```markdown
### 编码-编号 细项标题 ~~已废弃~~
```

**在细项内容中**：
```markdown
**废弃信息**：
- 废弃时间：YYYY-MM-DD
- 废弃原因：{具体原因}
- 替代方案：{替代细项编码}（如有）
- 废弃人：{废弃人姓名}

~~原内容：{原细项内容}~~
```

### 废弃原因分类

| 废弃原因 | 说明 | 示例 |
|----------|------|------|
| 需求变更 | 业务需求发生变化 | 业务模式调整，原有需求不再适用 |
| 设计优化 | 设计方案被更好的方案替代 | 采用新的架构设计，原设计被淘汰 |
| 技术升级 | 技术栈升级导致规则变更 | 升级到新版本，技术规范发生变化 |
| 业务重构 | 业务流程重新设计 | 组织架构调整，业务流程重组 |
| 约束解除 | 原有约束条件不再存在 | 监管政策变化，合规要求解除 |
| 重复冗余 | 与其他细项重复或冗余 | 发现功能重复，合并处理 |

### 替代方案处理

**有替代方案**：
```markdown
**替代方案**：RUL-018 新账号格式规范
**迁移指导**：详见 [迁移指南](#迁移指南-rul-004-to-rul-018)
```

**无替代方案**：
```markdown
**替代方案**：无
**处理方式**：直接移除相关引用，更新业务流程
```

## 引用处理规范

### 引用更新优先级

| 引用类型 | 处理优先级 | 处理方式 |
|----------|------------|----------|
| 核心需求引用 | 高 | 必须更新或明确说明 |
| 架构设计引用 | 高 | 必须更新到新方案 |
| 实现细节引用 | 中 | 可选择性更新 |
| 示例说明引用 | 低 | 可保留并标注废弃 |

### 引用更新模板

**直接替代模板**：
```markdown
# 废弃前
根据 [RUL-004（账号格式规范）](#rul-004) 的要求...

# 废弃后
根据 [RUL-018（新账号格式规范）](#rul-018) 的要求...
```

**移除引用模板**：
```markdown
# 废弃前
~~根据 RUL-004（账号格式规范）的要求~~（已废弃）
# 废弃后
（移除相关内容）
```

**保留说明模板**：
```markdown
~~根据 RUL-004（账号格式规范）的要求~~（已废弃，原要求不再适用）
```

## 特殊情况处理

### 批量废弃

**场景**：业务重构导致大量细项废弃

**处理流程**：
1. **制定废弃计划**：列出所有待废弃细项
2. **分批处理**：按优先级分批废弃
3. **统一更新**：批量更新引用和索引
4. **发布公告**：通知相关人员

### 紧急废弃

**场景**：发现重大问题需要立即废弃

**处理流程**：
1. **立即标记**：先标记废弃，后续完善信息
2. **快速评估**：快速评估影响范围
3. **紧急通知**：通知相关团队
4. **后续完善**：补充完整的废弃信息

### 争议废弃

**场景**：对是否废弃存在争议

**处理流程**：
1. **暂停废弃**：暂停废弃流程
2. **争议评审**：组织相关方评审
3. **决策记录**：记录评审结果
4. **执行决策**：根据评审结果执行

## 废弃审计

### 定期审计

**审计频率**：每季度一次

**审计内容**：
- 废弃细项的处理完整性
- 引用更新的准确性
- 废弃信息的规范性
- 替代方案的有效性

**审计报告**：
```markdown
# 废弃细项审计报告

**审计期间**：2024年 Q1
**审计人**：{审计人姓名}

## 审计结果
- 废弃细项总数：12
- 处理完整：10
- 引用未更新：2
- 信息不规范：1

## 问题发现
1. RUL-005 引用未完全更新
2. FR-012 废弃信息不完整

## 改进建议
1. 加强废弃流程的执行监督
2. 完善自动化检查工具
```

### 自动化检查

**检查脚本**：
```bash
#!/bin/bash
# 检查废弃细项处理完整性
check_deprecated_items() {
    local issues=()
    
    # 查找所有废弃细项
    grep -r "已废弃" ued/ --include="*.md" | while read line; do
        coding=$(echo "$line" | grep -o '[A-Z]\+-[A-Z]\+-[0-9]\+')
        
        # 检查是否还有引用
        if grep -r "$coding" ued/ --include="*.md" | grep -v "已废弃" > /dev/null; then
            issues+=("$coding 仍有活跃引用")
        fi
    done
    
    # 输出问题
    for issue in "${issues[@]}"; do
        echo "❌ $issue"
    done
}
```

## 最佳实践

### 预防措施

1. **谨慎创建**：避免创建可能很快废弃的细项
2. **充分评审**：创建前充分评审必要性和稳定性
3. **版本控制**：使用版本控制跟踪变更历史
4. **影响评估**：废弃前充分评估影响

### 处理原则

1. **完整性**：确保废弃处理完整，无遗漏
2. **可追溯性**：保留废弃历史和原因
3. **一致性**：遵循统一的废弃规范
4. **及时性**：及时处理，避免积累

### 沟通协调

1. **提前沟通**：废弃前与相关方沟通
2. **明确通知**：清晰通知废弃决定和影响
3. **提供指导**：提供替代方案和迁移指导
4. **记录反馈**：记录相关方的反馈和建议

通过这套完整的废弃处理规范，可以确保废弃细项得到规范、完整、可追溯的处理。
