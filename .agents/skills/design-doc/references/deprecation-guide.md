# 废弃细项处理规范

## 概述

废弃细项是设计文档演进过程中的正常现象。本规范确保废弃过程的规范性和一致性。

**与「文档状态」的区别**：`status-definitions.md` 中的 **草稿 / 正式 / 废弃** 主要针对**整份设计文档**在元信息表中的生命周期。本文中的 **「细项状态：已废弃」** 指某一 **FR / RUL / PRN 等编码细项** 的作废标注（标题删除线、`状态` 元数据等），二者**不要混用同一套枚举**——细项废弃时，承载它的文档仍可为「正式」，除非整份文档一并废止。

**与编码体系的关系**：全文示例使用项目码 **CRM**；`ued/README.md`、跨文档引用、锚点须与 [coding-system.md](coding-system.md) **全量编码**一致（如 `CRM-RUL-004`、`#crm-rul-004`）。同文档内叙述可沿用简写，但索引与对外引用须写全。

## 核心原则

**1. 保留原则**：废弃的细项必须保留，不能删除  
**2. 标注原则**：通过状态字段和标题标注废弃状态  
**3. 追溯原则**：反向追溯所有引用，同步标注废弃  
**4. 传播原则**：若某细项被废弃，依赖它的内容可能失效；**优先**将引用**更新为替代细项**（替代方案），仅在引用方语义不再成立时，再将引用方细项**标注废弃**。避免无必要的级联废弃。

## 废弃流程

### 第一步：标记细项废弃

**在细项中添加废弃标记**：

```markdown
### CRM-RUL-004 账号格式规范 ~~已废弃~~

**细项状态**：已废弃
**废弃时间**：2024-03-23
**废弃原因**：需求变更，采用新的账号格式规范
**替代方案**：CRM-RUL-018 新账号格式规范

原内容：账号长度 6-20 位，包含字母数字
```

**标记方式**：
- **标题标记**：在细项标题后添加 `~~已废弃~~`
- **状态字段**：在细项中明确标注 `细项状态：已废弃`（勿与文档元信息中的「文档状态」混淆）
- **保留原内容**：保留原细项内容，不删除

**字段说明**：
- **细项状态**：必须标注为"已废弃"
- **废弃时间**：格式为 YYYY-MM-DD，追溯废弃时间
- **废弃原因**：说明废弃的具体原因
- **替代方案**：如有替代细项，填写编码；如无替代方案，填写"无"

### 第二步：反向追溯引用

**搜索所有引用**（须同时搜简写与全量编码，避免漏网）：

在 `ued/` 目录中同时检索 `CRM-RUL-004` 与 `RUL-004`。

**追溯范围**：
- 同文档内的引用
- 其他文档的引用
- 跨项目的引用
- 索引中的引用

### 第三步：处理引用关系

**统一处理方式**：所有引用都保留，同步标注废弃或改为指向替代编码

**引用方处理**：

```markdown
# 废弃前
根据 [CRM-RUL-004（账号格式规范）](#crm-rul-004) 的要求，系统需要...

# 废弃后（仍保留句子结构时）
根据 ~~[CRM-RUL-004（账号格式规范）](#crm-rul-004)~~（已废弃）的要求，系统需要...
```

**引用方状态判断**：
1. **可以继续工作**：引用方保留，对原引用加废弃标记，或改写为引用替代细项
2. **彻底无效**：引用方细项也标注废弃（仅在语义无法由替代项承接时）

### 第四步：评估引用方状态

**评估标准**：

| 情况 | 处理方式 | 示例 |
|------|----------|------|
| 引用方仍有其他有效引用 | 标注原引用废弃，内容保留；优先增加对替代细项的引用 | CRM-FR-001 除 CRM-RUL-004 外仍引用 CRM-RUL-005 |
| 引用方完全依赖废弃细项且无法替代 | 标注引用方细项废弃 | CRM-FR-001 仅依赖已废弃规则且业务上不再成立 |
| 引用方可以更新后工作 | **优先**：更新为引用 CRM-RUL-018，并对 CRM-RUL-004 的引用标注废弃 | CRM-FR-001 更新引用至新规则 |

**引用方废弃示例**：

```markdown
### CRM-FR-001 用户身份认证功能 ~~已废弃~~

**细项状态**：已废弃
**废弃时间**：2024-03-23
**废弃原因**：依赖的 CRM-RUL-004（账号格式规范）已废弃，导致本需求无效
**替代方案**：无

原内容：用户使用手机号+密码登录系统
```

### 第五步：更新索引

**全局编码索引更新**：

```markdown
| 编码 | 类型 | 标题 | 状态 | 废弃时间 | 备注 |
|------|------|------|------|----------|------|
| CRM-RUL-004 | RUL | 账号格式规范 | 已废弃 | 2024-03-23 | 替代：CRM-RUL-018 |
| CRM-FR-001 | FR | 用户身份认证功能 | 已废弃 | 2024-03-23 | 依赖规则废弃，无替代方案 |
```

**文档细项清单更新**：

```markdown
|------|------|------|------|------|
```

## 废弃标记规范

### 标准标记格式

**细项标题标记**：

```markdown
### {项目码}-{类型码}-{序号} 细项标题 ~~已废弃~~
```

**细项内容标记**（全部为必须字段）：

```markdown
**细项状态**：已废弃
**废弃时间**：YYYY-MM-DD
**废弃原因**：{具体原因}
**替代方案**：{替代细项编码}（如无替代方案，填写"无"）
```

**字段说明**：
- **细项状态**：必须标注为"已废弃"，便于机器识别
- **废弃时间**：格式为 YYYY-MM-DD，追溯废弃时间，便于版本管理（AI 自动使用当前日期）
- **废弃原因**：说明废弃原因，便于决策追溯
- **替代方案**：如有替代细项，填写编码；如无替代方案，填写"无"

**引用标记**：

```markdown
~~[{项目码}-{类型码}-{序号}（细项标题）](#锚点)~~（已废弃）
```

### 废弃原因分类

| 废弃原因 | 说明 | 示例 |
|----------|------|------|
| 需求变更 | 业务需求发生变化 | 业务模式调整，原有需求不再适用 |
| 设计优化 | 设计方案被更好的方案替代 | 采用新的架构设计，原设计被淘汰 |
| 技术升级 | 技术栈升级导致规则变更 | 升级到新版本，技术规范发生变化 |
| 业务重构 | 业务流程重新设计 | 组织架构调整，业务流程重组 |
| 约束解除 | 原有约束条件不再存在 | 监管政策变化，合规要求解除 |
| 依赖废弃 | 依赖的细项已废弃 | 依赖的规则或需求已废弃，导致本细项无效 |
| 重复冗余 | 与其他细项重复或冗余 | 发现功能重复，合并处理 |

## 引用追溯规范

### 追溯流程

**1. 全面搜索**：

在 `ued/` 目录中同时检索 `CRM-RUL-004` 与 `RUL-004`，并区分“已废弃引用”与“未处理引用”。

**2. 分类处理**：
- **直接引用**：直接引用该细项的文档
- **间接引用**：通过其他细项间接引用的文档
- **索引引用**：在索引中的引用

**3. 逐个标注**：
- 在每个引用位置添加废弃标记或改为替代编码
- 评估引用方是否需要废弃
- 更新相关索引

### 引用方状态判断

**判断流程**：

```
1. 是否可将引用更新为「替代细项」？
   - 可以：更新引用，原引用处标注已废弃或删除线链接
   - 不可以：进入 2

2. 引用方是否仍成立？
   - 是：仅标注本条引用废弃，保留引用方细项
   - 否：引用方细项标注废弃
```

**判断示例**：

```markdown
# 情况1：引用方有其他有效引用（文档可为正式，细项仍有效）
### CRM-FR-001 用户认证需求
**文档状态**：正式
根据 ~~[CRM-RUL-004（账号格式规范）](#crm-rul-004)~~（已废弃）和 [CRM-RUL-005（密码安全规范）](#crm-rul-005) 的要求...

# 情况2：引用方完全依赖废弃细项且无法替代
### CRM-FR-002 登录功能 ~~已废弃~~
**细项状态**：已废弃
**废弃原因**：依赖的 CRM-RUL-004（账号格式规范）已废弃
根据 ~~[CRM-RUL-004（账号格式规范）](#crm-rul-004)~~（已废弃）的要求...

# 情况3：引用方可以更新后工作（推荐）
### CRM-FR-003 用户注册
**文档状态**：正式
根据 ~~[CRM-RUL-004（账号格式规范）](#crm-rul-004)~~（已废弃）（原引用）和 [CRM-RUL-018（新账号格式规范）](#crm-rul-018) 的要求...
```

## 特殊情况处理

### 批量废弃

**场景**：业务重构导致大量细项废弃

**处理流程**：
1. **制定废弃计划**：列出所有待废弃细项
2. **分批处理**：按依赖关系分批废弃
3. **统一标注**：批量添加废弃标记
4. **全面追溯**：批量处理引用关系

### 循环依赖废弃

**场景**：A 依赖 B，B 依赖 A，其中之一需要废弃

**处理流程**：
1. **同时废弃**：将循环依赖的细项同时标记废弃
2. **说明原因**：在废弃原因中说明循环依赖问题
3. **统一处理**：同时处理所有相关引用

### 紧急废弃

**场景**：发现重大问题需要立即废弃

**处理流程**：
1. **立即标记**：先标记废弃，后续完善信息
2. **快速追溯**：快速处理主要引用
3. **后续完善**：补充完整的废弃信息

## 工具支持

### 废弃处理能力

建议工具或 AI 至少支持以下能力：

1. 同时搜索完整编码与简写编码
2. 区分“定义位置”“有效引用”“已废弃引用”
3. 识别尚未标记废弃的引用位置
4. 校验文档清单与全局索引是否已同步更新

### 批量废弃处理

批量处理时建议统一执行以下步骤：

1. 读取待废弃编码列表
2. 逐个标记细项废弃
3. 检索并处理所有引用
4. 更新文档清单与全局索引
5. 汇总仍需人工判断的语义失效项

## 审核要点

### 废弃处理审核

**审核清单**：
- [ ] 废弃细项已标记 `~~已废弃~~`
- [ ] 废弃细项状态字段已更新
- [ ] 所有引用已追溯并标注废弃或已改为替代编码
- [ ] 引用方状态已正确评估（优先替代，避免过度级联废弃）
- [ ] 全局索引已更新废弃状态
- [ ] 文档清单已更新废弃状态

### 常见问题

**问题1：遗漏引用**
- 检查方法：全面搜索编码引用（含全量与简写）
- 解决方案：补充标注遗漏的引用

**问题2：引用方状态错误**
- 检查方法：评估引用方的实际状态
- 解决方案：重新评估并更新引用方状态

**问题3：索引未同步**
- 检查方法：对比细项状态和索引状态
- 解决方案：同步更新所有索引

## 最佳实践

### 预防措施

1. **谨慎创建**：避免创建可能很快废弃的细项
2. **充分评审**：创建前充分评审必要性和稳定性
3. **依赖分析**：分析细项间的依赖关系

### 处理原则

1. **完整性**：确保废弃处理完整，无遗漏
2. **一致性**：遵循统一的废弃标记格式与全量编码（见 coding-system）
3. **及时性**：及时处理废弃和引用更新
4. **可追溯性**：保留完整的废弃历史

通过这套简化的废弃处理规范，可以确保废弃细项得到统一、完整的处理。
